# GitHub Copilot Packaging

## Assignment

**Goal:** Determine how current GitHub Copilot surfaces package and distribute reusable agent functionality, and what Skill Issue must implement to deliver skills, Markdown guidance, scripts, a standalone CLI, configuration, references, assets, and supporting files as a functional bundle.

**Scope:** Internet-only research using current official GitHub and Microsoft documentation and primary GitHub repositories. The investigation covers GitHub Copilot CLI, Copilot cloud agent on GitHub.com, VS Code, other supported IDEs, repository-level customization, Copilot plugin marketplaces, agent skills, custom agents, hooks, MCP, CLI extensions, GitHub CLI skills/extensions, installation, update, discovery, invocation, trust, permissions, precedence, and enterprise controls.

**Exclusions:** Local product source, local Skill Issue implementation details, secondary tutorials, legacy GitHub Copilot Extensions as a recommended delivery target, and implementation work outside this research document.

## Findings

### Finding 1: Copilot now has a native multi-component plugin bundle

GitHub Copilot plugins are the closest direct match for Skill Issue. One plugin directory can bundle custom agents, agent skills, hooks, MCP server configuration, LSP server configuration, command directories, and Copilot CLI extensions. A required `plugin.json` names the package and optionally points to each component. Skills themselves are explicitly folders of instructions, scripts, resources, and arbitrary supporting files, so references, examples, templates, and assets that are consumed by a skill can travel inside the skill directory. The plugin installer caches the whole plugin directory, which also permits shared scripts and assets elsewhere in the plugin, although only declared components are automatically discovered.

**Evidence:** GitHub defines a plugin as “a bundle of components in a single installable unit” and lists agents, skills, hooks, MCP, and LSP as supported contents in [About GitHub Copilot plugins](https://docs.github.com/en/copilot/concepts/agents/about-plugins). The [CLI plugin reference](https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-plugin-reference) additionally defines `commands`, `extensions`, `mcpServers`, and `lspServers` component fields. [About agent skills](https://docs.github.com/en/copilot/concepts/agents/about-agent-skills) and [Adding agent skills for GitHub Copilot CLI](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-skills) state that skills include instructions, scripts, and resources, and that the agent can use scripts and examples within the skill directory.

**Implication:** Skill Issue should implement one Copilot/OpenPlugin-compatible plugin as the primary bundle: a manifest plus `skills/`, `agents/`, hook configuration, optional MCP/LSP configuration, optional `commands/`, optional `extensions/`, and bundled supporting files. Assets or references needed only by one skill should live under that skill for reliable relative access; genuinely shared files may live under a shared plugin directory and be referenced through a plugin-root mechanism where the target surface supports it.

### Finding 2: The portable manifest should target the shared plugin format deliberately

VS Code, Copilot CLI, and Claude Code now share a plugin repository shape, but format detection and root-path behavior differ. VS Code checks `.plugin/plugin.json`, root `plugin.json`, `.github/plugin/plugin.json`, then `.claude-plugin/plugin.json`. It treats root `plugin.json` as Copilot format, for which VS Code documents no plugin-root token; OpenPlugin format supplies `${PLUGIN_ROOT}`, and Claude format supplies `${CLAUDE_PLUGIN_ROOT}`. GitHub's CLI reference recognizes the same manifest locations and documents `${PLUGIN_ROOT}` for LSP paths plus `${COPILOT_PLUGIN_DATA}` for persistent writable data. The latest primary CLI changelog also says plugin hooks receive `PLUGIN_ROOT`, `COPILOT_PLUGIN_ROOT`, and `CLAUDE_PLUGIN_ROOT`, but this hook environment detail is not yet fully reflected in the stable plugin reference.

**Evidence:** Format detection order, root-token differences, hook-file differences, and the shared format claim are documented in [Agent plugins in VS Code](https://code.visualstudio.com/docs/agent-customization/agent-plugins). Manifest locations, `${PLUGIN_ROOT}` in LSP configuration, and `${COPILOT_PLUGIN_DATA}` are documented in the [CLI plugin reference](https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-plugin-reference). The environment-variable addition is recorded in the primary [`github/copilot-cli` changelog](https://github.com/github/copilot-cli/blob/main/changelog.md).

**Implication:** Skill Issue should prefer `.plugin/plugin.json` as the canonical cross-tool manifest when hook or MCP commands must address bundled files, and use `${PLUGIN_ROOT}` for OpenPlugin-aware surfaces. It should also keep hook scripts capable of resolving their own location, because token behavior is still surface- and format-dependent. Runtime state must go to `${COPILOT_PLUGIN_DATA}` rather than the installed-plugin cache. Cross-platform scripts need both POSIX and PowerShell launch paths where Windows support is promised.

### Finding 3: Installation, discovery, updating, and version pinning are first-class in Copilot CLI

Copilot CLI supports direct installation from a registered marketplace (`plugin@marketplace`), GitHub repository, repository subdirectory, arbitrary Git URL, or local path. Users can list, enable, disable, uninstall, and update one or all plugins. Marketplace catalogs are Git repositories or local directories containing `marketplace.json`; plugin entries carry versions and can point at repository-relative directories or remote GitHub/Git sources. Marketplace source definitions can pin a full commit SHA for reproducible installs. Direct and marketplace installs are cached under `~/.copilot/installed-plugins/`.

**Evidence:** Commands, install specifiers, manifest fields, marketplace source types, SHA pinning, and cache paths are specified in the [GitHub Copilot CLI plugin reference](https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-plugin-reference). User workflows are documented in [Finding and installing plugins](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/plugins-finding-installing), while the minimal Git-repository marketplace structure is documented in [Creating a plugin marketplace](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/plugins-marketplace).

**Implication:** Skill Issue should ship `plugin.json`, a semantically versioned `marketplace.json`, tagged releases, and a documented direct-install command for early adopters. The marketplace entry should pin release tags and optionally immutable commit SHAs. Release automation must bump the version in both the plugin manifest and marketplace entry, then validate installation from the remote source rather than from a working tree.

### Finding 4: Component cooperation is automatic, but names and precedence are part of the public contract

When enabled, plugin-provided skills, agents, hooks, MCP servers, commands, and extensions are loaded into the same Copilot environment. Skills can be selected automatically from their descriptions or explicitly with `/skill-name`; custom agents can be selected by users or inferred for delegation; commands are explicit slash commands; hooks execute deterministically at lifecycle events; MCP contributes tools; CLI extensions contribute tools and slash commands. Naming collisions are consequential: Copilot CLI uses first-found-wins for agents and skills, so project and personal definitions can silently suppress plugin definitions, while MCP is last-wins and a later plugin can override a same-named server. Built-ins cannot be overridden.

**Evidence:** Skill automatic and explicit invocation is documented in [Adding agent skills for GitHub Copilot CLI](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-skills). Agent invocation fields (`disable-model-invocation`, `user-invocable`) and tool selection are in [Custom agents configuration](https://docs.github.com/en/copilot/reference/custom-agents-configuration). The [CLI plugin reference](https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-plugin-reference) gives the component loading order and collision rules. VS Code confirms that all installed plugin customizations appear together in chat in [Agent plugins in VS Code](https://code.visualstudio.com/docs/agent-customization/agent-plugins).

**Implication:** Skill Issue must allocate globally distinctive kebab-case skill names, agent file IDs, command names, extension names, and MCP server IDs. Each skill description should state a narrow trigger, while high-impact workflows may set `disable-model-invocation: true` and remain explicit-only. The bundle should include a diagnostic skill or command that reports expected components and collision symptoms, because skill/agent conflicts can be silently ignored.

### Finding 5: Plugin installation does not imply unrestricted execution

Both Copilot CLI and VS Code keep execution trust separate from component discovery. Copilot CLI automatically permits read-only work but prompts for destructive shell commands, writes, external URLs, and other side effects unless the user or administrator grants scoped permission. Permissions can be scoped by tool, shell subcommand, MCP server/tool, path, URL, session, or repository location, and deny rules win. Skill `allowed-tools` can pre-approve tools when a skill is active, but GitHub explicitly warns against pre-approving shell/bash without reviewing every referenced script. Copilot CLI extensions execute with the user's privileges and require explicit trust; VS Code prompts when a plugin marketplace is first used, warns that plugins can run hooks and MCP servers, prompts before starting newly configured MCP servers, and separately confirms side-effecting tool calls.

**Evidence:** CLI approval layers and persisted location-scoped permissions are documented in [Allowing and denying tool use](https://docs.github.com/en/copilot/how-tos/copilot-cli/use-copilot-cli/allowing-tools) and path/URL rules in [Configuring GitHub Copilot CLI](https://docs.github.com/en/copilot/how-tos/copilot-cli/set-up-copilot-cli/configure-copilot-cli). The shell pre-approval warning appears in [Adding agent skills for GitHub Copilot CLI](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-skills). Plugin and MCP trust are documented in [Agent plugins in VS Code](https://code.visualstudio.com/docs/agent-customization/agent-plugins), [Add and manage MCP servers in VS Code](https://code.visualstudio.com/docs/agent-customization/mcp-servers), and [Tools in VS Code](https://code.visualstudio.com/docs/agents/concepts/tools). Extension privilege and trust warnings are in [About extensions for GitHub Copilot CLI](https://docs.github.com/en/enterprise-cloud@latest/copilot/concepts/agents/copilot-cli/about-cli-extensions).

**Implication:** Skill Issue should request the least possible tools, avoid `allowed-tools: "*"`, leave shell/write/network confirmation intact by default, document exactly what each script or MCP server executes, and provide read-only or dry-run modes. Marketplace and README material should expose the bundle's executable surfaces before install. Installation should never silently add broad CLI permission flags or modify enterprise policy.

### Finding 6: GitHub.com cloud agent supports plugins declaratively, with cloud-specific execution limits

Copilot cloud agent does not use the interactive CLI install command. A repository enables a plugin through `enabledPlugins` in `.github/copilot/settings.json`, and adds non-default marketplaces through `extraKnownMarketplaces`. Plugin content then runs in the cloud agent's Linux, non-interactive, ephemeral sandbox. Hooks have restricted outbound networking, no persisted filesystem, and only Bash/cross-platform command support. MCP secrets must be configured as Copilot Agents organization or repository secrets/variables. Cloud custom agents ignore some IDE-only fields, and their profile-embedded `mcp-servers` configuration is not used by VS Code or other IDE custom agents.

**Evidence:** Declarative cloud installation is specified in [About GitHub Copilot plugins](https://docs.github.com/en/copilot/concepts/agents/about-plugins). Cloud hook OS, filesystem, networking, interactivity, and configuration constraints are documented in the [GitHub Copilot hooks reference](https://docs.github.com/en/copilot/reference/hooks-reference). Cloud/IDE agent-field differences and MCP secret processing are in [Custom agents configuration](https://docs.github.com/en/copilot/reference/custom-agents-configuration).

**Implication:** Skill Issue should provide a ready-to-commit `.github/copilot/settings.json` snippet and treat cloud support as a distinct compatibility target. Cloud-compatible scripts must run on Linux without user prompts, must not rely on local home-directory state, and must tolerate restricted network access. Secrets belong in documented Agents secret/variable setup, never in the plugin. Agent-level MCP configuration should have an equivalent repository/plugin MCP configuration for IDE portability.

### Finding 7: VS Code provides the richest graphical plugin marketplace surface, but it remains preview and governed

VS Code can browse agent plugins with `@agentPlugins`, install from configured marketplaces or directly from a Git URL, enable or disable globally or per workspace, uninstall, and check for updates. It automatically discovers plugins installed by Copilot CLI from `~/.copilot/installed-plugins/`, enabling one local installation to serve both CLI and VS Code. Workspace files can recommend marketplaces and plugins using `extraKnownMarketplaces` and `enabledPlugins`; users see recommendations rather than an unconditional silent install. Updates are checked automatically every 24 hours when extension auto-update policy permits, but npm/PyPI sourced plugins require explicit update confirmation. Organization policy can disable agent plugins or constrain marketplaces.

**Evidence:** Installation, CLI-plugin discovery, enable/disable state, updates, recommendations, trust prompts, and cross-tool compatibility are documented in [Agent plugins in VS Code (Preview)](https://code.visualstudio.com/docs/agent-customization/agent-plugins). Enterprise policy coverage for Copilot CLI, cloud agent, and VS Code 1.122+ is documented in [About enterprise-managed plugin standards](https://docs.github.com/en/copilot/concepts/agents/about-enterprise-plugin-standards).

**Implication:** Skill Issue can offer one install route through Copilot CLI and rely on VS Code discovery, while also documenting the VS Code marketplace UI and direct-source command. It should call the VS Code plugin experience preview-quality, require a supported VS Code version, and design onboarding for policy-denied environments. Workspace configuration should recommend rather than assume consent.

### Finding 8: Other IDEs have customization primitives, but no verified native plugin-marketplace bundle

Current official support matrices show that JetBrains, Eclipse, Xcode, and Visual Studio consume different subsets of repository instructions, path-specific instructions, prompt files, custom agents, and agent skills. Agent skills are expressly supported in agent mode for VS Code and JetBrains, while prompt files are available in VS Code, Visual Studio, and JetBrains. Custom agents are preview in JetBrains, Eclipse, and Xcode. The current enterprise plugin-standard documentation names only Copilot CLI, cloud agent, and VS Code 1.122+; no inspected official source establishes Copilot agent-plugin marketplace installation for JetBrains, Visual Studio, Eclipse, or Xcode.

**Evidence:** Per-surface instruction support is enumerated in [Support for different types of custom instructions](https://docs.github.com/en/copilot/reference/custom-instructions-support). Cross-surface skill availability is stated in [About agent skills](https://docs.github.com/en/copilot/concepts/agents/about-agent-skills). Custom-agent preview status is in [Custom agents configuration](https://docs.github.com/en/copilot/reference/custom-agents-configuration). Plugin-governance clients are listed in [About enterprise-managed plugin standards](https://docs.github.com/en/copilot/concepts/agents/about-enterprise-plugin-standards).

**Implication:** Skill Issue should advertise the one-click plugin bundle for Copilot CLI, Copilot cloud agent, and supported VS Code only. For JetBrains and other IDEs, it should provide a fallback installer that copies portable skills to `.github/skills` or `.agents/skills`, custom agents to `.github/agents` where supported, prompt files to `.github/prompts` where supported, and instructions to `.github/copilot-instructions.md` or `.github/instructions/`. Unsupported components must be omitted or surfaced as unavailable rather than claimed to install.

### Finding 9: Repository-level guidance is adjacent to plugins, not a declared plugin component

Repository-wide instructions, path-specific instructions, `AGENTS.md`, and prompt files are separate repository customization mechanisms. They are loaded from fixed workspace paths and have broader or different applicability than an on-demand skill. The plugin manifest has no declared `instructions` or `prompts` component; its closest equivalents are skills, custom agents, and commands. Therefore a plugin can carry Markdown guidance as skill/agent content, but it cannot by itself make a bundled `.github/copilot-instructions.md` apply to an arbitrary repository without that file being present in the repository.

**Evidence:** Fixed repository instruction paths and prompt-file paths are documented in [Adding repository custom instructions for GitHub Copilot in your IDE](https://docs.github.com/en/copilot/how-tos/configure-custom-instructions-in-your-ide/add-repository-instructions-in-your-ide) and the cross-surface matrix in [Support for different types of custom instructions](https://docs.github.com/en/copilot/reference/custom-instructions-support). The exhaustive component path fields in the [CLI plugin reference](https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-plugin-reference) include agents, skills, commands, hooks, extensions, MCP, and LSP, but no instruction/prompt-file field.

**Implication:** Skill Issue should divide Markdown by semantics: always-on repository policy belongs in optional files installed into `.github/`; task-specific guidance belongs in plugin skills; specialized personas belong in agents; manually invoked templates belong in commands/prompt files. A fallback bootstrap command or standalone installer is required if Skill Issue promises to place repository-wide instructions or settings into existing projects.

### Finding 10: A Copilot plugin cannot by itself deliver a normal standalone executable on `PATH`

A plugin can carry executable scripts and can expose agent-facing operations through skills, hooks, MCP, and a plugin-provided Copilot CLI extension. However, Copilot CLI extensions are experimental, JavaScript-only Node modules that run as subprocesses inside an interactive Copilot CLI session and expose tools or slash commands; they are not normal shell executables. GitHub's plugin manifest has no field that installs an arbitrary binary into the user's `PATH`. GitHub CLI has a separate, mature extension installer that installs release binaries or executable scripts as `gh` subcommands, while a truly standalone `skill-issue` command still needs a conventional package/release installer.

**Evidence:** Extension scope, JavaScript-only entry files, plugin discovery, experimental opt-in, trust, and slash-command/tool capabilities are documented in [About extensions for GitHub Copilot CLI](https://docs.github.com/en/enterprise-cloud@latest/copilot/concepts/agents/copilot-cli/about-cli-extensions) and [Creating extensions for GitHub Copilot CLI](https://docs.github.com/en/enterprise-cloud@latest/copilot/tutorials/create-an-extension). The [plugin manifest reference](https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-plugin-reference) exposes no binary/PATH installation field. [`gh extension install`](https://cli.github.com/manual/gh_extension_install) documents release-binary and script-extension installation for GitHub CLI.

**Implication:** Skill Issue needs two coordinated artifacts: the Copilot plugin for agent-host integration and a separately distributed standalone CLI (for example, release binaries and/or ecosystem package managers). The plugin's skills, hooks, or MCP/extension adapter may invoke that CLI after detecting it, but should provide a clear missing-CLI diagnostic and installation command. If a GitHub-native shell surface is useful, a `gh-skill-issue` extension can be an additional adapter, not a substitute for the standalone binary. An experimental Copilot CLI extension should only be used for Copilot-session-specific state or slash commands that cannot be expressed as skills.

### Finding 11: `gh skill` is a useful fallback for skills only, not for the full Skill Issue bundle

GitHub CLI now provides preview commands to search, preview, install, update, and publish agent skills. It installs to host-specific project or user locations, supports GitHub Copilot and many other agents, tracks source metadata, resolves tags or commit SHAs, and can install all skills from a repository. It validates and publishes Agent Skills releases. It does not install hooks, MCP configuration, custom agents, plugin settings, or a standalone executable as part of the skill operation.

**Evidence:** Scope, supported hosts, destinations, version resolution, and source tracking are documented in [`gh skill install`](https://cli.github.com/manual/gh_skill_install); update behavior is in [`gh skill update`](https://cli.github.com/manual/gh_skill_update); validation and release publishing are in [`gh skill publish`](https://cli.github.com/manual/gh_skill_publish). The command group is explicitly preview in the [`gh skill` manual](https://cli.github.com/manual/gh_skill).

**Implication:** Skill Issue can publish portable skills that work with `gh skill install` as a graceful cross-agent fallback. The full Copilot experience still requires the plugin or Skill Issue's own bootstrap installer. Release metadata should align skill tags with plugin/CLI compatibility, and documentation must distinguish “skills installed” from “full bundle installed.”

### Finding 12: Legacy GitHub Copilot Extensions are a rejected distribution target

The older GitHub Copilot Extensions product and its developer policy are obsolete for a new bundle. GitHub announced deprecation effective 2025-11-10 in favor of MCP. The current plugin system, MCP, agent skills, and CLI/IDE extension mechanisms are the active customization paths.

**Evidence:** The warning is explicit in the [GitHub Copilot Extension Developer Policy](https://docs.github.com/en/site-policy/github-terms/github-copilot-extension-developer-policy), last updated 2025-10-20.

**Implication:** Skill Issue should not build a legacy Copilot Extension or depend on the former Copilot Extensions marketplace. Any remote capability should be exposed through MCP; packaging and discovery should use Copilot plugin marketplaces, GitHub repositories, and optional conventional CLI distribution.

### Finding 13: Recommended Skill Issue delivery architecture

The strongest supported design is a layered release with one primary Copilot bundle and explicit fallbacks. The primary plugin should contain portable agent skills, custom agents, commands, minimal hooks, optional MCP/LSP configuration, and only those Copilot CLI extensions that truly require session integration. A separately versioned standalone CLI should own deterministic installation, validation, configuration migration, and non-agent usage. Repository bootstrap should be opt-in and should write repository instructions, settings, or prompt files only after showing the planned changes. A marketplace repository supplies discovery and updates; direct Git installs support early testing; `gh skill` exposes the portable skill subset; enterprise settings support governed automatic availability.

**Evidence:** The native bundle and marketplace capabilities are established by [About GitHub Copilot plugins](https://docs.github.com/en/copilot/concepts/agents/about-plugins), the [CLI plugin reference](https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-plugin-reference), and [Agent plugins in VS Code](https://code.visualstudio.com/docs/agent-customization/agent-plugins). The separate repository customization paths are established by [Support for different types of custom instructions](https://docs.github.com/en/copilot/reference/custom-instructions-support). The executable gap and alternatives are established by [About extensions for GitHub Copilot CLI](https://docs.github.com/en/enterprise-cloud@latest/copilot/concepts/agents/copilot-cli/about-cli-extensions), [`gh extension install`](https://cli.github.com/manual/gh_extension_install), and [`gh skill`](https://cli.github.com/manual/gh_skill).

**Implication:** Concretely, Skill Issue should implement: (1) `.plugin/plugin.json`; (2) `marketplace.json`; (3) `skills/<name>/SKILL.md` plus colocated scripts/references/assets; (4) `agents/*.agent.md`; (5) namespaced MCP IDs and optional `.mcp.json`; (6) cross-platform hooks with least privilege; (7) optional JavaScript Copilot CLI extension behind an experimental compatibility flag; (8) a standalone `skill-issue` binary distributed independently; (9) `skill-issue init copilot` or equivalent to install repository-only instructions/settings with preview and rollback; (10) `gh skill`-compatible skill publication; and (11) a compatibility matrix and diagnostic command covering Copilot CLI, cloud agent, VS Code, JetBrains, Visual Studio, Eclipse, Xcode, and GitHub Copilot app.

## Notes

- **Caveat — preview surfaces:** VS Code agent plugins, enterprise-managed plugin standards, `gh skill`, custom agents in JetBrains/Eclipse/Xcode, and Copilot CLI extensions are documented as preview or experimental and may change.
- **Caveat — cloud hooks and installed plugins:** The hooks reference says a cloud job has no installed plugins “by default,” while the newer plugin documentation supports declarative cloud-agent plugins through repository settings. This is best read as the baseline sandbox before configured plugin projection, but the exact plugin-hook materialization path is not documented. Cloud end-to-end validation is required before promising plugin hooks there.
- **Caveat — root variables:** VS Code documents no root token for Copilot-format plugins but does define `${PLUGIN_ROOT}` for OpenPlugin format. Copilot CLI's primary changelog reports additional root environment variables for hooks. Use OpenPlugin format and self-locating scripts until stable docs converge.
- **Unsupported:** No inspected official source establishes native Copilot agent-plugin marketplace installation in JetBrains, Visual Studio, Eclipse, Xcode, or the GitHub Copilot app. Only their separately documented customization primitives should be claimed.
- **Unsupported:** No inspected plugin schema installs a standalone executable into `PATH`, merges arbitrary repository instruction files, or performs general post-install configuration migration.
- **Useful validation commands:** `copilot plugin install ./plugin`, `/plugin list`, `/skills list`, `/agent`, `/env`, `/mcp show`, `/extensions manage`, and VS Code `@agentPlugins` plus “Extensions: Check for Extension Updates.”
