# Warp Packaging

## Assignment

**Goal:** Determine how current Warp plugin or extension packaging works, if such a system exists, and what Skill Issue would need to implement to deliver skills, Markdown guidance, scripts, a standalone CLI, configuration, references, assets, and supporting files as one functional bundled experience.

**Scope:** Current official Warp documentation and primary Warp repositories covering local Agent Mode/Oz, Agent Skills, Rules, Warp Drive objects, MCP servers, Workflows, Notebooks, terminal configuration, themes, launch configurations, the Oz CLI and cloud agents, team sharing, discovery, invocation, installation, updates, trust, and permissions.

**Exclusions:** Local Warp product configuration; undocumented behavior inferred from a local installation; general terminal shell configuration except where Warp officially documents integration; third-party summaries; implementation of the proposed Skill Issue bundle.

## Findings

### Finding 1: Warp has native Agent Skills, but no documented general-purpose plugin package format

Warp's current extensibility is split across purpose-specific primitives: Agent Skills for reusable agent instructions and files, MCP servers for external tools, Warp Drive objects for saved/shared knowledge and commands, repository Rules, and file-based terminal customizations. The official docs do not describe a unified plugin manifest, arbitrary in-app extension API, installable extension archive, dependency resolver, or third-party extension marketplace that atomically installs all of those primitives. The historical official plugin discussion records an aspiration to become more extensible, but it is too old to establish current support. Therefore the defensible current classification is **no documented general-purpose Warp plugin packaging system**, rather than a claim that no internal or future mechanism exists.

**Evidence:** Warp's current [capabilities overview](https://docs.warp.dev/agent-platform/capabilities/) enumerates Skills, Rules, MCP, profiles, and other agent capabilities individually. The official [Skills documentation](https://docs.warp.dev/agent-platform/capabilities/skills/) defines directory-based `SKILL.md` units, while [MCP](https://docs.warp.dev/agent-platform/capabilities/mcp/), [Warp Drive](https://docs.warp.dev/knowledge-and-collaboration/warp-drive/), and [Custom Themes](https://docs.warp.dev/terminal/appearance/custom-themes) each have separate install/configuration flows. The official `warpdotdev/oz-skills` repository instructs users to copy folders and currently shows no GitHub releases or packages ([repository](https://github.com/warpdotdev/oz-skills)). The only official page found explicitly discussing generic plugins is the 2021 feature discussion [#435](https://github.com/warpdotdev/Warp/discussions/435), which does not document a shipped SDK or package manager.

**Implication:** Skill Issue cannot target a single Warp-native plugin artifact today. It should define its own versioned bundle manifest and installer/CLI, then project the bundle into Warp's supported primitives. Marketing and documentation should call this a Warp bundle or integration, not imply installation through a native Warp plugin marketplace.

### Finding 2: An Agent Skill directory is the strongest native unit for Skill Issue's files

Warp Agent Skills can carry a `SKILL.md` plus scripts, templates, configuration, and other supporting resources in the same directory. The agent discovers available skill metadata, loads full instructions when relevant, can invoke a skill automatically from natural language, or can invoke it explicitly as `/{skill-name}`. Skills can be project-scoped or global, and Warp scans several cross-harness directory conventions, with `.agents/skills/` recommended. This is a close match for Skill Issue's Markdown guidance, scripts, configuration, references, assets, and supporting files, provided `SKILL.md` explicitly references the files it needs.

**Evidence:** The official [Skills page](https://docs.warp.dev/agent-platform/capabilities/skills/) documents `SKILL.md` frontmatter, automatic discovery, slash-command invocation, arguments, project and home scopes, and supporting scripts/templates/configuration referenced by relative path. It recommends `.agents/skills/` and also scans `.warp/skills/`, `.codex/skills/`, `.claude/skills/`, and other provider directories. The official [oz-skills repository](https://github.com/warpdotdev/oz-skills) confirms the operational distribution pattern: copy a skill folder into a project's `.agents/skills/` or global `~/.agents/skills/`, then Warp detects it on the next interaction.

**Implication:** Skill Issue should make `.agents/skills/<skill-name>/` its canonical Warp-compatible payload. Each skill's scripts, references, assets, templates, and default configuration should remain inside that directory or use stable bundle-relative paths referenced from `SKILL.md`. A repository can contain multiple sibling skills, but the Skill Issue installer must place or link every folder and manage name collisions; Warp prioritizes home/global skills during background resolution and exposes duplicate names for explicit selection.

### Finding 3: A Git repository can deliver a cohesive project-scoped experience without installing into Warp account state

A repository can co-locate project skills, persistent `AGENTS.md` guidance, repository YAML Workflows, executable scripts/CLI source, references, and assets. Warp discovers skills and rules from the working repository and exposes repository workflows to anyone who clones it. This gives Skill Issue a functional, version-controlled bundle surface with normal Git distribution and updates. It is cohesive at repository scope, although it is a composition of recognized files rather than an installed Warp plugin.

**Evidence:** Warp discovers project skills in repository directories such as `.agents/skills/` and recommends committing them for the team ([Skills](https://docs.warp.dev/agent-platform/capabilities/skills/)). Project Rules live in `AGENTS.md` or supported equivalents and apply by directory precedence ([Rules](https://docs.warp.dev/agent-platform/capabilities/rules/)). Repository workflows live at `.warp/workflows/`, automatically load into Warp's workflow menu, and are available to anyone who clones the repository ([YAML Workflows](https://docs.warp.dev/terminal/entry/yaml-workflows)). Warp's [Skills via Oz CLI](https://docs.warp.dev/reference/cli/skills/) can identify a skill by `owner/repo:skill-name` or `owner/repo:path/to/SKILL.md` when the repository is accessible to the environment.

**Implication:** The simplest supported Skill Issue distribution is a Git repository with `.agents/skills/`, optional `AGENTS.md`, optional `.warp/workflows/`, the standalone CLI and scripts, and normal reference/asset directories. Installation can be as small as cloning the repository for project use. Updates then follow Git or a package manager used for the standalone CLI. Global cross-project installation still needs Skill Issue's installer to copy or link the skill folders into `~/.agents/skills/`.

### Finding 4: Warp cloud saved agents can compose multiple skills and runtime resources, but they are account/team configuration rather than portable plugin packages

For cloud execution, a saved Oz agent is the closest Warp-native representation of one configured experience: it can include a prompt, one or more skills, harness, model, environment, and secrets. Environments separately define repositories, container image, and setup commands. This can make multiple Skill Issue components cooperate in cloud runs, but it requires Warp account/team objects and repository access; the docs do not describe exporting that saved-agent graph as a redistributable package.

**Evidence:** The official [Oz web app](https://docs.warp.dev/platform/oz-web-app/) says saved agents can include prompt, skills, harness, model, environment, and secrets, and that agents can include one or more skills in their default configuration. The official [Environments](https://docs.warp.dev/platform/environments/) page defines runtime composition from one or more repositories, a Docker image, setup commands, environment variables, and separately injected secrets. The [Oz CLI](https://docs.warp.dev/reference/cli/) can run a saved agent by ID for cloud execution, while local runs can select profiles and a skill; cloud runs do not use local agent profiles.

**Implication:** Skill Issue can offer an optional cloud recipe that creates a named agent pointing at its repository skills and an environment that installs the standalone CLI. This should be treated as a second-stage, account-scoped setup flow. Since no authoritative export/import package for saved agents was found, automated provisioning of named agents should remain caveated until Skill Issue validates a supported API/CLI creation path for every required field.

### Finding 5: Warp Drive is shared synced content, not executable plugin packaging

Warp Drive stores and immediately syncs Workflows, Notebooks, Prompts, Rules, and Environment Variables for personal or team use. Notebooks combine Markdown with runnable shell blocks; notebooks can embed Workflows; Agent Mode can use Drive objects as context; and the Oz CLI can reference saved prompts, workflows, notebooks, and rules by ID. These objects can cooperate, but Drive import/export is type-specific and incomplete, and importing does not install arbitrary scripts, CLI binaries, themes, or skill directories.

**Evidence:** [Warp Drive](https://docs.warp.dev/knowledge-and-collaboration/warp-drive/) documents immediate sync, folders, team/direct/link sharing, inherited permissions, directory import, and its conversion rules: Workflows use YAML, Notebooks use Markdown, while Prompt import and Environment Variable import are unsupported. [Notebooks](https://docs.warp.dev/knowledge-and-collaboration/warp-drive/notebooks/) can contain runnable shell snippets and embed existing Workflows. [Agent Mode Context](https://docs.warp.dev/knowledge-and-collaboration/warp-drive/agent-mode-context/) lists Workflows, Notebooks, variables, Rules, and MCP servers as context sources. The [Warp Drive CLI context reference](https://docs.warp.dev/reference/cli/warp-drive/) supports saved-prompt IDs and `<workflow:id>`, `<notebook:id>`, and `<rule:id>` references.

**Implication:** Skill Issue may ship an optional `warp-drive/` export directory containing supported Workflow YAML and Notebook Markdown for manual directory import, plus instructions or a bootstrap skill to connect imported objects. It cannot rely on that import to install the executable CLI, scripts outside notebooks, prompts, environment variables, MCP credentials, themes, or skills. Drive links are useful for distribution and updates of shared objects, but they create Warp-account permissions and team-policy dependencies.

### Finding 6: MCP supports plugin-like tools and multi-server composition, with explicit install, credential, and permission boundaries

Warp describes MCP servers as plugin-like modules for tools and data. Users can paste a JSON object containing multiple servers, and Warp adds all defined servers. Team sharing scrubs sensitive `env` values; recipients explicitly install the shared configuration and supply missing values. The Oz CLI can attach multiple MCP servers by UUID, inline JSON, or file. MCP configuration syncs between machines on the same Warp account, but referenced environment-variable values do not. Agent profiles and team policy independently control whether MCP calls are allowed or require approval.

**Evidence:** The official [MCP page](https://docs.warp.dev/agent-platform/capabilities/mcp/) documents CLI-command and remote URL servers, multiple-server JSON input, start/stop management, out-of-box servers, team sharing, automatic scrubbing, recipient installation, and credential prompts. [MCP via Oz CLI](https://docs.warp.dev/reference/cli/mcp-servers/) supports repeated `--mcp` values and account/team UUIDs, inline JSON, or a file path; it explicitly states that server configuration syncs but environment variables do not. [Profiles & Permissions](https://docs.warp.dev/agent-platform/capabilities/agent-profiles-permissions/) adds MCP allowlists and denylists, with denylists forcing approval.

**Implication:** Skill Issue should ship an MCP configuration template and let its CLI invoke `oz ... --mcp <file>` for per-run use, or guide the user through Warp's MCP Add/install flow for persistent use. The installer must never bake secrets into the distributable bundle. It should prompt for credentials, validate server executables and working directories, and explain the Warp approval boundary. A skill may call or guide use of MCP tools, but installing the skill alone does not register or authorize those servers.

### Finding 7: Workflows, themes, launch configurations, and settings are separate file/config channels, not extensions

Warp's terminal customization surfaces are file-based and independently discovered. Repository or local YAML Workflows expose parameterized commands; custom themes are YAML files in OS-specific locations; launch configurations are YAML files that can open panes and run commands. Settings Sync covers most cloud-backed settings but explicitly excludes custom themes and custom keybindings. These channels can improve a bundled experience, but none provides executable extension code or a manifest tying the channels together.

**Evidence:** [YAML Workflows](https://docs.warp.dev/terminal/entry/yaml-workflows) documents local and `.warp/workflows/` repository locations and automatic loading. [Custom Themes](https://docs.warp.dev/terminal/appearance/custom-themes) documents manual download or cloning the official theme repository into Warp's theme path. [Launch Configurations](https://docs.warp.dev/terminal/sessions/launch-configurations) documents OS-specific YAML locations and a `commands` field. [Settings Sync](https://docs.warp.dev/terminal/more-features/settings-sync) says most settings sync through Warp's cloud, while custom themes and custom keybindings do not.

**Implication:** Skill Issue should make repository workflows optional ergonomic entry points that call its standalone CLI or invoke documented tasks. It may provide optional theme and launch-config files, but its installer must place them in OS-specific paths, preserve user files, and expose separate update/uninstall behavior. These files should not be presented as evidence of a Warp extension system.

### Finding 8: Invocation and discovery are strong; installation and updates remain the missing packaging layer

Warp can discover repository/global skills automatically, invoke them from natural language or slash commands, search Workflows and Drive objects, and run repository skills locally or in cloud environments through Oz. However, the official pre-built skill collection uses manual copying, and no native dependency/version/update mechanism is documented for third-party skill bundles. Warp's own CLI has package-manager update paths, but that updates Oz, not Skill Issue content.

**Evidence:** [Skills](https://docs.warp.dev/agent-platform/capabilities/skills/) covers automatic discovery and `/{skill-name}` invocation. [Skills via Oz CLI](https://docs.warp.dev/reference/cli/skills/) covers repository-qualified skill selection. The official [oz-skills README](https://github.com/warpdotdev/oz-skills) instructs users to copy folders and says detection occurs on the next interaction. The [Oz CLI reference](https://docs.warp.dev/reference/cli/) documents Homebrew and Linux package-manager installation/update for Oz itself and warns that direct macOS CLI downloads do not auto-update.

**Implication:** Skill Issue's standalone CLI must own bundle install, update, uninstall, and diagnostics. A practical contract is: install a pinned bundle release; place or link skills into project `.agents/skills/` or global `~/.agents/skills/`; optionally install `.warp/workflows/`, themes, and launch configurations; preserve a manifest of installed files and versions; update atomically; detect conflicts; and provide `doctor` output for Warp discovery, executable permissions, missing runtimes, MCP setup, and credentials. Git-only users can use clone/pull, but global installations need the CLI to avoid stale manual copies.

### Finding 9: Trust is enforced at execution time, not conferred by bundle installation

Skills are instructions and may reference executable scripts, while Workflows and launch configurations can run shell commands. Warp's agent permissions control reads, diffs, plans, commands, terminal interaction, and MCP access; command denylists override ordinary allow behavior, and enterprise admins can enforce policies. Warp Drive viewers can execute shared Workflows and use shared environment variables under Drive permissions. Therefore a functional Skill Issue bundle must accommodate approvals and policy failures rather than assume an installed bundle is trusted.

**Evidence:** [Profiles & Permissions](https://docs.warp.dev/agent-platform/capabilities/agent-profiles-permissions/) documents autonomy levels, command allow/deny regular expressions, MCP allow/deny controls, and denylist precedence. [Roles and permissions](https://docs.warp.dev/enterprise/team-management/roles-and-permissions) documents organization-enforced agent policies, sharing controls, and team-resource access. [Warp Drive](https://docs.warp.dev/knowledge-and-collaboration/warp-drive/) documents view/edit/full-access capabilities, including Workflow execution and environment-variable use. [MCP](https://docs.warp.dev/agent-platform/capabilities/mcp/) documents explicit recipient installation and entry of scrubbed secrets.

**Implication:** Skill Issue should publish a clear permission manifest for each skill and script, keep defaults least-privileged, avoid asking users to weaken global denylists, and make approval-required steps recoverable. Installer output should show every destination and executable, verify checksums or signatures for released assets, request confirmation before modifying global Warp paths, and avoid installing credentials. Team deployment guidance must account for admin-enforced restrictions and Drive sharing policies.

### Finding 10: One bundled experience is feasible through composition, with a Skill Issue bootstrap layer

One coherent user experience is possible if "bundle" means a versioned repository/installer that composes supported Warp primitives. The native center should be one bootstrap or router skill that exposes a stable slash-command/natural-language entry point and delegates to sibling skills, scripts, references, the standalone CLI, and optional MCP tools. Project users can clone the repository; global users can run the Skill Issue installer; cloud users can attach multiple repository skills to a saved Oz agent and configure an environment. The experience cannot currently be delivered as one documented Warp marketplace install that atomically provisions local files, Drive objects, account-scoped MCP, credentials, profiles, and cloud resources.

**Evidence:** Skill directories support all required supporting file classes and direct invocation ([Skills](https://docs.warp.dev/agent-platform/capabilities/skills/)); repositories provide qualified local/cloud skill selection ([Skills via Oz CLI](https://docs.warp.dev/reference/cli/skills/)); saved cloud agents can attach multiple skills and runtime defaults ([Oz web app](https://docs.warp.dev/platform/oz-web-app/)); MCP files can be attached per run ([MCP Servers via Oz CLI](https://docs.warp.dev/reference/cli/mcp-servers/)); repository Workflows provide discoverable terminal shortcuts ([YAML Workflows](https://docs.warp.dev/terminal/entry/yaml-workflows)). The separate installation and permission flows in those same sources establish why a bootstrap layer is required.

**Implication:** Skill Issue should implement the following Warp adapter: (1) a canonical `.agents/skills/skill-issue/` bootstrap skill; (2) sibling focused skills, each self-contained with references/assets/scripts; (3) an independently packaged `skill-issue` executable; (4) optional `.warp/workflows/` shortcuts that call the CLI; (5) optional `AGENTS.md` guidance only where persistent project rules are genuinely required; (6) MCP JSON templates and guided registration/per-run flags; (7) optional Warp Drive import exports; and (8) a bundle manifest used by the CLI for install/update/uninstall/doctor. This yields a functional bundled experience while respecting Warp's actual extension boundaries.

## Notes

- **Unsupported:** No authoritative current documentation was found for a general Warp extension SDK, third-party executable plugin manifest, extension dependency graph, or all-component marketplace installer. The conclusion is limited to documented public behavior as of July 19, 2026.
- **Caveat:** The Oz web app has "Suggested" skills, and Warp maintains `warpdotdev/oz-skills`, but official usage instructions are copy-based. This is a curated discovery surface, not verified package installation/update infrastructure.
- **Caveat:** Saved cloud agents are a meaningful multi-component composition surface, but no authoritative portable export/import artifact for a saved agent plus environment, skills, MCP, and secrets was found during this assignment.
- **Caveat:** Warp Drive directory import preserves supported folder structure, but Prompts and Environment Variables cannot currently be imported; arbitrary supporting files are outside the documented Drive object model.
- **Dead end:** Historical GitHub discussion #435 is useful only to show that general plugins were once a feature request; it cannot validate current availability.
- Useful follow-up search terms: `Warp named agent API create`, `Oz agent configuration export`, `Warp MCP noninteractive install`, `Warp skill registry install`, `Warp extension manifest`.
