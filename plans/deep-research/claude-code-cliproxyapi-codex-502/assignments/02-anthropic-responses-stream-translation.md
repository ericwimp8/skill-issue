# Anthropic Messages to OpenAI Responses Stream Translation

## Assignment

**Goal:** Trace the internet-visible protocol path for Claude Code 2.1.205 sending a streaming Anthropic-compatible `POST /v1/messages?beta=true` request through router-for-me/CLIProxyAPI 7.2.91 to the Codex OAuth executor for `gpt-5.6-sol`, with particular attention to request translation, SSE lifecycle, terminal failures, retry boundaries, cancellation, keep-alives, and the reported late `502`/`server_error` followed by a successful identical rerun.

**Scope:** Internet sources only. The implementation source of CLIProxyAPI at tag `v7.2.91`, official Anthropic Messages streaming and error documentation, official OpenAI Responses streaming and cancellation documentation, the Go HTTP contract, and directly relevant public issue reports were inspected.

**Exclusions:** Credential acquisition or extraction, local configuration mutation, destructive recovery steps, incident-specific log forensics unavailable in the supplied evidence, and claims about private Claude Code 2.1.205 internals that cannot be validated from an official source or reproducible public artifact.

## Findings

### Finding 1: `/v1/messages?beta=true` selects the Claude-compatible surface, while the body selects streaming

CLIProxyAPI registers `POST /v1/messages` directly to `ClaudeCodeAPIHandler.ClaudeMessages`. The handler reads the JSON body and chooses streaming only when the body field `stream` exists and is true; the `beta=true` query parameter does not select the stream path. The query remains observable to generic model-router or interceptor code, but the Codex executor builds a new fixed upstream URL ending in `/responses` and does not append the inbound query. [`server.go` route registration](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/api/server.go#L535), [`ClaudeMessages` stream dispatch](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/claude/code_handlers.go#L70), and [`CodexExecutor.ExecuteStream` upstream URL construction](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L1396) establish this path.

**Evidence:** Anthropic's documented beta mechanism is the `anthropic-beta` request header, or an SDK `betas` parameter that produces that header; the official API examples still use `POST /v1/messages`. The documented mechanism does not assign beta semantics to a `?beta=true` query parameter. [Anthropic beta headers](https://platform.claude.com/docs/en/api/beta-headers). In CLIProxyAPI, inbound query values are copied into executor options for routers/interceptors, but `CodexExecutor.ExecuteStream` never consumes `opts.Query`; its upstream request is created for `baseURL + "/responses"`. [`modelExecutionQuery`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/model_execution.go#L205), [`CodexExecutor.ExecuteStream`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L1448).

**Implication:** `?beta=true` is compatible with this merged route because Gin ignores it for route matching, but it is not forwarded as an OpenAI Responses parameter. Any beta behavior that survives this bridge must be represented in body fields that the translator understands, in plugin behavior, or in separately propagated headers. The core Codex header builder forwards selected Codex headers such as `X-Codex-Beta-Features`, but not `Anthropic-Beta`. [`applyCodexHeadersFromSources`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L2007).

### Finding 2: The request is structurally rewritten from Messages objects to Responses input items

For a Codex-backed model, the executor invokes the translation registry from the incoming Claude format to the internal `codex` format, which is an OpenAI Responses-shaped request. Top-level Claude `system` text becomes a Responses `developer` message; Claude user/assistant text becomes `input_text`/`output_text`; images become `input_image`; `tool_use` and `tool_result` blocks become `function_call` and `function_call_output`; and Claude tool declarations become Responses function tools. [`ConvertClaudeRequestToCodex`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/translator/codex/claude/codex_claude_request.go#L23).

**Evidence:** The translator maps `thinking.budget_tokens` or adaptive `output_config.effort` into `reasoning.effort`, sets `reasoning.summary="auto"`, maps Claude `speed="fast"` to Responses `service_tier="priority"`, and hard-codes `stream=true`, `store=false`, and `include=["reasoning.encrypted_content"]`. It defaults reasoning effort to `medium` when the Claude request provides no recognized thinking selection. [`codex_claude_request.go` parameter mapping](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/translator/codex/claude/codex_claude_request.go#L297). The executor then forces the routed model name, removes `previous_response_id`, `prompt_cache_retention`, `safety_identifier`, and `stream_options`, normalizes instructions and parallel tool calls, and sends the request to the Codex OAuth base URL plus `/responses`. [`codex_executor.go`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L1421).

**Implication:** This is a semantic adapter, not a transparent reverse proxy. Parameters without explicit mapping do not automatically cross the boundary. In this tag's Claude-to-Codex translator, `max_tokens`, `temperature`, `top_p`, and `stop_sequences` are not mapped to Responses fields such as `max_output_tokens`; configuration or plugins could still alter the translated payload later, but the core translator itself does not preserve those fields. The long runtime therefore cannot be interpreted using the incoming Claude `max_tokens` value without inspecting the actual translated upstream body.

### Finding 3: Codex OAuth uses a Responses SSE request with Codex-specific transport headers

When the selected credential is a Codex OAuth credential, the default upstream base is `https://chatgpt.com/backend-api/codex`; `ExecuteStream` posts to its `/responses` child. It supplies the OAuth bearer token, `Accept: text/event-stream`, `Connection: Keep-Alive`, a Codex originator/user agent, and, for OAuth, the ChatGPT account ID when available. [`CodexExecutor.ExecuteStream`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L1403), [`applyCodexHeadersFromSources`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L2007).

**Evidence:** OpenAI's current model guide identifies `gpt-5.6-sol` as the flagship GPT-5.6 model and recommends the Responses API for reasoning and tool-calling workflows. [OpenAI GPT-5.6 guidance](https://developers.openai.com/api/docs/guides/latest-model). CLIProxyAPI does not forward Claude's wire protocol to an Anthropic server; it converts the payload, performs a new upstream HTTP request, parses OpenAI Responses events, and converts them back.

**Implication:** A failure labeled `server_error` can originate from the OpenAI Responses lifecycle even though Claude Code's visible endpoint and final event shape are Anthropic-compatible. The visible `/v1/messages` path alone does not identify the upstream provider.

### Finding 4: Normal Responses lifecycle events become Anthropic Messages lifecycle events

The response translator is stateful. `response.created` becomes `message_start`; reasoning summary parts become a Claude `thinking` content block and deltas; output-text events become a Claude text block and `text_delta`; function-call events become `tool_use` plus `input_json_delta`; and a terminal `response.completed` or `response.incomplete` becomes `message_delta` followed by `message_stop`. [`ConvertCodexResponseToClaude`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/translator/codex/claude/codex_claude_response.go#L52).

**Evidence:** Anthropic documents the successful stream order as `message_start`, zero or more content blocks, one or more `message_delta` events, and final `message_stop`, with optional `ping` events. It separately allows an `event: error` inside an already-open stream. [Anthropic streaming event flow and error event](https://platform.claude.com/docs/en/build-with-claude/streaming#event-types). OpenAI separately defines `response.created`, `response.completed`, `response.incomplete`, and `response.failed`; the documented `response.failed` example carries `response.status="failed"` and `response.error.code="server_error"`. [OpenAI Responses `response.failed`](https://platform.openai.com/docs/api-reference/responses-streaming/response/refusal/done#responses-streaming-response-failed).

**Implication:** `response.incomplete` is treated by CLIProxyAPI as a completed Anthropic message with a stop reason, whereas `response.failed` is a terminal error. A valid successful Claude stream must reach `message_stop`; an `event: error` is an abnormal terminal outcome even though the underlying HTTP exchange may remain `200 OK`.

### Finding 5: `response.failed` is intercepted before normal response translation and converted to a proxy stream error

CLIProxyAPI's Codex executor examines each upstream `data:` line before calling the normal response translator. Both upstream `type="error"` and `type="response.failed"` are converted into a `statusErr` and emitted as `StreamChunk{Err: ...}`; the raw failure event is not forwarded through `ConvertCodexResponseToClaude`. [`codexTerminalFailureErr`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L143), [`ExecuteStream` terminal interception](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L1520).

**Evidence:** For `response.failed`, the executor extracts `response.error`; if that error contains an explicit `status_code` or `status` from 400 through 599 it uses it. Otherwise it maps known request/auth/rate-limit categories and defaults every remaining failure, including a plain `code="server_error"`, to HTTP 502. [`codexTerminalFailureStatus`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L154). The Claude handler receives this terminal stream error and emits `event: error` with a Claude error envelope. [`forwardClaudeStream`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/claude/code_handlers.go#L356).

**Implication:** The reported SSE error representing a `502` is an exact supported outcome of a late OpenAI Responses `response.failed` with `server_error` and no explicit upstream status. It does not require the original OpenAI HTTP response to have had status 502. The proxy synthesizes 502 as the failure's status classification, then serializes the failure as an Anthropic-compatible stream error.

### Finding 6: A late stream failure remains HTTP 200 on the wire even though CLIProxyAPI classifies it as 502

The Claude handler delays downstream SSE headers until it receives the first translated chunk. On that first chunk it sets `Content-Type: text/event-stream`, writes and flushes the chunk, thereby committing the HTTP status. If a later error arrives, `forwardClaudeStream` calls `c.Status(502)` and writes `event: error`, but the status line is already committed. [`handleStreamingResponse`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/claude/code_handlers.go#L257), [`forwardClaudeStream`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/claude/code_handlers.go#L356).

**Evidence:** Go's `http.ResponseWriter` contract states that the first `Write` implicitly writes `200 OK` when no status was written, and later attempts to change the normal 2xx-5xx status have no effect. [`net/http.ResponseWriter`](https://pkg.go.dev/net/http#ResponseWriter). Anthropic likewise documents that an SSE error can occur after the API has returned 200, and that such a mid-stream error does not use normal HTTP retry handling. [Anthropic API errors](https://platform.claude.com/docs/en/api/errors#http-errors).

**Implication:** The most precise interpretation of the incident is: the downstream HTTP request was successfully established, at least one SSE payload committed it, and the terminal application event represented a 502-class upstream failure. A packet capture may show `HTTP 200` while proxy/client logs describe `502`; those observations are compatible rather than contradictory.

### Finding 7: Automatic retry is intentionally bounded to failures before the first payload

CLIProxyAPI has two pre-payload retry layers. The auth manager reads until the first non-empty stream payload; an error before that boundary can trigger unauthorized refresh, alternate upstream model/credential selection, or configured request retry. The handler layer can also rerun eligible errors before any downstream payload when `streaming.bootstrap-retries` is positive. [`readStreamBootstrap`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/cliproxy/auth/conductor.go#L1743), [`Manager.ExecuteStream`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/cliproxy/auth/conductor.go#L2400), [`executeStreamWithAuthManagerFormats`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/handlers.go#L1330).

**Evidence:** The example configuration sets `request-retry: 3` and describes retryable statuses including 408, 500, 502, 503, and 504, while `streaming.bootstrap-retries` defaults to 0 unless explicitly enabled. [`config.example.yaml`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/config.example.yaml#L116), [`streaming` example](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/config.example.yaml#L191). Once the auth manager has returned a payload-bearing stream, `wrapStreamResult` only marks a later error against the selected auth and forwards it; it does not restart the request. [`wrapStreamResult`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/cliproxy/auth/conductor.go#L1775). The handler's `ForwardStream` writes the terminal error, flushes, cancels, and returns. [`ForwardStream`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/stream_forwarder.go#L32).

**Implication:** If OpenAI's normal early `response.created` event was translated to `message_start`, the safe-retry window closed immediately, even if no user-visible text had appeared and the model then ran for 6 minutes 39 seconds. Replaying after a late failure requires a new client request. The identical rerun succeeding is consistent with a transient upstream `server_error`; it does not show that CLIProxyAPI retried the failed stream internally.

### Finding 8: Anthropic SDK HTTP retries and mid-stream recovery are distinct mechanisms

Anthropic documents that official SDKs automatically retry transient connection, rate-limit, and 5xx failures twice by default, but explicitly states that an SSE error after a 200 response does not follow those standard mechanisms. For an interrupted stream, the documented recovery is a new continuation request constructed from the partial response; tool-use and thinking blocks cannot be partially recovered in the same way as plain text. [Anthropic API retry behavior](https://platform.claude.com/docs/en/api/errors#http-errors), [Anthropic streaming error recovery](https://platform.claude.com/docs/en/build-with-claude/streaming#error-recovery).

**Evidence:** CLIProxyAPI's own automatic retry boundary matches this distinction: before-first-payload failures may be retried; after-first-payload failures become a terminal SSE error. Public reports in CLIProxyAPI [#2189](https://github.com/router-for-me/CLIProxyAPI/issues/2189) and OpenAI Codex [#24973](https://github.com/openai/codex/issues/24973) independently illustrate that stream-event failures and HTTP-level failures can take different retry paths, although those issue reports do not establish the behavior of Claude Code 2.1.205.

**Implication:** A blanket “retry all 502s” description is incomplete. The supported boundary is whether a response payload has committed the stream. Retrying after partial tool or text output also carries duplication/continuation semantics that cannot be solved by transparently replaying the same request inside the proxy.

### Finding 9: Keep-alives are optional SSE comments and begin only after the first translated payload

CLIProxyAPI's streaming keep-alive is disabled by default. When configured, `ForwardStream` periodically writes `: keep-alive\n\n`, an SSE comment, and flushes it. [`StreamingKeepAliveInterval`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/handlers.go#L215), [`ForwardStream` keep-alive branch](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/stream_forwarder.go#L45).

**Evidence:** SSE comments are valid protocol records and are conventionally ignored by event consumers while keeping the connection active. [WHATWG Server-Sent Events](https://html.spec.whatwg.org/dev/server-sent-events.html). Anthropic's native stream examples use named `ping` events, but its event model also requires clients to tolerate additional event types. [Anthropic ping and other events](https://platform.claude.com/docs/en/build-with-claude/streaming#ping-events).

**Implication:** CLIProxyAPI keep-alive comments preserve transport liveness without becoming Claude content. However, the ticker is started inside `ForwardStream`, and the Claude handler does not call `ForwardStream` until the first translated chunk has arrived and been flushed. A stall before OpenAI emits the first translatable event receives no downstream proxy heartbeat; a long reasoning gap after `message_start` can receive comments when keep-alive is enabled. Keep-alive prevents some idle-path timeouts but cannot prevent an OpenAI `response.failed` server error.

### Finding 10: Cancellation is propagated by context and connection closure, not by the Responses cancel method

The Claude handler ties its execution context to the incoming HTTP request context. Client disconnect or cancellation cancels the proxy context; the Codex request is created with that context, and the executor stops scanning or forwarding when the context is done. [`GetContextWithCancel`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/handlers.go#L538), [`handleStreamingResponse` disconnect branch](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/claude/code_handlers.go#L285), [`CodexExecutor.ExecuteStream`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L1450).

**Evidence:** The translated body sets `store=false` and does not set `background=true`. OpenAI's explicit `POST /responses/{response_id}/cancel` operation is supported only for responses created with `background=true`. [OpenAI Cancel a response](https://developers.openai.com/api/reference/resources/responses/methods/cancel).

**Implication:** Aborting Claude Code should close/cancel the in-flight HTTP work through Go context propagation, but CLIProxyAPI does not issue a separate Responses cancel API call for this streaming path. The protocol-level terminal status for client cancellation may therefore be a closed connection rather than a final Anthropic `message_stop` or `error` event.

### Finding 11: Exit 1 with empty stderr is plausible, but the exact Claude Code 2.1.205 channel mapping is unsupported

An SSE error after headers is an application event carried in the HTTP response body, not an HTTP transport error. The network stack can therefore report a successful `200` connection while the Claude stream parser treats `event: error` as a failed turn and the process exits nonzero. Nothing in HTTP or SSE requires that event payload to be mirrored to the operating-system stderr stream.

**Evidence:** Anthropic explicitly documents error events after HTTP 200. [Anthropic API errors](https://platform.claude.com/docs/en/api/errors#http-errors). Claude Code's official CLI reference documents print mode and `text`, `json`, and `stream-json` output formats, but does not provide a contract that all API stream failures are written to stderr. [Claude Code CLI reference](https://docs.anthropic.com/en/docs/claude-code/cli-usage). Public Claude Code reports [#11715](https://github.com/anthropics/claude-code/issues/11715) and [#33949](https://github.com/anthropics/claude-code/issues/33949) show that streaming failures can be handled or logged separately from normal terminal output, but they concern other versions and are not authoritative evidence for 2.1.205.

**Implication:** `exit 1` plus empty captured stderr is protocol-plausible if Claude Code consumed the SSE error internally, rendered it through another output/logging channel, or suppressed it in the selected print format. The specific statement “Claude Code 2.1.205 always maps this event to exit 1 without stderr” could not be validated and should remain unsupported without the exact invocation, output mode, debug log, and raw captured stream.

### Finding 12: The incident pattern is best explained by a late transient upstream failure, with two material alternatives

The strongest end-to-end explanation is: the request was accepted and the stream committed, `gpt-5.6-sol` ran for approximately 6 minutes 39 seconds, OpenAI emitted a terminal `response.failed` or `error` event whose error was classified as `server_error`, CLIProxyAPI mapped the unspecified server failure to status 502, and the already-open Anthropic SSE stream ended with `event: error`. A new identical request then succeeded because the server failure was transient.

**Evidence:** OpenAI defines `response.failed` with a `server_error` example; CLIProxyAPI explicitly converts such an event to 502 and then to a Claude stream error; Anthropic explicitly allows stream error events after HTTP 200. These three independent sources align at every protocol boundary. [OpenAI `response.failed`](https://platform.openai.com/docs/api-reference/responses-streaming/response/refusal/done#responses-streaming-response-failed), [`codexTerminalFailureStatus`](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L154), [Anthropic error events](https://platform.claude.com/docs/en/build-with-claude/streaming#error-events).

**Implication:** Two alternatives remain materially possible from the supplied facts. First, OpenAI may have returned an actual HTTP 502 before any upstream SSE event; after proxy retry exhaustion this would normally surface as a real downstream non-stream JSON error, unless another layer re-encoded it as SSE. Second, an abrupt upstream EOF would be converted by this executor to a request-scoped 408-style “stream closed before response.completed” error, not the documented 502/server-error path. The observed SSE 502/server-error shape makes the late `response.failed`/`error` explanation substantially better supported than either alternative.

## Notes

- The exact failed request body, raw SSE transcript, response ID, downstream response headers, CLIProxyAPI streaming configuration, Claude Code invocation flags, and debug log were not available. Those absences prevent determining whether `response.created` arrived immediately, whether any content/tool event preceded failure, and whether proxy keep-alives were enabled.
- The query string `?beta=true` is visible in public Claude Code/CLIProxyAPI captures, including CLIProxyAPI issue [#1620](https://github.com/router-for-me/CLIProxyAPI/issues/1620), but no official Anthropic source found assigns beta-feature meaning to that query. The official beta contract is header-based.
- Public issue reports are useful for matching symptoms but were not treated as proof of implementation behavior. CLIProxyAPI tag source and official protocol specifications control the findings above.
- Useful search terms for further incident correlation: response ID, `response.failed`, `server_error`, `upstream_error`, `API_RESPONSE_TIMESTAMP`, `X-Client-Request-Id`, `request-retry`, `streaming.bootstrap-retries`, `streaming.keepalive-seconds`, and `stream closed before response.completed`.
