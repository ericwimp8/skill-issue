# Operational Mitigations and Safety for Long-Running Claude Code–CLIProxyAPI Streams

## Assignment

**Goal:** Identify durable, evidence-backed operational mitigations for long-running Claude Code 2.1.205 requests routed through CLIProxyAPI 7.2.91 to `gpt-5.6-sol` over the OpenAI Responses streaming path, where four requests succeeded, one streamed request failed after 6m39s with downstream `502`/`server_error`, and the same workload succeeded on rerun in 2m48s.

**Scope:** Internet-only inspection of CLIProxyAPI deployment documentation and exact v7.2.91 source, current release deltas, authoritative Claude Code timeout/retry guidance, OpenAI Responses/error/request-ID guidance, SSE standards, and representative reverse-proxy timeout semantics. The investigation covers timeout layers, keep-alives, proxy/server limits, concurrency/rate limits, retry placement and caps, health checks, logging and request IDs, upgrade/rollback evidence, workload decomposition, and repeatability testing.

**Exclusions:** No credential inspection or extraction; no copying OAuth material; no destructive reset; no mutation of user configuration; no unsupported authentication workarounds; no source or deployment changes; no claim that an upstream or local layer caused this incident without matching request-level evidence.

## Findings

### Finding 1 — The observed shape is most consistent with a terminal upstream Responses failure, not a demonstrated local timeout

OpenAI defines `response.failed` as a terminal streaming event whose response can contain `error.code: "server_error"`. CLIProxyAPI v7.2.91 explicitly recognizes `response.failed`; when the embedded error lacks an HTTP status and is not classified as a 400/401/403/404/429 class, it maps the event to HTTP 502. It then translates that terminal failure into a Claude-compatible SSE `event: error`. This means a downstream 502 can be CLIProxyAPI's representation of an upstream terminal event even though the original streaming HTTP response began successfully with status 200.

**Evidence:** OpenAI's streaming reference includes a `response.failed` example with `error.code: "server_error"` and `status: "failed"` ([Responses streaming events](https://developers.openai.com/api/reference/resources/responses/subresources/streaming-events)). CLIProxyAPI parses `response.failed` at [`codex_executor.go` lines 179-199](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L179), defaults unclassified terminal failures to 502 at [lines 154-176](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L154), and emits the Claude SSE error at [`code_handlers.go` lines 356-377](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/claude/code_handlers.go#L356). The Codex stream reader surfaces the terminal error from an already-open stream at [`codex_executor.go` lines 1501-1585](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L1501).

**Implication:** If the captured failure includes the upstream `response.failed`/`server_error` body, prioritize upstream reliability, request size/duration, bounded retries, and correlation evidence. Raising a proxy timeout or sending more downstream keep-alives cannot prevent the model service from deliberately ending the stream with `response.failed`.

### Finding 2 — The known timeout layers do not explain a 6m39s terminal error by themselves

Claude Code's default whole-request deadline is 600,000 ms (10 minutes). For custom gateway connections it also enables a five-minute _idle_ timeout, which resets when bytes arrive. CLIProxyAPI's Codex streaming HTTP client is constructed with a zero client timeout, and its Gin `http.Server` sets no read, write, header, or idle deadline. A 6m39s request that continued receiving bytes and ended in a structured server error therefore did not hit the documented Claude Code 10-minute deadline or a CLIProxyAPI server deadline. An unobserved reverse proxy, load balancer, NAT, or firewall remains possible only if one exists on the actual path.

**Evidence:** Claude Code documents `API_TIMEOUT_MS=600000` and `API_FORCE_IDLE_TIMEOUT` as a five-minute no-byte watchdog on gateway connections ([environment variables](https://code.claude.com/docs/en/env-vars#variables)); its error guide says the default request timeout is 10 minutes and recommends decomposition for long tasks ([error reference, “Request timed out”](https://code.claude.com/docs/en/errors#request-timed-out)). CLIProxyAPI passes timeout `0` to `NewUtlsHTTPClient` for the stream at [`codex_executor.go` lines 1475-1478](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L1475), and `NewUtlsHTTPClient` only sets `http.Client.Timeout` when the value is positive at [`utls_client.go` lines 155-192](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/helps/utls_client.go#L155). CLIProxyAPI constructs its HTTP server with only `Addr` and `Handler` at [`server.go` lines 405-409](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/api/server.go#L405).

**Implication:** Do not raise `API_TIMEOUT_MS` as the first fix for this case. Raise it temporarily only when logs show a real 10-minute client deadline on valid progressing requests. Do not set `API_FORCE_IDLE_TIMEOUT=0` globally; disable the idle watchdog only for a controlled test that has proven legitimate gaps longer than five minutes, because removing it converts some detectable stalls into indefinite hangs.

### Finding 3 — Keep-alives are durable only against verified idle intermediaries and cannot heal the upstream stream

CLIProxyAPI can emit a standard SSE comment (`: keep-alive`) to Claude Code on a configured interval. The default is disabled. The WHATWG standard recommends a comment about every 15 seconds to protect against legacy proxies that drop quiet event streams. This heartbeat exists on the CLIProxyAPI-to-Claude-Code leg; it does not cause the OpenAI-to-CLIProxyAPI leg to emit data and does not change an upstream `response.failed` event. If Claude Code connects directly to localhost, an extra downstream heartbeat has little value unless its own idle watchdog is the measured failure.

**Evidence:** `streaming.keepalive-seconds` defaults to zero and the example suggests 15 seconds at [`config.example.yaml` lines 191-196](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/config.example.yaml#L191). CLIProxyAPI writes `: keep-alive\n\n` on the downstream writer at [`stream_forwarder.go` lines 45-62 and 116-119](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/stream_forwarder.go#L45). WHATWG advises a comment every 15 seconds or so for proxy compatibility ([SSE authoring notes](https://html.spec.whatwg.org/dev/server-sent-events.html#authoring-notes)); RFC 8895 repeats the 15-second recommendation ([RFC 8895 §8.2](https://www.rfc-editor.org/rfc/rfc8895.html#section-8.2)).

**Implication:** Treat `streaming.keepalive-seconds: 15` as a route-specific canary for observed downstream idle disconnects, not as the primary remedy for this 502. Verify that Claude Code ignores the comment normally and that its stall watchdog still behaves as intended before retaining the setting. If an nginx-like reverse proxy exists, disable response buffering on this SSE route and set its per-read timeout above the largest measured inter-event gap; nginx documents that `proxy_read_timeout` is measured between successive reads, not across the total response ([nginx proxy module](https://nginx.org/en/docs/http/ngx_http_proxy_module.html#proxy_read_timeout)).

### Finding 4 — Retrying after visible output is unsafe; one bounded layer should own retry orchestration

Claude Code 2.1.205 already retries transient server errors, overloads, timeouts, temporary 429s, and connection drops with exponential backoff, up to ten attempts by default. Since v2.1.199 it deliberately does not replay a request after visible output has streamed, because duplicate execution can repeat tool calls. CLIProxyAPI makes the same safety distinction: authentication/request retry and `streaming.bootstrap-retries` can recover only before a payload is sent; after any payload, a terminal stream error is passed downstream.

**Evidence:** Claude Code documents the ten-attempt retry policy, exponential backoff, pre-visible-output replay, and the no-retry rule after visible output ([error reference, “Automatic retries”](https://code.claude.com/docs/en/errors#automatic-retries)). CLIProxyAPI's bootstrap recovery checks `!sentPayload`, retries only while `bootstrapRetries < maxBootstrapRetries`, and commits the stream after a payload at [`handlers.go` lines 1298-1431](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/handlers.go#L1298). The configured bootstrap retry default is zero at [lines 241-250](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/handlers.go#L241). OpenAI recommends random exponential backoff with a maximum retry count and notes that unsuccessful requests still consume rate-limit budget ([rate-limit guidance](https://developers.openai.com/api/docs/guides/rate-limits#retrying-with-exponential-backoff)).

**Implication:** Keep mid-stream retry disabled. For normal interactive operation, let Claude Code remain the outer retry owner because it knows whether output/tool calls became visible. If a proxy-side bootstrap retry is required for credential rotation, canary at most one (`streaming.bootstrap-retries: 1`) and reduce the client retry cap during that test so nested retry counts do not multiply. Do not enable `CLAUDE_CODE_RETRY_WATCHDOG=1` for interactive work: the documented behavior raises non-capacity transient retries to roughly three hours and can retry capacity errors indefinitely. A manual rerun after an isolated failure is a valid transient recovery, but it is not a durable mitigation.

### Finding 5 — CLIProxyAPI retry comments overstate what a single OAuth credential can recover after streaming begins

The example configuration says `request-retry` covers 403, 408, 500, 502, 503, and 504. The concrete v7.2.91 stream path first rotates among eligible credentials/models and handles only bootstrap failures; later retries are constrained by credential cooldown and `max-retry-interval`. Transient 408/5xx results mark a credential unavailable for the transient cooldown (legacy default 60 seconds), while the example `max-retry-interval` is 30 seconds. With one operator-owned Codex OAuth credential, this machinery cannot reconstruct a response after output has started and may have no alternate credential to select.

**Evidence:** The documented defaults are `request-retry: 3`, `max-retry-interval: 30`, and legacy transient cooldown 60 seconds at [`config.example.yaml` lines 116-135](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/config.example.yaml#L116). Stream execution selects/rotates credentials at [`conductor.go` lines 2768-2835](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/cliproxy/auth/conductor.go#L2768); retry waits are rejected when they exceed `maxWait` at [lines 3638-3669](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/cliproxy/auth/conductor.go#L3638); and 408/500/502/503/504 mark an auth as a transient upstream error at [lines 4647-4653](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/cliproxy/auth/conductor.go#L4647).

**Implication:** Do not increase `request-retry`, disable cooling, or add credentials as a generic fix. More proxy retries can multiply long expensive attempts, and credential fan-out is not a supported way to evade operator/account limits. If a controlled canary changes cooldown or retry values, change one variable, cap total attempts, and revert after the test unless repeatable evidence improves.

### Finding 6 — Workload decomposition is the strongest durable mitigation for long reasoning requests

The same workload taking 6m39s on failure and 2m48s on success shows high service-time variance. Shorter, checkpointed requests reduce exposure to transient provider failure, limit the work discarded by a terminal event, and make retries cheaper. Decomposition also reduces output and context pressure, which are independent contributors to long inference time. This is more durable than extending deadlines for a monolithic request.

**Evidence:** Claude Code's timeout guidance explicitly says to break long-running work into smaller prompts ([error reference](https://code.claude.com/docs/en/errors#request-timed-out)). Its stability guide recommends `/compact`, reading oversized inputs in smaller chunks, and moving bounded work into a separate context when a conversation refills the window ([troubleshooting](https://code.claude.com/docs/en/troubleshooting#auto-compaction-stops-with-a-thrashing-error)). OpenAI's rate-limit guidance recommends reducing maximum output tokens to the expected completion size ([rate-limit guide](https://developers.openai.com/api/docs/guides/rate-limits#reduce-the-max_tokens-to-match-the-size-of-your-completions)).

**Implication:** Split long jobs into independently verifiable phases (for example: map sources, inspect one subsystem, write a checkpoint, synthesize), resume from durable outputs, and keep each model turn comfortably below the observed failure duration. Narrow the supplied context and requested output for each phase. Prefer explicit continuation from the last completed checkpoint over replaying the full monolithic request.

### Finding 7 — Smooth concurrency and preserve rate-limit evidence; the 502 alone is not proof of rate limiting

Four successes followed by one failure can reflect provider load, connection churn, or chance; it does not establish a rate limit. A true OpenAI rate limit is normally 429 and carries request/token limit headers. Nevertheless, concurrent long requests amplify token pressure and retry storms. Recent public status history also records a resolved `gpt-5.6-sol` server-overload incident on 2026-07-17, showing that model-specific upstream capacity errors are a real failure class close to this run.

**Evidence:** OpenAI lists `x-ratelimit-*` response headers and recommends logging them with `x-request-id` ([API request debugging](https://developers.openai.com/api/reference/overview#debugging-requests)). It says to pace requests, use jittered bounded backoff, and avoid continuous resubmission because failed requests count toward per-minute limits ([rate limits](https://developers.openai.com/api/docs/guides/rate-limits#retrying-with-exponential-backoff)). Claude Code recommends reducing tool-use concurrency and parallel subagents for 429 conditions ([error reference, “Request rejected (429)”](https://code.claude.com/docs/en/errors#request-rejected-429)). OpenAI's status page recorded and resolved [“Codex 5.6-sol Experiencing Increased Server-Overload Errors”](https://status.openai.com/incidents/01KXRHE25717D2WQ1WFMT2B7WZ) on 2026-07-17.

**Implication:** Start the repeatability campaign at concurrency one, then test concurrency two and four separately. Avoid simultaneous retries across sessions; add jitter and a shared attempt budget. If 429 or decreasing `x-ratelimit-remaining-*` values appear, lower concurrency and ramp rate gradually. Do not label a 502 as quota exhaustion, and do not infer undocumented OAuth concurrency limits.

### Finding 8 — Correlation and layered health checks should precede further tuning

CLIProxyAPI exposes `/healthz`, but the handler only returns local process liveness. It does not validate the Codex OAuth credential, upstream reachability, model readiness, or streaming completion. CLIProxyAPI also generates an `X-CPA-TRACE-ID` after auth selection, can pass filtered upstream headers when `passthrough-headers` is enabled, and forwards an incoming `X-Client-Request-Id` to the Codex Responses request. OpenAI recommends logging its returned `x-request-id` and, for timeouts where no response header is available, sending a unique `X-Client-Request-Id`.

**Evidence:** `/healthz` is a static 200/JSON handler at [`server.go` lines 501-513](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/api/server.go#L501). `passthrough-headers` defaults false at [`config.example.yaml` lines 112-114](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/config.example.yaml#L112); CLIProxyAPI filters hop-by-hop and sensitive response headers but permits other upstream headers such as `x-request-id` at [`header_filter.go` lines 20-87](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/sdk/api/handlers/header_filter.go#L20). The Codex executor forwards `X-Client-Request-Id` at [`codex_executor.go` lines 2007-2029](https://github.com/router-for-me/CLIProxyAPI/blob/v7.2.91/internal/runtime/executor/codex_executor.go#L2007). OpenAI specifies both IDs and the uniqueness/ASCII/512-character constraints ([API request debugging](https://developers.openai.com/api/reference/overview#debugging-requests)). CLIProxyAPI documents error-request logs even when full request logging is disabled ([management API](https://help.router-for.me/management/api#request-error-logs)).

**Implication:** Use three probes: (1) `HEAD /healthz` for process liveness, (2) a low-frequency short inference for auth/upstream readiness, and (3) the real long-stream completion SLI. For each failed run retain timestamp/timezone, Claude Code and CLIProxyAPI versions, model, concurrency, client attempt number, first-byte/last-byte times, last SSE event type, `X-CPA-TRACE-ID`, OpenAI `x-request-id`, a unique client request ID, HTTP/2 or proxy errors, and process RSS/CPU. Use Claude Code `--debug "api"` with a dedicated debug file for a bounded reproduction window. Prefer CLIProxyAPI's error-only request logs; full `request-log` can capture large request/response bodies and should be enabled only briefly with restricted access and redaction before sharing.

### Finding 9 — Upgrade by canary, with exact-version rollback; current releases offer transport/performance improvements but no proven fix for this upstream error

CLIProxyAPI v7.2.92 is one release newer than 7.2.91 and contains Codex translator/executor allocation optimizations. Those may reduce local memory pressure on large translated histories, but its changelog contains no 502, timeout, or stream-recovery fix. Claude Code releases after 2.1.205 include relevant transport hardening: v2.1.208 fixed in-flight HTTP/2 GOAWAY crashes in supervised/background sessions, and v2.1.214 changed retries to open a fresh socket after stale-connection errors (and fixed a Windows corporate-proxy streaming issue). None is evidence that a server-generated OpenAI `response.failed` will disappear.

**Evidence:** CLIProxyAPI [v7.2.92 release notes](https://github.com/router-for-me/CLIProxyAPI/releases/tag/v7.2.92) list payload normalization/allocation changes only; [v7.2.91](https://github.com/router-for-me/CLIProxyAPI/releases/tag/v7.2.91) contains only the overlong encrypted reasoning-ID sanitization change. Claude Code's official [`CHANGELOG.md`](https://github.com/anthropics/claude-code/blob/main/CHANGELOG.md) records the v2.1.208 HTTP/2 GOAWAY fix and the v2.1.214 stale-socket retry change. Current Claude Code documentation also notes that gateways must stay updated as Claude Code adds capabilities ([gateway guidance](https://code.claude.com/docs/en/llm-gateway#what-a-gateway-provides)).

**Implication:** Canary an exact CLIProxyAPI v7.2.92 binary and a current pinned Claude Code release separately, one variable at a time. Verify published checksums, preserve the known-good 7.2.91 and 2.1.205 binaries, do not alter or copy credential material, drain active streams before restart, and roll back the binary immediately if failure rate, latency, or memory worsens. Do not claim upgrade success until the repeatability matrix improves; do not use “latest” as an unpinned production channel.

### Finding 10 — Ranked mitigation plan: durable controls first, transient retry last

The mitigations below are ranked by ability to reduce recurrence or diagnosis time without unsafe replay or authentication changes.

**Evidence:** Ranking synthesizes the source-backed boundaries established above: upstream failure mapping, no CLIProxyAPI deadline, Claude Code's replay safety rule, SSE keep-alive scope, rate-limit guidance, and current release deltas.

| Rank | Mitigation                                                               | Class                               | Operational decision                                                                                                             |
| ---- | ------------------------------------------------------------------------ | ----------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| 1    | Add per-request correlation, error-only logs, and layered SLIs           | Durable                             | Required before tuning; distinguishes upstream `response.failed`, network close, idle watchdog, rate limit, and process failure. |
| 2    | Decompose long work into checkpointed requests                           | Durable                             | Highest direct reduction in failure exposure and replay cost.                                                                    |
| 3    | Cap and smooth concurrency; coordinate retry budgets                     | Durable                             | Prevents load bursts and nested retry storms; begin at concurrency one.                                                          |
| 4    | Align verified reverse-proxy per-read timeouts and disable SSE buffering | Durable when an intermediary exists | Apply only to the actual streaming route and only after measuring inter-event gaps.                                              |
| 5    | Enable 15-second CLIProxyAPI downstream SSE comments                     | Conditional                         | Useful for proven quiet-link drops; ineffective against explicit upstream `response.failed`.                                     |
| 6    | Keep one bounded pre-first-byte retry owner                              | Conditional                         | Proxy bootstrap retry at most one; never replay after visible output.                                                            |
| 7    | Canary CLIProxyAPI/Claude Code upgrades with exact rollback              | Conditional durable improvement     | Transport/allocation fixes justify testing, but no release note proves this 502 is fixed.                                        |
| 8    | Manual rerun after a brief wait                                          | Transient                           | Appropriate for an isolated server error and confirmed by the successful 2m48s rerun; does not prevent recurrence.               |

**Implication:** Implement ranks 1–3 first. Apply ranks 4–7 only when the evidence for that layer is present. Retain rank 8 as operator recovery, not architecture.

### Finding 11 — A repeatability matrix can separate provider variance from local transport and version effects

A single failure followed by success is insufficient to estimate reliability. A bounded, one-variable-at-a-time campaign can determine whether failures correlate with duration, concurrency, idle gaps, versions, or an intermediary.

**Evidence:** OpenAI recommends request-ID logging for production troubleshooting ([API overview](https://developers.openai.com/api/reference/overview#debugging-requests)); Claude Code exposes per-invocation debug and timeout/retry controls ([CLI reference](https://code.claude.com/docs/en/cli-usage), [environment variables](https://code.claude.com/docs/en/env-vars)); CLIProxyAPI exposes exact-version configuration, trace IDs, health, and error logs through the sources cited above.

**Implication:** Run the following bounded campaign with no credential movement and no destructive cleanup:

1. Establish a direct-localhost v7.2.91 / Claude Code 2.1.205 baseline at concurrency one: ten short controls and ten long reproductions, with client retries capped to two or three for the campaign so one sample cannot expand into hours of hidden attempts.
2. Repeat the same inputs with only `streaming.keepalive-seconds: 15` changed. Retain it only if recorded idle disconnects disappear without masking stall detection.
3. Repeat with only CLIProxyAPI v7.2.92 changed, then only the pinned current Claude Code release changed. Compare completion rate, p50/p95 duration, first-token latency, maximum event gap, RSS, number of upstream attempts, and error taxonomy.
4. Repeat the baseline with the workload split into two or more checkpointed phases. Compare end-to-end completion and total retry cost, not only individual request latency.
5. After sequential stability, test concurrency two and four as separate cells. Stop the cell on repeated 429/5xx, a growing retry queue, or resource saturation rather than continuing to hammer the provider.
6. For every attempt, use a unique request correlation value where the invocation makes one model request, capture `X-CPA-TRACE-ID` and upstream `x-request-id`, and record whether the failure occurred before first payload, after visible text, or after a tool call. Identical prompts are not identical upstream executions because load, session identifiers, prompt-cache state, and model sampling vary.

### Finding 12 — Unsafe and unsupported workarounds must remain rejected

The operator-owned Codex OAuth boundary is part of the security and usage contract. Reliability work should use documented login/configuration surfaces and observability, never move or derive secrets or defeat protective timeouts and backoff without evidence.

**Evidence:** Claude Code gateway guidance states that the provider credential remains server-side and developers receive gateway credentials, and warns that third-party gateways are operator-maintained infrastructure ([gateway guidance](https://code.claude.com/docs/en/llm-gateway#what-a-gateway-provides)). OpenAI's API overview says credentials are secrets and must not be shared or exposed ([authentication guidance](https://developers.openai.com/api/reference/overview#authentication)). CLIProxyAPI's supported OAuth/auth directory and management surfaces are documented in its [basic configuration](https://help.router-for.me/configuration/basic) and [management API](https://help.router-for.me/management/api).

**Implication:** Explicitly reject all of the following:

- Copying, exporting, scraping, decrypting, or otherwise extracting Codex OAuth tokens or auth JSON for another process, host, account, or retry pool.
- Adding accounts or rotating credentials to evade provider concurrency, quota, or cooldown behavior.
- Deleting or resetting the CLIProxyAPI auth directory, Claude Code state, caches, keychain entries, or configuration as a response to a transient 502.
- Permanently mutating user shell profiles or global Claude Code settings during diagnosis; use documented per-invocation environment variables or an operator-owned canary configuration and revert it.
- Disabling TLS verification, weakening certificate validation, or bypassing an organizational proxy/security control.
- Setting unbounded retries, `CLAUDE_CODE_RETRY_WATCHDOG=1`, very large deadlines, global idle-timeout disablement, `disable-cooling: true`, or repeated immediate reruns as a generic reliability strategy.
- Restarting or killing CLIProxyAPI while active streams are running; drain first and use a binary rollback rather than a destructive state reset.

## Notes

- **Caveat:** The assignment did not include the raw failed SSE transcript, response headers, reverse-proxy topology, or exact failure timestamp. The upstream-terminal-failure conclusion is validated only when the captured body contains `response.failed`/`server_error`. If the client saw a plain HTTP 502 before any SSE event, investigate the producing hop from its `Server`/`Via` headers and logs.
