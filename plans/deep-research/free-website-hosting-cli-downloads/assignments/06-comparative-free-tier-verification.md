# Comparative Free-Tier Hosting Verification

## Assignment

**Goal:** Independently verify the important comparative claims for a small public open-source project website that distributes cross-platform CLI binaries, covering OpenAI Sites, GitHub Pages plus GitHub Releases on GitHub Free, and Firebase Hosting on Spark.

**Scope:** Internet-only verification against current primary official OpenAI, GitHub, Firebase, and Google documentation inspected on 2026-07-19. Material capabilities are classified as **Verified free**, **Paid**, **Unavailable**, **Unclear**, or **Unsupported**. The comparison covers direct downloads, bandwidth and transfer, storage, quotas, file and executable limits, default domains, custom domains, and modest-usage trade-offs.

**Exclusions:** No account-level deployments, no live artifact uploads, no third-party hosting products, no local source material as evidence, and no final project synthesis or implementation plan.

### Finding 2: GitHub Releases is the strongest verified binary origin

GitHub describes Releases as deployable software iterations for a wider audience and explicitly supports links to binary files. A public Release asset can be fetched without authentication through its `browser_download_url`, through a versioned `/releases/download/<tag>/<asset>` path, or through `/releases/latest/download/<asset>`. The release contract allows up to 1,000 assets per release, requires each file to be under 2 GiB, and states that neither total release size nor bandwidth usage is limited. The REST representation includes a SHA-256 digest field and download count.

**Evidence:** Binary packaging and the 1,000-asset, under-2-GiB, no-total-size, and no-bandwidth-limit rules are stated in [About releases](https://docs.github.com/en/repositories/releasing-projects-on-github/about-releases). Stable latest-asset links are documented in [Linking to releases](https://docs.github.com/en/repositories/releasing-projects-on-github/linking-to-releases). Public unauthenticated access, binary responses, `browser_download_url`, media type, digest, and download count are documented in [REST API endpoints for release assets](https://docs.github.com/en/rest/releases/assets?apiVersion=2026-03-10). GitHub separately recommends Releases instead of tracking large binaries in ordinary Git in [About large files on GitHub](https://docs.github.com/en/repositories/working-with-files/managing-large-files/about-large-files-on-github).

**Implication:** For this project, use Releases for versioned macOS, Linux, and Windows artifacts and link to them from the website. This keeps binaries outside the 1 GB Pages site limit and the ordinary-Git 100 MiB file block, while preserving direct public URLs and a documented digest.

### Finding 3: GitHub Pages is the strongest free website companion, with a soft bandwidth boundary

GitHub Free explicitly includes Pages for public repositories. Pages has a 1 GB published-site maximum, a recommended 1 GB source-repository limit, a soft 100 GB/month bandwidth limit, a soft 10-builds/hour limit for the standard build path, a 10-minute deployment timeout, and possible HTTP `429` rate limiting. GitHub may stop serving an over-quota site or suggest moving high-bandwidth content to Releases or another host. Pages is intended primarily to showcase projects rather than operate a commercial SaaS or transaction site.

**Evidence:** Free-plan inclusion appears in [GitHub's plans](https://docs.github.com/en/get-started/learning-about-github/githubs-plans) and [What is GitHub Pages?](https://docs.github.com/en/pages/getting-started-with-github-pages/what-is-github-pages). Numeric limits and over-quota behavior are in [GitHub Pages limits](https://docs.github.com/en/pages/getting-started-with-github-pages/github-pages-limits). Intended use is stated in the Pages section of the [GitHub Terms for Additional Products and Features](https://docs.github.com/en/site-policy/github-terms/github-terms-for-additional-products-and-features#pages).

**Implication:** A small open-source project landing page is aligned with the documented purpose. Keeping release artifacts on Releases makes the 100 GB/month soft Pages boundary largely a website-traffic concern rather than a binary-download ceiling.

### Finding 4: Firebase Spark is a credible free site host but a conditional download host

Firebase Hosting serves deployed files from a public root as public static assets, supplies `web.app` and `firebaseapp.com` endpoints over SSL, supports file-specific response headers, permits a 2 GB individual file, and provides 10 GB of no-cost Hosting storage. Retained earlier releases count toward that storage. However, Spark blocks upload, hosting, and serving of `.exe`, `.dll`, `.bat`, `.apk`, and `.ipa` on projects created on or after 2023-09-28. Blaze removes this extension restriction but is a pay-as-you-go plan.

**Evidence:** Public static assets and default endpoints are documented in [Get started with Firebase Hosting](https://firebase.google.com/docs/hosting/quickstart). File-specific response headers are documented in [Configure Hosting behavior](https://firebase.google.com/docs/hosting/full-config). The 2 GB file maximum, 10 GB retained-release storage allowance, and deployment blocking at the storage limit are documented in [Hosting usage, quotas, and pricing](https://firebase.google.com/docs/hosting/usage-quotas-pricing). The dated Spark blocklist and Blaze exemption are documented in [FAQ and troubleshooting](https://firebase.google.com/docs/hosting/faq-and-troubleshooting).

**Implication:** Firebase Spark is suitable as a no-cost front end, but direct Firebase delivery is incomplete for a cross-platform CLI that publishes a Windows executable. Archives and extensionless binaries are not named in the blocklist, yet the absence of a named restriction is not a positive compatibility guarantee; GitHub Releases remains the lower-risk artifact origin.

### Finding 5: Firebase's current transfer allowance is genuinely unresolved across official sources

The detailed Hosting usage guide states that CDN transfer is free up to 10 GB/month, counts cache hits and misses, and disables a non-Blaze site after a short grace period until the next month. The current Firebase pricing table instead states 360 MB/day for Hosting data transfer on Spark. These values are similar in monthly magnitude but differ in accounting period and enforcement semantics. No inspected official source reconciles them.

**Evidence:** The monthly figure and disablement behavior are in [Hosting usage, quotas, and pricing](https://firebase.google.com/docs/hosting/usage-quotas-pricing). The daily figure is in [Firebase pricing](https://firebase.google.com/pricing). The general Spark-plan rule says exceeding a no-cost product quota shuts that product off for the remainder of the calendar month in [Firebase pricing plans](https://firebase.google.com/docs/projects/billing/firebase-pricing-plans).

**Implication:** Classify Firebase Spark transfer as **Unclear**, not as a single verified quota. For conservative planning, keep daily transfer below 360 MB and monthly transfer below 10 GB. A 20 MB artifact reaches 360 MB after 18 downloads in a day and reaches 10 GB at roughly 500 downloads, before accounting for site assets.

### Finding 6: Free domain and TLS support favors GitHub Pages or Firebase

GitHub Pages supplies predictable `github.io` URLs on the Free public-repository path and supports owned apex or subdomains with HTTPS and HTTPS enforcement. Firebase supplies two no-cost project subdomains, supports owned apex domains and subdomains, and provisions and automatically renews SSL after DNS ownership and routing are configured. Neither platform registers the custom domain; the owner must acquire and maintain it. A Pages custom domain does not rebrand the final GitHub Release asset URL. OpenAI Sites may supply an OpenAI-owned subdomain, but only within eligible paid access; custom-domain availability is conditional, requires an already-owned domain, is unavailable in Enterprise workspaces at launch, and lacks a published TLS lifecycle.

**Evidence:** GitHub default URL shapes are in [What is GitHub Pages?](https://docs.github.com/en/pages/getting-started-with-github-pages/what-is-github-pages); DNS setup is in [Managing a custom domain for your GitHub Pages site](https://docs.github.com/en/pages/configuring-a-custom-domain-for-your-github-pages-site/managing-a-custom-domain-for-your-github-pages-site); HTTPS support is in [Securing your GitHub Pages site with HTTPS](https://docs.github.com/en/pages/getting-started-with-github-pages/securing-your-github-pages-site-with-https). Firebase default domains and SSL are in [Get started with Firebase Hosting](https://firebase.google.com/docs/hosting/quickstart), while certificate provisioning and renewal are in [Connect a custom domain](https://firebase.google.com/docs/hosting/custom-domain). OpenAI's domain conditions are in [Creating and managing ChatGPT Sites](https://help.openai.com/en/articles/20001339) and the conditional OpenAI-owned subdomain is in the [ChatGPT Sites Terms](https://openai.com/policies/chatgpt-sites-terms/).

**Implication:** Both GitHub Pages and Firebase offer a practical free hosting attachment for an already-owned branded domain. GitHub is operationally simpler when the source and releases already live in the same public repository; Firebase is a viable site-only alternative when its CDN and response configuration are preferred.

### Finding 7: OpenAI Sites is a paid beta front end, not a verified free or binary-distribution option

ChatGPT Sites is unavailable on Free and Go. On eligible paid plans, usage is included only to changing plan-specific beta limits that apply across all Sites and are shown in-product. Reaching a limit can stop new Site creation, storage additions, or public availability of a high-usage Site. The runtime supports R2 object storage for generic uploads, but the inspected official sources do not define public object URLs, anonymous CLI downloads, permitted executable extensions, per-file limits, transfer limits, MIME behavior, or response-header controls. Sites is also a Beta Service with no warranty of continued availability, uninterrupted service, or content preservation.

**Evidence:** Paid-plan availability, public publishing, in-product limits, domain conditions, and unsupported-use boundaries are in [Creating and managing ChatGPT Sites](https://help.openai.com/en/articles/20001339). Generic R2 uploads are described in the [Sites developer guide](https://learn.chatgpt.com/docs/sites). The OpenAI-owned subdomain, removal right, and beta designation are in the [ChatGPT Sites Terms](https://openai.com/policies/chatgpt-sites-terms/). Beta-service disclaimers are in the [Service Terms](https://openai.com/policies/service-terms/).

**Implication:** OpenAI Sites is **Paid** for hosting access and **Unsupported** for direct CLI distribution under the current public documentation. If an eligible subscriber values Sites' interactive authoring, it can serve as a presentation layer that links to GitHub Releases, but it should not be the sole release origin.

### Finding 8: Strongest recommendation and conditional alternatives

The strongest recommendation is **GitHub Pages on GitHub Free for the website plus GitHub Releases for binaries**. This combination supplies a verified free public website, optional free attachment of an already-owned custom domain with managed HTTPS, explicit public binary delivery, versioned and latest download URLs, automatic digest data, under-2-GiB per-asset capacity, and no documented Release bandwidth limit.

**Evidence:** The recommendation follows directly from GitHub Free Pages eligibility in [GitHub's plans](https://docs.github.com/en/get-started/learning-about-github/githubs-plans), domain and HTTPS support in [Managing a custom domain](https://docs.github.com/en/pages/configuring-a-custom-domain-for-your-github-pages-site/managing-a-custom-domain-for-your-github-pages-site) and [Securing a Pages site with HTTPS](https://docs.github.com/en/pages/getting-started-with-github-pages/securing-your-github-pages-site-with-https), and the release-download contract in [About releases](https://docs.github.com/en/repositories/releasing-projects-on-github/about-releases), [Linking to releases](https://docs.github.com/en/repositories/releasing-projects-on-github/linking-to-releases), and [REST API endpoints for release assets](https://docs.github.com/en/rest/releases/assets?apiVersion=2026-03-10).

**Implication:** Conditional alternatives are: (1) Firebase Hosting Spark for the website plus GitHub Releases for binaries when Firebase's CDN or header configuration is preferred; (2) Firebase-only hosting only for modest traffic and artifact formats that are accepted after a real deployment probe, while respecting both published transfer figures; and (3) OpenAI Sites only for an already-eligible paid account that values the Sites experience and still uses GitHub Releases as the release system of record.

## Notes

- Genuine unresolved blocker: Firebase's official pricing table says 360 MB/day of Hosting transfer, while its detailed Hosting usage guide says 10 GB/month. The documents do not establish whether both limits are enforced, one supersedes the other, or the daily figure is only a rounded presentation of the monthly allowance.
- The classification **Unsupported** for OpenAI Sites binary delivery means no affirmative public contract was found in the inspected official sources; it does not prove that a benign binary cannot work in a live Site.
- Firebase's blocklist expressly names `.exe`, `.dll`, `.bat`, `.apk`, and `.ipa`. Official sources do not affirmatively guarantee that archives, extensionless Unix executables, or every other CLI artifact format will be accepted on Spark.
- GitHub documents no bandwidth limit for Releases, but all use remains subject to GitHub's terms and acceptable-use enforcement; “no documented limit” should not be interpreted as immunity from abuse controls.
