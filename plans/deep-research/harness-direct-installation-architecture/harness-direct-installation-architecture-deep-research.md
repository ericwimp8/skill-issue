# Skill Issue Direct-Installation Architecture

## Executive Direction

Skill Issue should ship one immutable, standards-oriented Agent Skill directory and install it through one cross-platform CLI. The shared engine should validate, preview, stage, back up, atomically materialize, record ownership, activate, verify, diagnose, update, repair, roll back, and remove that directory. Each harness adapter should own only the documented destination resolution, feature gates, trust/policy checks, refresh action, collision interpretation, and verification command. This is the strongest supported design because GitHub Copilot, Claude Code, OpenAI Codex, Cursor, Antigravity, Gemini CLI, OpenCode, Kilo Code, and Pi all expose direct skill-folder discovery, while Grok Build exposes direct skill roots but leaves its loose-skill entry schema insufficiently documented. [Cross-harness assignment](assignments/10-cross-harness-adapter-contract.md) [GitHub Agent Skills](https://docs.github.com/en/copilot/concepts/agents/about-agent-skills) [Claude Code skills](https://code.claude.com/docs/en/slash-commands) [OpenAI Codex skills](https://developers.openai.com/codex/skills) [Cursor skills](https://cursor.com/docs/skills) [Antigravity skills](https://antigravity.google/docs/skills) [Gemini CLI skills](https://geminicli.com/docs/cli/skills/) [OpenCode skills](https://opencode.ai/docs/skills) [Kilo skills](https://kilo.ai/docs/customize/skills) [Pi skills](https://pi.dev/docs/latest/skills) [Grok Build skills](https://docs.x.ai/build/features/skills-plugins-marketplaces)

The canonical payload should be one directory whose basename equals a unique lowercase, hyphen-separated `name`, containing exact-case `SKILL.md` with YAML `name` and non-empty `description`, plus only relative supporting files such as `scripts/`, `references/`, `assets/`, `examples/`, and `templates/`. The shared validator should enforce the strict portable subset: name length at most 64 characters, directory/name equality, complete referenced-file closure, and no host plugin manifest, hooks, MCP/LSP configuration, permission widening, or host-specific frontmatter by default. This subset satisfies the documented contracts of the strong targets; Grok Build remains a live-verified exception because xAI does not document the required entry filename or frontmatter schema. [Cross-harness assignment](assignments/10-cross-harness-adapter-contract.md) [OpenCode frontmatter](https://opencode.ai/docs/skills#write-frontmatter) [Kilo skill format](https://kilo.ai/docs/customize/skills#skillmd-format) [Pi frontmatter](https://pi.dev/docs/latest/skills)

The CLI should prefer each host's native project or user root rather than trying to maintain one shared physical copy. Shared `.agents/skills` and compatible `.claude/skills` roots are valid only when the operator explicitly wants cross-host coupling; they change precedence, collision risk, and surface reach. A successful operation must distinguish `materialized`, `discovered`, and `activated`. Filesystem presence alone is never sufficient verification. [Cross-harness assignment](assignments/10-cross-harness-adapter-contract.md)

## Exact Scopes and Paths

### GitHub Copilot

- Project: `.github/skills/<name>/SKILL.md`, `.agents/skills/<name>/SKILL.md`, or `.claude/skills/<name>/SKILL.md`. Prefer `.github/skills` for GitHub-native cross-surface reach. Copilot CLI also walks parent `.github/skills` roots in monorepos. [Assignment 01](assignments/01-github-copilot-direct-installation.md) [Copilot skill locations](https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-command-reference#skill-locations)
- User: `~/.copilot/skills/<name>/SKILL.md` or `~/.agents/skills/<name>/SKILL.md`. Prefer `~/.copilot/skills` unless intentional cross-agent sharing is required. [GitHub add-skills guide](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-skills#creating-and-adding-a-skill)
- Advanced CLI-only extra roots: comma-separated `COPILOT_SKILLS_DIRS`; this is runtime configuration, not a portable repository scope. [Copilot CLI reference](https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-command-reference#skill-locations)
- System: none documented. Organization and enterprise skills are remotely managed, outside direct filesystem installation. [Assignment 01](assignments/01-github-copilot-direct-installation.md)

### Claude Code

- Project: `<repository>/.claude/skills/<name>/SKILL.md`; Claude also discovers parent roots to the repository boundary and nested roots when working in their subtrees. [Claude skill locations](https://code.claude.com/docs/en/slash-commands#where-skills-live)
- User: `~/.claude/skills/<name>/SKILL.md`; on Windows `~` is `%USERPROFILE%`, and `CLAUDE_CONFIG_DIR` replaces `~/.claude`. [Claude directory reference](https://code.claude.com/docs/en/claude-directory)
- Added directory: `<added-dir>/.claude/skills/<name>/SKILL.md` only when the directory is supplied with `--add-dir` or `/add-dir`; `permissions.additionalDirectories` grants access but does not discover skills. [Claude additional directories](https://code.claude.com/docs/en/slash-commands#skills-from-additional-directories)
- Managed/system: managed settings roots are macOS `/Library/Application Support/ClaudeCode/`, Linux/WSL `/etc/claude-code/`, and Windows `C:\Program Files\ClaudeCode\`. The exact managed skills subpath is not documented, so the CLI must not guess it. [Assignment 02](assignments/02-claude-code-direct-installation.md) [Claude settings files](https://code.claude.com/docs/en/settings#settings-files)

### OpenAI Codex

- Project: every `$CWD/.agents/skills/<name>/SKILL.md` ancestor through `$REPO_ROOT/.agents/skills/<name>/SKILL.md`. Prefer the repository root for whole-repository reach. [Codex skills](https://developers.openai.com/codex/skills)
- User: `$HOME/.agents/skills/<name>/SKILL.md`. `$CODEX_HOME/skills` is deprecated compatibility and must be a migration probe rather than a new-install target. [Assignment 03](assignments/03-openai-codex-direct-installation.md) [Codex loader](https://github.com/openai/codex/blob/main/codex-rs/core-skills/src/loader.rs)
- Administrator: `/etc/codex/skills/<name>/SKILL.md`, requiring host administration. System/bundled skills are product-owned and read-only, not an installer target. [Codex skills](https://developers.openai.com/codex/skills)

### Cursor

- Project: `.cursor/skills/<name>/SKILL.md` or `.agents/skills/<name>/SKILL.md`; prefer `.cursor/skills` for a Cursor-specific install. Compatible Claude/Codex roots are documented but lower fit. [Cursor skills](https://cursor.com/docs/skills)
- User: `~/.cursor/skills/<name>/SKILL.md` or `~/.agents/skills/<name>/SKILL.md`; prefer the native root. [Cursor skills](https://cursor.com/docs/skills)

### Google Surfaces

- Antigravity 2.0 project: `.agents/skills/<name>/SKILL.md`; legacy-compatible `.agent/skills/<name>/SKILL.md` exists but is lower fit. User/global: `~/.gemini/config/skills/<name>/SKILL.md`. [Antigravity skills](https://antigravity.google/docs/skills)

## Payload and Host Metadata

The common payload must remain self-contained and be copied as a complete directory. `SKILL.md` is the entrypoint for every strong target. Supporting files remain relative to the skill directory and must not be flattened; hosts progressively expose or load them when the skill activates. GitHub Copilot, Claude, Codex, Cursor, Antigravity, Gemini, OpenCode, Kilo, and Pi all document support for scripts and/or references/assets/resources. [Assignments 01-05](assignments/01-github-copilot-direct-installation.md) [Assignments 07-09](assignments/07-opencode-direct-installation.md)

Host-specific metadata should be opt-in overlays only:

- Codex may use `agents/openai.yaml` for UI metadata, dependencies, default prompt, and `allow_implicit_invocation`; it is unnecessary for baseline discovery. [Codex skills](https://developers.openai.com/codex/skills)
- Claude supports extensions including invocation controls, tool lists, model, effort, context, agent, hooks, paths, and shell. These should not enter the portable core. [Claude frontmatter](https://code.claude.com/docs/en/slash-commands#frontmatter-reference)
- Cursor supports `paths`, `disable-model-invocation`, and `metadata`; Copilot CLI and Pi have their own invocation/tool fields. These change behavior and require intentional per-host review. [Cursor skills](https://cursor.com/docs/skills) [Copilot CLI skill fields](https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-command-reference#skill-frontmatter-fields) [Pi skills](https://pi.dev/docs/latest/skills)
- Grok Build's loose-skill manifest, filename, and frontmatter remain unsupported claims. The adapter must materialize the canonical candidate only after preview and must accept it only when `grok inspect` resolves it. [Assignment 06](assignments/06-grok-direct-installation.md)

No normal direct install needs `plugin.json`, `.claude-plugin/plugin.json`, `gemini-extension.json`, an npm package, a VSIX, hooks, MCP/LSP configuration, rules files, or host permission settings. Those belong to separate component and distribution lifecycles. [GitHub plugins](https://docs.github.com/en/copilot/concepts/agents/about-plugins) [Claude plugins](https://code.claude.com/docs/en/plugins) [Antigravity plugins](https://www.antigravity.google/docs/plugins) [OpenCode plugins](https://opencode.ai/docs/plugins) [Kilo plugins](https://kilo.ai/docs/automate/extending/plugins) [Pi packages](https://pi.dev/docs/latest/packages)

## Discovery, Conflicts, and Configuration

Every adapter must enumerate all documented roots that can supply the canonical `name` before writing. Same-name handling is not portable: Copilot is first-found by documented precedence; Claude applies enterprise/personal/project and nested qualification rules; Codex may expose both and make the unqualified selector ambiguous; Gemini and Kilo define higher-priority scope winners; Pi keeps the first discovered; OpenCode treats global uniqueness as required; Cursor, Antigravity, and Grok do not document a safe loose-skill precedence. The safe cross-host rule is therefore: reject a same-name non-owned candidate, never merge directories, and replace only a ledger-owned target or an explicitly authorized foreign target. [Cross-harness assignment](assignments/10-cross-harness-adapter-contract.md)

Native-root installation should not edit configuration. Advanced routes may require structural patches: Claude `skillOverrides`, Codex `[[skills.config]]`, Grok `[skills] paths`, OpenCode `permission.skill`, Kilo `skills.paths`, and Pi `skills` arrays. Such a patch must parse the host format, change one exact selector/path, preserve unrelated content and comments where fidelity is available, write atomically, record the reversible fragment, and refuse when managed policy owns the value. Pi project arrays can override global arrays rather than append during effective merging; whole-file or naive-array replacement is unsafe. [Claude settings](https://code.claude.com/docs/en/settings#settings-precedence) [Codex skill configuration](https://developers.openai.com/codex/skills) [Grok settings](https://docs.x.ai/build/settings#scopes) [OpenCode permissions](https://opencode.ai/docs/skills#configure-permissions) [Kilo additional paths](https://kilo.ai/docs/customize/skills#additional-skill-paths-and-remote-urls) [Pi settings](https://pi.dev/docs/latest/settings)

## Lifecycle and Owned State

The CLI should keep an ownership ledger outside installed skill content. Each record should contain adapter and host versions, scope/root/final path, canonical name, source URI/ref/version, payload and manifest digests, executable-file inventory, installation time, backup path/digest, exact configuration fragment owned, and last materialization/discovery/activation results. It must store no credentials and must not use mutable skill content as the only ownership proof. This is necessary because most harnesses expose discovery but no package-managed lifecycle for loose folders. [Cross-harness assignment](assignments/10-cross-harness-adapter-contract.md)

- **Install:** lock the target; preview the full manifest, digest, scripts, conflicts, policy/trust warnings, and rollback plan; stage beside the final target; revalidate; rename atomically where supported; record placement; activate; verify. Back up only an existing ledger-owned target. [Cross-harness assignment](assignments/10-cross-harness-adapter-contract.md)
- **Update:** require matching ownership or explicit replacement authorization; stage the complete new directory so stale files cannot survive; preserve the last verified owned version; replace; activate; verify; then advance the ledger. Host package update commands apply only to host-managed installs such as `gh skill`, Gemini tracked installs, Pi packages, or plugins, not to a plain Skill Issue copy. [Assignment 01](assignments/01-github-copilot-direct-installation.md) [Assignment 05](assignments/05-google-antigravity-gemini-cli-direct-installation.md) [Assignment 09](assignments/09-pi-direct-installation.md)
- **Repair:** compare path, manifest, digest, `SKILL.md`, frontmatter, collision state, configuration patch, trust/policy, and live discovery against the receipt. Restore the last verified owned backup or reinstall the declared source only after preview. Stop when ownership is absent or foreign. [Cross-harness assignment](assignments/10-cross-harness-adapter-contract.md)
- **Partial-install recovery:** an incomplete staging directory is installer-owned and may be discarded/retried; an absent or incomplete final directory may be restored only when the ledger and backup prove ownership. Never repair by copying only `SKILL.md`, because referenced resources are part of the supported unit. [Assignment 05](assignments/05-google-antigravity-gemini-cli-direct-installation.md) [Assignment 08](assignments/08-kilo-code-direct-installation.md)
- **Rollback:** restore the last verified owned directory and only the exact recorded configuration fragment, then run the host activation and verification sequence. Return unsupported if no verified backup exists. No assignment establishes native automatic rollback for arbitrary copied skills. [Cross-harness assignment](assignments/10-cross-harness-adapter-contract.md)
- **Diagnostics:** separately report executable/version detection, scope resolution, filesystem materialization, manifest/frontmatter validity, collision, disabled visibility, policy/trust denial, stale session, config patch drift, host discovery, and activation. Host-supported tools include Copilot `/skills info`, Claude `/skills` and `doctor`, Codex `skills/list`, Gemini `/skills list`, Grok `grok inspect --json`, Kilo `/reload` plus skill-tool observation, Pi startup/header and `/skill:<name>`, and caveated current-build OpenCode `debug skill`. [Assignments 01-09](assignments/01-github-copilot-direct-installation.md)
- **Uninstall:** require ledger ownership and target identity; remove only the recorded skill directory and exact recorded config fragment; retain the root, unrelated skills, other scopes, settings, and backups needed by policy; activate/restart; verify absence; then archive or remove the receipt. `gh skill` currently has no uninstall command for its managed install, while Gemini's uninstall applies only when Gemini tracks that install. [GitHub CLI skill manual](https://cli.github.com/manual/gh_skill) [Gemini skill management](https://geminicli.com/docs/cli/using-agent-skills/)

## Shared Adapter Contract

The later CLI should expose the following behavior contract without prescribing a particular implementation language or module design:

| Capability | Required result |
| --- | --- |
| `detect` | Identify the exact host surface, executable/app, version, OS/home/config overrides, documented feature gates, and policy/trust blockers. Return only supported capabilities. |
| `enumerateScopes` | Return each documented project/user/admin scope, exact resolved root, surface reach, collision model, config requirements, and refresh/verification actions; explicitly return absent or unvalidated scopes. |
| `preview` | Validate the portable payload; show source/digests, complete file and executable manifest, final path, conflicts, planned config patch, security warnings, and rollback availability without writing. |
| `install` | Run the owned, staged directory transaction and produce distinct materialized/discovered/activated outcomes. |
| `activate` | Perform only the host's documented reload, new-session, restart, or structured refresh action; otherwise return `restart-required` or `verification-pending`. |
| `verify` | Confirm expected name, resolved path, visible/enabled state, and harmless explicit use. Prefer structured output when documented. |
| `diagnose` | Re-run detection, scope, ledger, manifest, collision, config, trust/policy, discovery, and activation checks without inventing unsupported fixes. |
| `update` | Replace a ledger-owned complete payload with preview, backup, activation, verification, and ledger advancement. |
| `repair` | Restore the last verified owned state or re-materialize the declared source; stop on foreign/ambiguous ownership. |
| `rollback` | Restore one recorded verified backup and exact config fragment, then activate and verify. |
| `uninstall` | Remove only receipt-owned state, verify absence, and preserve all unrelated host/user state. |

Feature gates must be capability-probed as well as version-checked. Known documented floors are OpenCode `v1.0.186+`, Pi `v0.50.0+` for the ResourceLoader/`--skill`/`/reload` lifecycle, and Kilo CLI `1.0+`; GitHub `gh skill` requires GitHub CLI `2.90.0+` and remains preview. Gemini should expose its current skills commands; Grok must expose `grok inspect`. Claude and Cursor have narrower feature-specific version gates, so baseline discovery should be proven live rather than inferred from a single minimum version. ## Permissions, Trust, and Policy

Skills are instruction supply-chain content and may include executable helpers. The CLI must preview every file, identify executables, retain provenance/digests, avoid executing helpers during installation, and never broaden host permissions as a side effect. Installation does not grant shell, filesystem, network, connector, MCP, secret, or cloud permissions. [GitHub skill warning](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-skills#enabling-a-skill-to-run-a-script) [Codex approvals and security](https://learn.chatgpt.com/docs/agent-approvals-security) [Pi security](https://pi.dev/docs/latest/security)

- Copilot directory trust, organization CLI policy, and tool approvals remain authoritative; `allowed-tools` can pre-approve shell behavior and should be absent from unreviewed portable content. [Copilot configuration](https://docs.github.com/en/copilot/how-tos/copilot-cli/set-up-copilot-cli/configure-copilot-cli)
- Claude workspace trust, permission denies, sandbox, `strictPluginOnlyCustomization`, and `disableSkillShellExecution` can block or constrain skills. A policy lock is a blocker, not a reason to sideload elsewhere. [Claude permissions](https://code.claude.com/docs/en/permissions#managed-settings)
- Codex local sandbox/approval/network policy remains in force; writing user/admin roots may require an operator-managed copy. Optional invocation metadata is not a trust grant. [Codex security](https://learn.chatgpt.com/docs/agent-approvals-security)
- Cursor permission files, sandbox/network controls, and enterprise policy remain unchanged. Background Agents have materially broader remote/network risk and must not inherit assumptions from desktop trust. [Cursor permissions](https://docs.cursor.com/cli/reference/permissions) [Cursor Background Agents](https://docs.cursor.com/background-agent)
- Gemini workspace skills require trusted folders and activation consent; Antigravity has separate project permissions, while `agy` applies `deny > ask > allow`. [Gemini trusted folders](https://geminicli.com/docs/cli/trusted-folders/) [Antigravity permissions](https://www.antigravity.google/docs/cli-permissions)
- Grok sandbox is fixed at process start, permission denies override allows, and managed requirements cannot be overridden. Ordinary skill installation must not enable `always-approve`, hooks, MCP, or LSP. [Grok enterprise security](https://docs.x.ai/build/enterprise#security-controls)
- OpenCode managed configuration and `permission.skill` may hide or deny a skill. Kilo's last matching permission wins and sensitive files retain prompts. Pi project trust must be user-approved and Pi has no built-in sandbox. [OpenCode skill permissions](https://opencode.ai/docs/skills#configure-permissions) [Kilo permissions](https://kilo.ai/docs/customize/agent-permissions) [Pi security](https://pi.dev/docs/latest/security)

No supplied evidence establishes a shared signing or checksum standard for loose skills. The CLI should therefore use its own source provenance, release digest, manifest digest, and receipt for integrity, while accurately describing these as Skill Issue guarantees rather than host-native signing. Cursor explicitly lacks a documented direct-skill checksum/signature contract, and the other assignments do not establish a portable one. [Assignment 04](assignments/04-cursor-direct-installation.md) [Cross-harness assignment](assignments/10-cross-harness-adapter-contract.md)

## Candidate Evaluation

### Discovered Candidates

The ecosystem research encountered direct native skill roots; compatible shared roots; host-managed skill commands (`gh skill`, Gemini `skills install/link/uninstall`); optional extra-path settings; native plugins/marketplaces/extensions/packages; rules or instruction files; remote skill catalogs; IDE/plugin caches; and remote/cloud repository delivery. [Assignments 01-10](assignments/10-cross-harness-adapter-contract.md)

### Selected Deep Dives

The selected route is direct materialization into each host's documented native project/user skill root, with a common payload and receipt-backed lifecycle. It is the only route that matches the fixed product decision and has broad first-party discovery evidence without multiplying package ecosystems. Grok Build receives the same payload only through a bespoke, fail-closed live-verification adapter. [Cross-harness assignment](assignments/10-cross-harness-adapter-contract.md)

### Skim-Only Conditional Alternatives

- Shared `.agents/skills` or compatible `.claude/skills` roots can reduce physical duplication when intentional cross-host sharing outweighs collision and precedence coupling. They are explicit alternate targets, not the default. [Cross-harness assignment](assignments/10-cross-harness-adapter-contract.md)
- `gh skill` may provide provenance, pinning, dry-run updates, and force repair when its preview/version status is acceptable. Gemini tracked installs and links provide host-owned lifecycle for Gemini alone. Kilo Marketplace and Pi packages provide distribution semantics but introduce independent lifecycle and trust. [GitHub CLI update](https://cli.github.com/manual/gh_skill_update) [Gemini management](https://geminicli.com/docs/cli/using-agent-skills/) [Kilo Marketplace](https://kilo.ai/docs/customize/marketplace) [Pi packages](https://pi.dev/docs/latest/packages)
- Extra-path configuration is justified only for an externally managed source tree and requires host-aware merging and receipts. One-run Pi `--skill` is suitable for a trial. Remote catalogs remain lower fit because they add network, manifest, cache, and provenance behavior. [Pi skills](https://pi.dev/docs/latest/skills) [Kilo remote skills](https://kilo.ai/docs/customize/skills#additional-skill-paths-and-remote-urls)

### Rejected or Lower-Fit Interpretations

- A native plugin per harness is rejected. Plugins, extensions, packages, hooks, MCP/LSP, and VSIX artifacts bundle broader components or managed distribution; they are not required to discover an ordinary direct skill. [Assignments 01-09](assignments/01-github-copilot-direct-installation.md)
- Rules files are rejected as a substitute for skills. They are persistent project context with different activation semantics; Cursor `.cursor/rules`, Grok `AGENTS.md`/`.grok/rules`, and similar mechanisms should be used only for always-on guidance. [Assignment 04](assignments/04-cursor-direct-installation.md) [Grok project rules](https://docs.x.ai/build/features/project-rules)
- Undocumented filesystem roots, caches, managed skills subpaths, and inferred cloud home directories are rejected. This includes guessed Claude managed-skill paths, deprecated Codex roots for new installs, automatic Kilo `.kilo`/`.kilocode` duplication, and transferring Gemini commands to `agy`. [Assignments 02, 03, 05, 08](assignments/02-claude-code-direct-installation.md)
- Grok as only a model inside Cursor is rejected as an ecosystem interpretation. Grok Build is a standalone xAI coding harness; Grok selected in Cursor remains governed by Cursor. [Assignment 06](assignments/06-grok-direct-installation.md)
