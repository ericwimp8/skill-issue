# Completed Foundation: Product, Support, and Evidence Contract

## A: Starting Position

- The parent plan identifies nine selected harness targets, but the earlier split between a three-harness complete-workflow tier and a five-harness skill-calling tier has now been superseded.
- This foundation owns the boundary between package or installation support, complete workflow proof, skill-calling evaluation coverage, and public benchmark claims. It defines the evaluation commitment but does not execute the evaluations.
- Existing project evidence identifies `gpt-5.6-sol` with high reasoning as the model used for the completed Codex evaluation campaign.
- The minimum release now requires both complete end-to-end proof and skill-calling evaluation in OpenAI Codex, Claude Code, Cursor, Pi, and OpenCode across thirteen harness-and-model combinations.
- Fable is intentionally the top-level Anthropic model tier above Claude, alongside Anthropic's Haiku, Sonnet, and Claude tiers. The exact provider identifiers or aliases available through each harness still require normalization when the evaluations are configured.

## B: Desired Position

The project has one definitive contract under which OpenAI Codex, Claude Code, Cursor, Pi, and OpenCode form a unified minimum qualification tier. Every harness in that tier receives complete end-to-end release proof and every applicable model combination receives a skill-calling evaluation at a medium reasoning setting in a default harness environment containing only the supplied Skill Issue and evaluation components. Results are published as transparent descriptive data rather than pass or fail judgments.

## Path from A to B

### 1. Establish the unified minimum qualification tier

- Require complete CLI installation, complete Skill Issue workflow proof, cleanup or removal proof, and skill-calling evaluation for OpenAI Codex, Claude Code, Cursor, Pi, and OpenCode.
- Define complete end-to-end proof in each minimum harness as CLI installation, installed-package discovery, natural-language skill creation, generated-skill evaluation and refinement, existing-skill refinement, and cleanup or removal.
- Treat every harness in this five-harness tier as a minimum release requirement rather than giving Cursor or Pi a lower evidence tier.
- Keep the other four selected harnesses and additional model or reasoning combinations as later expansion beyond the minimum viable release.

### 2. Lock the minimum evaluation matrix

- Record the dictated matrix as:

| Harness      | Model           | Reasoning target |
| ------------ | --------------- | ---------------- |
| OpenAI Codex | Codex Sol       | Medium           |
| Claude Code  | Claude Opus 4.8 | Medium           |
| Claude Code  | Claude Fable    | Medium           |
| Cursor       | Codex Sol       | Medium           |
| Cursor       | Claude Opus 4.8 | Medium           |
| Cursor       | Claude Fable    | Medium           |
| Cursor       | Grok            | Medium           |
| Pi           | Codex Sol       | Medium           |
| Pi           | Claude Opus 4.8 | Medium           |
| Pi           | Claude Fable    | Medium           |
| OpenCode     | Codex Sol       | Medium           |
| OpenCode     | Claude Opus 4.8 | Medium           |
| OpenCode     | Claude Fable    | Medium           |

- Resolve each label to the exact provider model identifier, version or alias, and harness-specific configuration available when the evaluation is run.
- Use the harness's medium reasoning level or the closest documented equivalent across all thirteen cells.
- Document that medium is intentionally chosen to represent the value-oriented settings likely to be used by the general population rather than measuring only top-end reasoning performance.
- Require one tooling-complete three-scenario suite for each of the thirteen minimum matrix cells during the minimum viable campaign; each scenario contains the same fixed 30-turn structure, and repeated trials or statistical thresholds remain later expansions.
- Reserve stronger reasoning settings and additional models for later evaluation expansion.

### 3. Establish the minimum evaluation environment

- Use each harness's default configuration with only the Skill Issue skills, required supporting skills, and evaluation instrumentation added.
- Ask external evaluators to prepare and attest to that default environment; accept this human-controlled setup for the minimum viable campaign rather than requiring automated isolation first.
- Treat tooling failures as work to fix and rerun rather than as model evaluation outcomes.
- Confirm supportability before a combination enters the minimum campaign instead of publishing unsupported or unrun combinations as results.

### 4. Define valid results and public claims

- Count a completed, correctly instrumented scenario suite as a valid evaluation even when the model misses every expected skill call; absence of calls is evaluation data rather than an invalid run.
- Treat “tooling-complete” as successful completion of all three governed scenarios and evidence capture, independent of how many expected skills the model called.
- Publish observed and missing skill-call data in graphs without assigning pass or fail labels to a harness or model.
- Present the evaluation method, environment instructions, instrumentation, raw structured results, limitations, and graph derivation so readers can inspect the evidence for themselves.
- State only what was done, what was observed, and how the campaign was run; avoid claims of perfection, guarantee, or universal model behavior.
- Keep installation success, proactive skill selection, loaded-skill body behavior, and complete skill-generation workflow results as distinguishable evidence inside the unified qualification tier.

### 5. Reconcile the downstream contract

- Carry the confirmed minimum matrix into evaluation assets, evaluator recruitment, qualification, campaign execution, release limitations, website benchmark data, and submission copy.
- Keep the thirteen model evaluations distinguishable within the unified five-harness release requirement.
- Ensure future additions or substitutions update the contract once rather than creating conflicting matrices across plans and public artifacts.

## C: Observable Completion Criteria

- One authoritative contract names the selected nine harnesses and the unified five-harness minimum qualification tier.
- All five minimum harnesses have complete end-to-end proof and all thirteen minimum matrix cells have completed skill-calling evaluations before the minimum release is presented as complete.
- Each minimum harness has evidence for CLI installation, package discovery, natural-language skill creation, generated-skill evaluation and refinement, existing-skill refinement, and cleanup or removal.
- Each minimum matrix cell has one tooling-complete three-scenario suite; the initial release does not require repeated trials or a pass threshold.
- Every matrix cell records its exact provider model identifier or alias, medium reasoning configuration or equivalent, harness version, operating system, default-environment attestation, and raw structured result.
- Published graphs show observed skill-calling data without pass or fail labels and link their meaning to a transparent method and inspectable result set.
- Tooling failures are repaired and rerun, while unsupported or unrun combinations remain outside the published minimum result set rather than appearing as model outcomes.
