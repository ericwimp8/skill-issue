# Work Block 1 Supporting Plan: Create the CLI-Owned Supporting-Skills Bundle

## A: Starting Position

- Skill Issue relies on four discipline skills currently owned by the Eric Wimp Toolkit source project:
  - `document-update-discipline`
  - `prompt-writing`
  - `skill-authoring-discipline`
  - `system-change-ownership`
- Their authoritative source directories are under `plugins/eric-wimp-toolkit/skills/` in the Eric Wimp Toolkit source repository.
- Skill Issue does not require Dictate Plan, Expand Dictate Plan, Audit Orchestrator, orchestration wrappers, or custom-agent installers as runtime dependencies.
- The Skill Issue CLI installs skills directly into each supported harness's native skill directories. This collection is therefore a CLI-owned source bundle, not a separately installed harness plugin.
- Evaluation-only instructions remain outside the canonical skill sources. The CLI generates instrumented evaluation copies when requested.

## B: Desired Position

The public Skill Issue repository contains a root `supporting-skills/` directory holding complete copies of the four required discipline skills. Together with the three primary workflows under `skills/`, these files form the canonical skill source consumed by the CLI for normal and generated evaluation installations.

## Path from A to B

### 1. Copy the accepted supporting-skill inventory

- Create `supporting-skills/`.
- Copy the complete source directory for each accepted skill, including every referenced file, script, template, and asset.
- Preserve the copied skills' existing semantics.
- Do not copy unrelated toolkit skills.

### 2. Validate the copied bundle

- Confirm that every `SKILL.md` and every referenced local file is present.
- Confirm that the four copied skills do not rely on omitted toolkit skills, custom agents, orchestration wrappers, or unavailable project files.
- Validate the copied skills with the repository's applicable skill checks.
- Reconcile any genuine missing dependency at its owning skill rather than expanding the bundle speculatively.

### 3. Define the canonical CLI handoff

- Treat `skills/` and `supporting-skills/` as the canonical source collections for the Skill Issue payload.
- Give Work Block 2 a deterministic inventory of every directory and file that must enter the embedded payload.
- Keep normal sources free of evaluation-only instructions.
- Leave evaluator-copy generation, instrumentation injection, installation, refresh, verification, and removal to the CLI.

### 4. Run the supporting-bundle audit

- Verify that the accepted four skills are present and complete.
- Verify that no unapproved toolkit skill entered the bundle.
- Verify that the copied sources remain usable independently of the original toolkit installation.
- Verify that downstream plans point to `supporting-skills/` rather than a separate plugin artifact or source repository.

## C: Observable Completion Criteria

- `supporting-skills/` contains complete copies of the four accepted discipline skills.
- The copied skills have no unresolved dependency on omitted toolkit skills, custom agents, or orchestration wrappers.
- Dictate Plan, Expand Dictate Plan, Audit Orchestrator, and unrelated toolkit skills are absent.
- `skills/` and `supporting-skills/` together provide the complete canonical skill-source inventory required by Skill Issue.
- Work Block 2 can embed the canonical inventory without recreating, discovering, or importing skill sources from another repository.
- Evaluation installations are defined as CLI-generated derivatives of the current canonical sources rather than separately maintained files.
