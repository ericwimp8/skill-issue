# Skill Issue Project Completion A-to-B Plan

## A: Starting Position

- Skill Intake, Skill Generation, and Skill Evaluation and Refinement exist as structurally valid production skills with explicit handoffs and supporting references.
- A completed twenty-one-iteration Codex campaign refined and passed those production skills inside its recorded Codex Desktop and `gpt-5.6-sol` high-reasoning environment. The campaign proves that bounded environment only and stops the generated-skill end-to-end case at an explicit Evaluation handoff without claiming the deferred runtime result.
- The four required discipline skills are copied into the CLI-owned `supporting-skills/` bundle. Cross-harness sub-agent launch research remains historical input for skill behavior, while skill-calling evaluation now uses direct primary-agent replay. Same-run primary handoffs, the blind instrumentation contract, all three governed 30-turn scenarios, and the reusable Go runner are complete. Real-harness qualification remains unfinished.
- The selected support set is GitHub Copilot, Claude Code, OpenAI Codex, Cursor, Google Antigravity or Gemini CLI, Grok Build, OpenCode, Kilo Code, and Pi. OpenAI Codex, Claude Code, Cursor, Pi, and OpenCode form the minimum end-to-end and skill-calling qualification tier across the thirteen medium-setting cells defined by the completed support contract.
- Authoritative direct-install research defines a strict portable Agent Skill payload, native project or user roots, common and bespoke adapters, receipt-backed lifecycle behavior, three-stage materialization/discovery/activation evidence, and explicit caveats for unvalidated surfaces.
- The pure-Go CLI now embeds the canonical primary and supporting skill payload plus complete governed evaluation units. Ordinary lifecycle operations retain researched native project and user roots; evaluation runs always use the selected workspace's native project root. The CLI records ownership receipts, generates blind temporary copies, drives primary-agent sessions, attributes opaque signals by private turn state, exports evidence, and restores prior project installations. Its adapters and binaries still require real-harness and native-platform qualification.
- A complete React, TypeScript, Vite, and Recharts website mock-up exists with deterministic validation and a GitHub Pages build. It still uses mock release and benchmark content and has not been publicly deployed as the finished product.
- Research already covers harness popularity, historical plugin packaging, direct installation, sub-agent launch behavior, cross-platform CLI distribution, free website and binary hosting, video tooling, and Speechify-to-Remotion narration. Each report remains available to the work block that consumes it, subject to its recorded caveats and current verification gates.
- Public binaries, native platform qualification, the thirteen-cell campaign, final product copy, real website results, release automation, the video, the YouTube URL, and the hackathon submission remain unfinished.

## B: Desired Position

Skill Issue is a complete public product that turns an ordinary-language or dictated request into an idiomatic, evaluated, semantically refined agent skill and separately evaluates and improves an existing skill. A self-contained Go CLI detects and confirms the user's harness and supported project or user scope, installs the complete canonical payload, generates disposable evaluation installations from current canonical skills, verifies materialization, discovery, and activation, and safely manages update, repair, rollback, and uninstall without disturbing unrelated state. The minimum five harnesses have complete workflow evidence and all thirteen medium-setting evaluation cells have transparent, graph-ready skill-calling results. Qualified binaries, documentation, limitations, a deployed website, final video, and verified hackathon submission are publicly available.

## C: Observable Completion Criteria

### Skill System

- `supporting-skills/` contains the four accepted discipline skills, excludes unapproved toolkit behavior, and provides canonical sources for normal and generated evaluation installations.

### Installer Product

- A user can run a self-contained macOS, Windows, or Linux CLI without installing a programming-language runtime.
- The CLI detects and presents the likely harness and exact supported project or user scope, accepts confirmation or correction, previews the change, and records owned state.
- Every installed payload contains the complete primary workflow and supporting-skills bundle without requiring the full Eric Wimp Toolkit.
- Normal and evaluation installations remain separate; evaluator copies are regenerated from current canonical skills and can be removed without changing ordinary sources.
- Update, diagnosis, repair, rollback, and uninstall preserve unrelated skills, configuration, trust, policy, and user state.

- The campaign distinguishes installation support, skill selection, loaded-skill behavior, and complete workflow results; missing calls remain data and tooling failures are repaired and rerun.
