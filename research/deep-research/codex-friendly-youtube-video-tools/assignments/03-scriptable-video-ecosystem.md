# Scriptable Video Production Ecosystem

## Assignment

**Goal:** Discover and rank current code-defined, command-line, and headless-capable tools for a polished, generic, public 1080p YouTube demonstration under three minutes, combining screen footage, titles/intro/outro, transitions, audio, captions where useful, and final delivery on macOS.

**Scope:** Local command inventory plus official documentation, repositories, licenses, downloads, and pricing/licensing material inspected 2026-07-19. The ranking emphasizes Codex/LLM-authorable artifacts, fast iteration, minimal setup, macOS availability, local-versus-remote handling, export restrictions, and the capabilities needed for the stated video.

**Exclusions:** Browser-template products and ordinary desktop-screen-recorder comparisons owned by adjacent assignments; a full implementation trial; subjective visual-template comparison; unverified claims of a free tier, watermark policy, or model quality.

## Findings

### Candidate Ranking

The candidate set is ranked for the specific short product-demo job, rather than for general film, live-streaming, or mathematical-animation production. “Deep-dive” means the candidate merits a targeted next-wave verification; it is not a final recommendation. “Local” is an inference from the documented local CLI/application architecture and should be read as “can be run without uploading media,” not as a claim that every optional package, asset, telemetry setting, or distribution channel is offline.

| Rank | Candidate | Classification | Fit and verified evidence |
| --- | --- | --- | --- |
| 1 | [FFmpeg](https://ffmpeg.org/ffmpeg.html) | **Deep-dive** | The strongest free local stitch/render backbone: complex filtergraphs, codecs, audio/subtitle streams, and `concat` support; its macOS AVFoundation input lists capture devices. It is already on PATH locally. It is fast and deterministic for Codex-authored scripts, titles (`drawtext`), fades/transitions (`xfade`), mixing (`amix`/`afade`), subtitles, and H.264 1080p. Main cost is command/filtergraph learning and visual polish; base licensing is LGPL 2.1+ but optional GPL components alter the build’s license. No watermark or service privacy exposure is inherent to local operation. |
| 2 | [Remotion](https://www.remotion.dev/docs/cli/render) | **Deep-dive** | TypeScript/React compositions provide the most LLM-friendly path to polished branded intro/outro, titles, transitions, captions, and audio around supplied screen footage. The CLI renders an entry/composition to a file, accepts JSON props, supports width/height/FPS/duration, H.264/H.265/ProRes, audio controls, and headless Chrome. It is significantly more code/setup than FFmpeg but enables higher-motion-design quality. Licensing must be checked against entrant status: its docs say users ineligible for the Free License need a Company License; optional licensing telemetry does not transmit rendered-media content, while `@remotion/web-renderer` always emits telemetry and records IP address. |
| 3 | [OBS Studio](https://obsproject.com/download) | **Deep-dive** | Best purpose-built capture complement. Official macOS downloads support macOS 12+ on Intel and Apple Silicon, and built-in WebSocket since OBS 28 enables external automation of scenes and sources. It can produce clean 1080p screen footage with mic/system-audio routing and scene overlays; it does not replace a scriptable final editor. Free/open source, local capture, no stated watermark. The setup/UI and macOS permission/audio-routing work reduce minimalism, so validate actual start/stop and scene control before committing. |
| 4 | [Editly](https://github.com/mifi/editly) | **Deep-dive** | A focused Node + FFmpeg declarative editor: its official README states it makes videos from clips, images, audio and titles with transitions/music; JSON/JSON5 specs support 16:9 outputs, overlays/subtitles, picture-in-picture, audio mixing, ducking, and normalization. This is a smaller, more direct Codex-generated final-edit surface than raw FFmpeg or Remotion and requires only Node plus `ffmpeg`/`ffprobe` on PATH. MIT-licensed and local by architecture; verify current package maintenance and a real 1080p render because the inspected README, not a fresh release, is the availability evidence. |
| 5 | [Motion Canvas](https://github.com/motion-canvas/motion-canvas) | **Deep-dive** | TypeScript generator-based vector animation with an editor/preview, transitions, media, and rendering documentation. It is MIT-licensed and a good code-defined option for a short animated title card or explainer inserted into an FFmpeg/OBS workflow. It is weaker as the sole screen-video NLE and has a learning curve. The repository lists its most recent release as `v3.17.2` on 2024-12-14, so downstream work must verify practical compatibility/current maintenance before selection. |
| 6 | [whisper.cpp](https://github.com/ggml-org/whisper.cpp) | **Deep-dive** | A local caption/transcript component, rather than the editor: it documents MIT licensing, a `whisper-cli`, first-class Apple Silicon optimization (Accelerate/Metal/Core ML), and macOS Intel/Arm support. Combined with FFmpeg or Remotion, it can generate timed text for captions without sending narration to a hosted ASR service. Model download and transcript accuracy/timing still need a sample run; the documented CLI expects 16-bit WAV input, which FFmpeg can create. |
| 7 | [MoviePy](https://github.com/Zulko/moviepy) | **Skim-only** | Python API for cuts, concatenation, title insertion, compositing, effects, and common audio/video output; the project says it supports macOS and is MIT-licensed. It is approachable for Codex-generated scripts, but upstream explicitly notes it is slower than direct FFmpeg because of heavier frame import/export. It remains a reasonable fallback only if Python composition is materially faster for the author than FFmpeg/Editly. |
| 8 | [MLT / `melt`](https://www.mltframework.org/docs/melt/) | **Skim-only** | An actively released (`v7.40.0` listed June 2026) multitrack command-line editor with inputs, tracks, filters, and consumers; Homebrew publishes an `mlt` formula under LGPL-2.1-only. It can support scripted timelines/transitions and local export, but its own documentation calls it powerful yet obscure. Lower Codex ergonomics and a narrower learning-resource surface than FFmpeg/Editly make it a fallback, not first choice. |
| 9 | [auto-editor](https://github.com/WyattBlue/auto-editor) | **Skim-only** | Current CLI (`30.3.0`, May 2026) automatically removes silence or low-motion sections and can export timelines for Resolve, Final Cut Pro, Shotcut, and Kdenlive. Its public-domain source and cross-platform binaries make it a fast optional rough-cut preprocessor for narration-heavy screen capture. It does not supply polished titles, motion graphics, transitions, or final audio design, so it is only an adjunct to a renderer/editor. |
| 10 | [Manim Community](https://docs.manim.community/en/stable/guides/configuration.html) | **Skim-only** | Python-driven renderer for educational/mathematical animation and MP4 output. It offers deterministic code animation and free/open-source licensing, but its domain-specific primitives and scene model make it disproportionate for a generic product intro/outro unless the story genuinely benefits from diagram animation. It needs a macOS install/render probe before any commitment. |
| 11 | [DaVinci Resolve](https://www.blackmagicdesign.com/event/davinciresolvedownload) | **Skim-only** | Available for macOS 12+ and offers a powerful professional free desktop editor; its scripted/fusion surface makes partial automation possible. It can produce excellent transitions, titles, audio, and stitching, but it is a GUI-first, high-learning-curve workflow with edition/codec/feature differences that need precise verification. Include only if a human wants hands-on polish beyond the scriptable-first paths. |
| 12 | [LosslessCut](https://github.com/mifi/lossless-cut) | **Skim-only** | Current free/open-source macOS Intel/Apple-Silicon downloads, fast FFmpeg-backed trim/merge, a basic CLI/HTTP API, and visible FFmpeg command logs make it a useful human-in-the-loop rough-cut companion. Its own documentation says it is not designed for advanced batching/automation, and it is not a polished title/transition/motion editor. |
| 13 | [VHS](https://github.com/charmbracelet/vhs) | **Skim-only** | MIT-licensed, macOS-installable code-defined terminal recording that renders GIF, MP4, WebM, or PNG sequences. It is excellent for a reproducible terminal-only insert, with typed commands, themes, framing, and CI support. It cannot capture the actual graphical product UI or assemble a full narrated product demo, so scope it to a small optional clip only. |
| 14 | [GStreamer Editing Services / `ges-launch-1.0`](https://gstreamer.freedesktop.org/documentation/tools/ges-launch.html) | **Skim-only** | Officially supports command-defined timelines and rendering; macOS packages expose `gst-launch-1.0` tools. It could make a code-driven local pipeline, but the documentation describes `gst-launch-1.0` as primarily a debugging tool and advises building an application for production. Its pipeline/plugin complexity is too high for this short deadline unless an existing GStreamer skillset exists. |
| 15 | [Blender](https://www.blender.org/download/) | **Reject** | Blender can be automated/headlessly and has a VSE, but it is an unnecessarily large 3D/DCC environment for a sub-three-minute screen-demo edit. The browser could not fetch the authoritative CLI manual, so the exact current command surface remains unsupported in this run; its expected setup and learning cost fail the minimal/fast criterion. |
| 16 | [GStreamer `gst-launch-1.0`](https://gstreamer.freedesktop.org/documentation/tools/gst-launch.html) alone | **Reject** | It is available on macOS and technically scriptable, but official docs explicitly position it as a basic-pipeline/debugging tool rather than an application-building base. Treat GES as the only potentially relevant GStreamer-level candidate; standalone `gst-launch` should not be selected as the video-production workflow. |

**Evidence:** The first six are the only candidates that jointly cover the needed production roles without forcing a paid, watermark-prone browser service: OBS or FFmpeg capture; FFmpeg/Editly/Remotion final rendering; Motion Canvas or Remotion title animation; and whisper.cpp captions. Each has an official local executable/code path and an inspectable license/availability source. Editly’s README specifically verifies the required layers, transitions, subtitles, dimensions, audio mixing/ducking/normalization, and macOS prerequisites; OBS verifies native external control; Remotion verifies CLI codecs, dimensions, duration, props, audio, and headless defaults.

**Implication:** Preserve two distinct finalist paths for the deep-dive wave: **OBS + FFmpeg** for the lowest-risk, fully local and minimal stack; and **OBS/FFmpeg + Remotion** for higher-motion-design quality when a TypeScript composition is acceptable. Evaluate Editly as the possible short-form middle ground, Motion Canvas only for standalone animated inserts, and whisper.cpp as a local accessibility component rather than a competing editor.

### Capability Coverage and Automation Boundary

No single scriptable candidate dominates every stage. FFmpeg is the broad local media engine and already available in this workspace, but its graph syntax makes sophisticated branded motion design costly. Remotion and Motion Canvas shift titles/animation into TypeScript, making them more legible to Codex and easier to keep under version control; neither is an interactive screen recorder. OBS captures UI and can receive external WebSocket control but needs post-production. Editly composes existing assets via a small declarative format, while auto-editor and LosslessCut only accelerate the rough-cut stage.

**Evidence:** FFmpeg documents simple and complex filtergraphs, explicit stream maps, output codecs, subtitle handling, device input, and the `concat` filter. Remotion’s render CLI accepts composition props, dimensions, duration, codec/audio settings, and default headless operation. Motion Canvas describes itself as a TypeScript animation library plus real-time editor, designed for informative vector animations synchronized with voice-overs. OBS states that external tools can automate/control its scenes and sources through its built-in WebSocket. Editly documents a JSON/JSON5 edit specification, title/subtitle overlays, transitions, audio controls, and 16:9 output.

**Implication:** Give Codex ownership of a repository-held script/specification, asset manifest, reproducible render commands, and automated output inspection. Keep human work at the quality-critical boundary: record the real product in OBS or with a validated FFmpeg device input, approve the narrative/assets, review the final 1080p file, and publish it. Do not rely on an LLM to drive an arbitrary GUI editor for the primary cut.

### Cost, Watermark, License, and Privacy Risks

The preferred deep-dive tools avoid a vendor export watermark or subscription for local source code/rendering: FFmpeg, OBS, Editly, Motion Canvas, and whisper.cpp are free/open-source; their relevant licenses are LGPL/GPL-conditional (FFmpeg), GPL (OBS source), MIT (Editly, Motion Canvas, whisper.cpp), or public domain (auto-editor). Remotion is source-available under its own license model, not a blanket open-source substitute: free-license eligibility and Company License requirements must be confirmed for the entrant’s status. Remote caption, narration, stock music, and browser-editor services are outside this candidate scan; do not assume any is watermark-free, commercially usable, or private without its current official terms.

**Evidence:** FFmpeg’s license says most files are LGPL 2.1+ and enabling GPL components changes the build to GPL v2+; Motion Canvas, Editly, MoviePy, VHS, and whisper.cpp list MIT licenses; auto-editor states its source is public domain. Remotion’s current licensing docs identify Free License eligibility, Company License requirements, and explicit telemetry behavior. OBS identifies itself as open source; its WebSocket guide recommends password protection because it permits external control.

**Implication:** Use local-only assets and processing by default, preserve provenance for music/fonts/visual assets, and bind the render to a known local FFmpeg build. If using Remotion, select a rendering package deliberately and document any telemetry/licensing choice. If exposing OBS WebSocket, bind it safely and use a password; never store its secret in the video repository.

### Immediate Discovery-Wave Shortlist

1. **FFmpeg + OBS + whisper.cpp** — best-supported free, local, reproducible baseline; validate capture permissions/audio, a 1080p FFmpeg composition, captions, and final H.264 playback.
2. **FFmpeg/OBS + Remotion** — best conditional path for a polished code-authored intro/outro, title treatment, and animated explainer; validate license eligibility, render speed, and any telemetry setting before adoption.
3. **Editly + FFmpeg** — strongest potentially minimal code-first complete editor; validate recent package health and a small real video spec before relying on it.
4. **Motion Canvas + FFmpeg** — targeted animation-insert alternative; validate current Node compatibility and MP4 export workflow before selection.

**Evidence:** These paths use complementary tools rather than asking one candidate to be both recorder, motion-graphics suite, NLE, audio mixer, and captioning system. Their primary evidence is current official docs/repositories, and every selected path retains a no-hosted-media alternative.

**Implication:** The downstream deep dives should compare a small, concrete 1080p trial for the first three paths. They should report measured render time, actual output dimensions/FPS/audio, required permissions, asset/command count, and any license or telemetry decision; until then, claims of fastest practical end-to-end production remain unsupported.
