# Cross-Harness Adapter Contract

## Assignment

**Goal:** Compare the eight researched harness contracts, rank their suitability for CLI-managed direct installation of one canonical Skill Issue payload, and define the minimum shared adapter contract for the later lifecycle CLI.

**Scope:** The eight constrained assignment reports, with targeted first-party cross-checks of the shared Agent Skills shape and Grok Build's independent skill roots. The product decision is fixed: the later CLI directly installs a canonical payload; it does not create a native plugin for each harness.

**Exclusions:** Implementing the lifecycle CLI, publishing marketplace/package artifacts, guessing undocumented host behavior, managing remote/cloud workspaces without a documented local-filesystem contract, or changing source materials.

## Findings

### The canonical payload is a directory, not a lowest-common-denominator plugin

The common direct-install artifact is one owned directory named with a portable skill identifier, containing `SKILL.md` with YAML `name` and non-empty `description`, plus referenced supporting files. GitHub Copilot, Claude Code, Codex, Cursor, Gemini CLI, OpenCode, and Pi explicitly support this pattern. Antigravity uses the same folder shape. Grok Build documents reusable skill folders with Markdown instructions, scripts, and resources, but does not publish the loose-skill entry-file or frontmatter contract; it therefore cannot validate the canonical shape as strongly as the other targets.

**Evidence:** Assignments 01–05 and 07–09 each document a required `SKILL.md` directory contract. Assignment 06 records Grok Build's folder contract and its missing schema; the independently checked [xAI Skills documentation](https://docs.x.ai/build/features/skills-plugins-marketplaces) lists loose skill roots and does not specify `SKILL.md` or frontmatter. [Codex](https://developers.openai.com/codex/skills), [Claude Code](https://code.claude.com/docs/en/slash-commands), and [OpenCode](https://opencode.ai/docs/skills) independently corroborate the mainstream direct-folder model.

**Implication:** Maintain one complete, immutable-at-release canonical payload. The shared validator requires `SKILL.md`, a portable lowercase-hyphen `name` equal to the directory basename, a non-empty `description`, and all referenced support files inside the directory. It must copy the entire directory and must not add plugin manifests, hook configuration, MCP configuration, runtime code, or host-specific frontmatter by default. Grok accepts this payload only as a **caveated** materialization candidate after live `grok inspect` verification.

**Evidence:** The path, activation, policy, and lifecycle evidence is synthesized from assignments 01–09. Assignment 05 establishes that Antigravity Desktop, Antigravity CLI, and Gemini CLI are separate path contracts. Assignment 06's first-party evidence classifies Grok Build as a supported standalone direct-harness target, while limiting loose-skill claims; the xAI cross-check also documents compatible `~/.agents/skills` discovery.

**Implication:** The lifecycle CLI should implement one declarative filesystem engine plus host descriptors. It should offer common adapters for the eight strong local contracts, a surface-selecting Google descriptor, and a Grok descriptor that fails closed when `grok inspect` cannot prove discovery. Treat remote/cloud claims as per-surface capabilities, never as an inferred consequence of a repository copy.

### Filesystem materialization is shared; activation and verification are bespoke

All supported local targets can receive an owned directory without a normal configuration edit. Their discovery cadence differs: Copilot reloads or starts a new session; Claude watches already-existing roots; Codex automatically detects changes with restart/app-server reload fallback; Cursor needs a new chat; Gemini provides `/skills reload`; Antigravity requires a new conversation/relaunch; OpenCode requires a fresh process/server; and Pi requires `/reload` or restart. Only Gemini offers a direct skills-management command set for tracked installs; Copilot's `gh skill` manages its own provenance path, rather than arbitrary copied folders.

**Evidence:** The harness assignments document direct-folder discovery and their respective reload/verification interfaces. The absence of a direct loose-skill lifecycle API is explicit for Antigravity, Grok, OpenCode, Cursor, Pi, and the generic Copilot manual-copy route.

**Implication:** `activate` and `verify` belong to each adapter, not to the shared copier. A successful transaction means: final directory exists, the host reports the expected discovered path/name, and a harmless explicit invocation or host-equivalent activation succeeds. It must report "materialized, verification pending" where no live host process is available.

### Collision, configuration, and trust controls prohibit blind install

Same-name behavior varies: Copilot is precedence-based; Claude uses scope/nesting rules; Codex can expose both and makes an unqualified selector ambiguous; Gemini defines higher-scope precedence; Pi keeps first discovered; and Cursor, Antigravity, and Grok do not document a safe loose-skill precedence contract. Native-root copies normally need no settings mutation, but optional extra-path mechanisms do: Copilot environment variables, Claude `skillOverrides`, Codex `[[skills.config]]`, Grok `[skills] paths`, Pi `skills` arrays, and OpenCode source-current or configured paths. Project trust and managed policy can also suppress an otherwise correctly copied folder.

**Evidence:** Assignments 01–09 provide the host-specific collision, configuration, trust, and policy rules. Grok's first-party page independently confirms `[skills] paths` and enabled-plugin roots, while its collision rules remain undocumented.

**Implication:** The normal adapter path is native-root copy with **no** config patch. Before any change, enumerate known roots and fail on a same-name non-owned candidate. Extra-path configuration is an explicitly requested advanced mode that must use a host-aware structural patch, preserve unrelated data/comments where the host format permits, record the exact inserted selector/value, and roll back only that record. The CLI must never auto-trust a project, alter approval/sandbox settings, enable hooks, or weaken policies.

### The common lifecycle contract is an implementable, receipt-backed interface

The later CLI needs a host descriptor and an owned-state ledger because direct folder installs are not uniformly package-managed. The contract below is the minimum; adapters may return a documented `unsupported`, `caveated`, policy-denied, or verification-pending state rather than manufacturing a host capability.

**Evidence:** Assignments 01–09 consistently distinguish direct copies from plugins/packages and identify missing native rollback/repair commands. Assignments 01, 03, 05, 07, 08, and 09 explicitly recommend a staged directory replacement and owned receipt as adapter safeguards.

**Implication:** Implement these operations around a shared transaction engine:

| Operation            | Required behavior and record                                                                                                                                                                                                                                                                                                                                                              |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `detect`             | Locate executable/app surface, read version where available, determine operating system/home/config overrides, detect policy/trust blockers, and return only documented capabilities. Examples: Copilot `gh skill` is preview/optional; OpenCode needs `>=1.0.186`; Pi feature handling is `>=0.50.0`; Gemini must expose skills commands; Grok needs `grok inspect`. |
| `enumerateScopes`    | Return native writable project/user scopes, exact roots, surface reach, restart/reload action, collision model, and whether admin/cloud scope is supported. It must keep Antigravity Desktop, `agy`, and Gemini separate and must label unsupported remote surfaces.                                                                                                                      |
| `preview`            | Validate the canonical payload before writes; produce a manifest, digest, source version/ref, executable-file inventory, target path, existing same-name candidates, planned config patch, trust/policy warnings, and rollback plan. Preview performs no copy, config change, trust approval, or remote download.                                                                         |
| `ownedStateLedger`   | Store outside the installed payload: adapter ID/version, host version, scope/root/final path, canonical skill name, source URI/ref/digest, manifest digest, install time, backup location/digest, verification result/time, and exact config patch identity. Store no credentials and never use user-editable skill content as the only ownership proof.                                  |
| `installTransaction` | Lock the target, stage the whole payload beside the final directory, revalidate manifest/frontmatter, back up only a ledger-owned existing target, atomically rename where supported, write the ledger only after placement, then call `activate` and `verify`. On failure, restore the prior owned backup; retain non-destructive staging evidence when restore itself fails.            |
| `patchConfig`        | Optional only. Parse the host format (TOML/JSON/JSONC), make one idempotent exact-path/entry change, preserve unrelated configuration, write atomically, and record the reversible fragment. Refuse if policy controls the value or faithful structural preservation is unavailable.                                                                                                      |
| `activate`           | Execute the host descriptor's documented refresh action or return `restart-required`: for example `/skills reload`, `/reload`, fresh session/chat, process restart, or Codex `skills/list` force reload. It must not invoke unrelated skill/plugin/package installers.                                                                                                                    |
| `verify`             | Check expected name, resolved path, enabled/visible state, and a harmless explicit invocation. Prefer structured host output (`grok inspect --json`, Codex app-server list, Gemini list) where documented. Return distinct `installed`, `discovered`, and `activated` results.                                                                                                            |
| `diagnose`           | Re-run detection and enumeration, inspect receipt/manifest/final path/frontmatter, report collisions, disabled settings, policy/trust denial, stale session, config patch state, and host diagnostics. It must identify unvalidated host facts as caveated rather than guessing a fix.                                                                                                    |
| `update`             | Preview a new payload; require matching ledger ownership or explicit user-authorized replacement; stage, back up, replace, activate, verify, then update digest/ref in the ledger. Do not delegate a plain copy to host package update commands.                                                                                                                                          |
| `repair`             | Compare final manifest and path with the ledger; restore the last verified owned backup or reinstall the declared source after preview. Stop on absent/foreign ownership rather than deleting a possibly user-owned directory.                                                                                                                                                            |
| `rollback`           | Restore the recorded last verified backup and the exact recorded config fragment only, then activate/verify. Report unsupported when no verified owned backup remains.                                                                                                                                                                                                                    |
| `uninstall`          | Confirm ledger ownership and target identity, remove only that directory and its exact recorded config fragment, retain unrelated roots/skills/settings, activate/restart as applicable, verify absence, then remove or archive the ledger.                                                                                                                                               |

### Candidate routes have a clear depth ranking

The canonical direct-folder adapters are the only **deep-dive** route: they meet the product decision and have first-party discovery evidence. Shared `.agents/skills` roots and host-compatible `.claude/skills` roots are **skim-only** portability options: they can reduce duplicated payloads, but change precedence/collision and surface reach and must be deliberately selected per host. Native `gh skill` and Gemini `skills install/link` are **skim-only** lifecycle alternatives because they add host-owned provenance, links, or distribution semantics rather than a universal direct-copy contract.

Native plugins, marketplace packages, extensions, hooks, MCP/LSP configuration, rules files, IDE caches, hidden filesystem roots, and automatic trust/permission changes are **rejected** as the default route. They either extend behavior beyond an instruction skill, introduce independent package/config lifecycle, or lack evidence as direct-install targets. The following remain explicitly **unsupported or caveated**: Grok loose-skill entry schema and collision precedence; Antigravity loose-skill reload/rollback/collision semantics; Codex cloud direct-skill discovery; Cursor Background Agent skill loading; and Claude user-scope skill propagation to cloud/Cowork.

**Evidence:** Assignments 01–09 each distinguish plain skills from plugins/packages and record their host-specific unsupported boundaries. The first-party [xAI Grok Build skills documentation](https://docs.x.ai/build/features/skills-plugins-marketplaces) validates Grok's folder roots, plugin distinction, and compatibility discovery without supplying the missing loose-skill schema.

**Implication:** Build the lifecycle CLI around the deep-dive route first. Expose skim-only modes as explicit, separately named adapters with their own receipts. Return a clear unsupported/caveated result for the listed gaps; do not silently fall back to a plugin, config mutation, duplicate directory, or cloud claim.

## Notes

- **Cross-check result:** The xAI source supports assignment 06's classification of Grok Build as a standalone direct-harness target and confirms `.grok/skills`, `~/.grok/skills`, plugin roots, `[skills] paths`, plus `~/.agents/skills` compatibility. It still does not define a loose-skill `SKILL.md`/frontmatter schema or collision precedence, so Grok remains a bespoke, caveated adapter rather than an unsupported harness.
- **Caveat:** “Common adapter” means shared canonical payload and shared filesystem transaction mechanics. It does not mean shared discovery roots, automatic activation, trust handling, configuration merge behavior, or cloud availability.
