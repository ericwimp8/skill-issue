# Grok Build Cross-Check

## Assignment

**Goal:** Cross-check Grok Build harness-portability claims against current primary xAI and OpenAI sources, resolve product and access ambiguity, and decide the candidate under the required first and Codex-subscription second gates.

**Scope:** Assignments 01, 02, 04, 05, 08, and 13; current Skill Issue production source; current first-party xAI Grok Build documentation and public source repository; and current official OpenAI billing and Codex documentation. Research date: 2026-07-21.

**Exclusions:** Installing, authenticating, configuring, or running Grok Build; model inference; credential inspection or extraction; proxy setup; implementation planning; and claims about Grok Chat, API-only usage, IDE integrations, or non-xAI CLI clients.

## Findings

### Product boundary and current local route

Grok Build is xAI's first-party terminal coding agent, released as `grok`, with interactive, headless, and ACP modes. It is distinct from Grok Chat, direct xAI API use, xAI IDE integrations, and any other terminal client. Skill Issue registers `grok-build` for ordinary skill materialization at `.grok/skills`, but its evaluation registry admits only Codex, Cursor, Claude Code, and Pi. The Grok replay command map is therefore an unreachable stub rather than a qualified route.

**Evidence:** The [overview](https://docs.x.ai/build/overview) identifies Grok Build as a coding agent usable through a TUI, headless scripts, and ACP; the [first-party repository](https://github.com/xai-org/grok-build) identifies the `grok` CLI/TUI and agent runtime. `harness.ParseEvaluationID` excludes Grok Build; `prepareRuntime` returns an unsupported-evaluation-harness error before authentication, temporary skill materialization, or replay. The local availability probe found no `grok` executable.

**Implication:** The local access condition is **blocked pending named evidence or access**: an operator-provided `grok` executable, authenticated model access, and non-destructive live qualification evidence. It is not current evaluator support.

### Codex-subscription route and second-gate outcome

The xAI-native Grok Build routes require xAI session authentication or an xAI API key. Its documented custom-model/BYOK configuration takes a provider endpoint and provider credential through a model configuration; xAI explicitly treats endpoints outside `x.ai` as third-party BYOK. OpenAI documents API billing as separate from ChatGPT, while Codex documents its own ChatGPT-plan sign-in separately from API-key use. Neither product documents a Grok Build authentication provider, entitlement exchange, or supported credential route that consumes an existing Codex/ChatGPT subscription allowance.

**Evidence:** [Enterprise Deployments](https://docs.x.ai/build/enterprise) lists Grok's browser OIDC, device code, external auth provider, and xAI API-key methods, and states that third-party endpoints retain their own IAM. [Settings](https://docs.x.ai/build/settings) documents `base_url`, `env_key`, and API backend selection for a model. [OpenAI billing guidance](https://help.openai.com/en/articles/8156019-is-api-usage-included-in-chatgpt-subscriptions-even-if-i-have-a-paid-chatgpt-account) states that API service is separately billed and managed. The [Codex repository](https://github.com/openai/codex) documents ChatGPT-plan sign-in for Codex and a separate API-key mode. No inspected primary source documents a Grok Build `openai-codex` provider, ChatGPT/Codex subscription exchange, or use of Codex-managed OAuth tokens.

**Implication:** The Codex-subscription second-gate outcome is **remove from the supported campaign**. Under the caller's rule, this second-gate failure is decisive even if the first-gate evidence were later supplied.

### Privacy, security, and rejected alternatives

The candidate has concrete security boundaries that cannot be normalized into a subscription route. A fresh `$GROK_HOME` can keep created skill/session/config state disposable, while project and system policy still participate in discovery. A narrow sandbox and permission rules can constrain tool use, but they do not prove absence of ambient instructions or full child cleanup. Grok Build's normal inference/authentication paths also require external xAI services and xAI-issued access rather than the user's existing OpenAI subscription entitlement.

**Evidence:** [Settings](https://docs.x.ai/build/settings), [Skills, Plugins & Marketplaces](https://docs.x.ai/build/features/skills-plugins-marketplaces), and [Enterprise Deployments](https://docs.x.ai/build/enterprise) collectively document the separate configuration scopes, discovery sources, authentication endpoints/methods, and sandbox behavior. Assignment 05 establishes that user-owned native credential state and subscription access stay with the selected harness, while assignments 08 and 13 establish the absence of a safe Grok-to-Codex route.

**Implication:** Reject these alternatives:

- Native xAI login or `XAI_API_KEY`: lower fit because it requires separate xAI access and fails the specified Codex-subscription condition.
- OpenAI API-key BYOK: lower fit because API billing and credentials are separate from a ChatGPT/Codex subscription; it also needs an independently qualified provider/model/configuration route.
- External auth provider configured as a Codex bridge: unsupported because no xAI or OpenAI source documents a supported Codex entitlement exchange.
- Copying Codex OAuth, personal-access, or refresh credentials into Grok Build: rejected unsafe because credential ownership and refresh remain Codex-managed.
- A credential-extracting proxy or a broadly approving `--always-approve`/`--yolo` run: rejected unsafe because it expands credential or tool-execution exposure without satisfying the required route.

### Exact unsupported evidence and final decision

No inspected source or local probe supplies: a local Grok Build executable; native authenticated model access; a supported Grok Build route using only existing Codex/ChatGPT subscription access; a complete clean/default discovery exclusion; exclusive temporary-skill activation; per-turn marker attribution; a Grok-specific terminal completion contract in Skill Issue; a bounded all-child cancellation proof; or a live cleanup proof. The documented BYOK interface proves configurable endpoint support, not compatibility with a Codex subscription or with any particular OpenAI endpoint/model.

**Evidence:** The local probe, current production runtime gate, and primary source set cited above. The absence statements are bounded to the inspected primary materials and no live Grok Build run was performed.

**Implication:** **remove from the supported campaign**. Preserve the rejected alternatives and unsupported evidence as the record of why no implementation or workaround should be advanced for Grok Build under this campaign's access constraint.
