# OpenCode Cross-Check: Harness Portability Decision

## Assignment

**Goal:** Cross-check OpenCode’s local command surface and the earlier portability assignments against current first-party OpenCode and OpenAI sources, then issue one evidence-based harness-campaign decision.

**Scope:** Assignments `01`, `02`, `04`, `05`, `09`, and `14`; current OpenCode documentation and source; the locally installed OpenCode executable; and existing Skill Issue production source.

**Exclusions:** Authentication, credential inspection, model calls, session creation/deletion, configuration or skill changes, permanent workarounds, source changes, and implementation planning.

### First technical gate is blocked pending finite technical evidence

OpenCode has enough released-source and local-help primitives to remain a candidate: headless one-shot execution, JSON envelopes, session continuation, model variants, configurable permissions, skills, and session deletion. That is not a qualified harness route. The local binary is older than the current-source control set, its two read-only discovery/configuration probes fail on its pre-existing database, and Skill Issue cannot presently launch it.

**Evidence:** Local `1.14.39` help and error observations; released `v1.14.39` source establishes `sessionID`, `tool_use`, `error`, `idle`, auth/data roots, and configuration/skill precedence; `04` and `09` establish the current evaluator rejection and absence of candidate-specific protocol/cleanup evidence.

**Implication:** The first gate changes only with all of the following finite evidence on one pinned OpenCode release and one user-authorized provider/model: a clean process launch that proves its exact supported ambient-exclusion controls; generated-skill discovery while ambient skills are absent; two JSON turns using one stable `sessionID`; completed marker-command attribution and denied-marker failure classification; requested/effective model and variant; interruption proving no surviving child or server; explicit deletion of the created session; and removal of every run-owned root. This is evidence required to change the decision, not a claim that a route is already available.

### Native OpenAI/Codex-subscription route passes only its source-level viability condition

Current OpenCode documentation and upstream source describe an `openai` provider route using ChatGPT Plus/Pro authorization and a native OpenAI/Codex OAuth plugin. That route is meaningfully different from OpenAI API-key or compatible-endpoint routes: OpenAI documents API billing separately from ChatGPT subscriptions, so API keys, proxies, and BYOM configurations do not prove use of an existing paid ChatGPT/Codex entitlement without separate credentials or billing.

The route remains conditional, not local evidence. The installed `1.14.39` help shows generic provider-login flags only; no local provider method, authorization status, entitlement, model availability, or `openai/<model>` plus `--variant` execution was inspected. Its credential store lives in user-owned OpenCode data and may refresh in place. No safe fully temporary auth bridge was evidenced.

**Evidence:** [OpenCode providers documentation](https://opencode.ai/docs/providers), the current [OpenAI/Codex plugin source](https://github.com/anomalyco/opencode/blob/4872c48c230728150e8e3406722943450ed58dcb/packages/opencode/src/plugin/openai/codex.ts), `v1.14.39` [auth source](https://github.com/anomalyco/opencode/blob/v1.14.39/packages/opencode/src/auth/index.ts), and assignment `14` distinguish native OAuth from API-key/proxy routes.

**Implication:** **Second gate: conditional native viability only.** It changes to a passed second gate only when the operator already owns an eligible OpenCode native `openai` login and an authorized `openai/<model>`/variant can be preflighted without credential inspection or state modification. Existing Codex CLI login cannot be treated as transferable OpenCode credentials, and credential copying or permanent-configuration workarounds are rejected.

### Decision: blocked pending named evidence or access

**Decision:** **Blocked pending named evidence or access.** Keep OpenCode out of the supported campaign and do not implement from the present record.

**Evidence:** The current evaluator rejects OpenCode before any replay path; the released event key is incompatible with its generic session parser; local configuration/skill discovery fails before output; no authorized provider/model run exists; isolation, marker attribution, permission denial, cancellation/children, session deletion, and root cleanup are unsupported; and the current upstream subscription route has not been shown on the installed release.

**Implication:** The block is finite and can be revisited with the first-gate evidence named above plus access to a user-owned eligible OpenCode `openai` login for a non-mutating preflight. The record rejects credential copying, `OPENCODE_AUTH_CONTENT`, auth-file bridges, API-key/proxy substitution for the subscription gate, and permanent OpenCode configuration changes.

## Notes

- Local executable observations are limited to non-mutating help/path/debug commands. The two database failures are local access blockers, not evidence that an isolated temporary runtime would fail.
- Current upstream OpenCode source and documentation establish capabilities that may differ from the locally installed `1.14.39`; this document treats them separately throughout.
- No candidate ecosystem classification applies: this is a single installed CLI cross-check, with no package/repository selection round.
