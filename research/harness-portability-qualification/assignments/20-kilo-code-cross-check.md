# Kilo Code CLI Cross-Check

## Assignment

**Goal:** Cross-check the Kilo Code CLI candidate against current primary documentation and first-party source, distinguish CLI behavior from IDE-only behavior, and make one qualification decision for the Skill Issue harness-portability campaign.

**Scope:** Assignments 01, 02, 04, 05, 10, and 15; current Kilo documentation; Kilo-Org/kilocode source at commit `0cf9f902e0eaead2746a935e9440d36b84df45bf`; and a read-only local executable lookup on 2026-07-21.

**Exclusions:** Installing Kilo, logging in, issuing a model request, changing Kilo or Skill Issue configuration, copying credentials, implementing an adapter, and using IDE-only observations as CLI proof.

### First Gate Decision

**Decision: blocked pending named evidence or access.** Kilo has a documented headless CLI, JSON events, session-selection interface, temporary project-skill location, granular permissions, and a user-owned native ChatGPT/Codex OAuth route. That is insufficient for implementation because the local executable is absent and the first gate has no proof of exclusive skill/config discovery, safe marker authorization, a terminal/resume event contract, daemon-free process ownership, interruption behavior, or bounded state cleanup.

**Evidence:** Assignment 02 requires executable/version, native access, fixed model/reasoning, isolation, replay, structured completion/attribution, and cleanup evidence before qualification. Assignment 04 establishes that Kilo is not locally installed and current production rejects it before any temporary-skill or replay path. Kilo's own documentation and pinned source confirm the relevant interfaces, while its architecture confirms daemon attachment and persistent-state surfaces.

**Implication:** Do not add a Kilo evaluation adapter or describe Kilo as campaign-supported. This is a finite evidence block rather than removal from the expansion-candidate campaign.

### Finite Evidence That Could Change The Decision

For one explicitly selected installed Kilo CLI release at version `>= 1.0`, record all of the following in a user-authorized, non-destructive qualification run. The run must use a user-owned native Kilo OpenAI OAuth login already present before the run; it must not copy credentials, inject serialized credential content, substitute a home directory containing copied auth, or make permanent configuration changes.

1. Resolve the executable and record `kilo --version`, platform, and a non-secret native OpenAI/Codex account-and-model availability check.
2. Use a unique evaluator-owned workspace, output root, temporary project skill, and temporary configuration/state roots where possible. Record `kilo debug config` and `kilo debug skill` before each turn, proving that project/global/compatibility skills, configured paths/URLs, plugins, MCP, and remote features are either absent or deliberately excluded; show the generated skill is the sole evaluation skill.
3. Run an initial and `--session` resumed prompt with `--format json`, fixed model and variant, then retain private raw evidence that shows one stable session ID, explicit successful terminal completion for each turn, no error event, and turn-attributed marker execution.
4. Pass a run-scoped restrictive permission profile that allows the generated skill and exact marker command only. Demonstrate that an unrelated shell or external-directory request is denied without `--auto` or `--dangerously-skip-permissions`.
5. Prove a managed daemon cannot be attached, or prove the candidate invocation and cleanup own every attached/created server, child, and descendant. Repeat with cancellation and show no live Kilo process/session work remains.
6. Inventory every file and network endpoint touched during normal completion and cancellation. Demonstrate that all newly created removable state lies in evaluator-owned paths, while the pre-existing user OAuth state remains unread, unmodified, and outside cleanup.

### Second Gate Outcome

**Conditionally passes native subscription access; overall candidate remains blocked.** Kilo's current first-party documentation explicitly states that ChatGPT Plus/Pro subscription OAuth works in the CLI, consumes subscription limits instead of separately billed API usage, and is limited to Kilo's Codex catalogue. Its documented eligible plan list also includes Business, Edu, and Enterprise where Codex access is included. Effective account entitlement and selected-model availability remain live prerequisites.

**Evidence:** [Kilo's ChatGPT subscription documentation](https://kilo.ai/docs/ai-providers/openai-chatgpt-plus-pro) identifies the CLI OAuth route, billing boundary, eligible plans, and catalogue limitation. Assignment 15 cross-checks the same route against pinned Kilo source and official OpenAI billing guidance.

**Implication:** Native Kilo OAuth is the only acceptable Codex-subscription route for this candidate. OpenAI API keys, Kilo Gateway credentials, BYOK, custom proxies, extracted OAuth tokens, copied auth files, and permanent-config workarounds do not satisfy this gate.

## Notes

- **Unsupported exact evidence:** local Kilo release/version; authenticated model availability; `auth list` machine-readable success semantics; exclusive ambient exclusion; exact Kilo permission matching for the marker command; two-turn JSON completion schema; live skill activation and marker attribution; no-daemon mode; cancellation/descendant cleanup; touched-state inventory; bounded deletion; and whether an authenticated run can preserve native OAuth while keeping all generated state evaluator-owned.
- **CLI-only boundary:** the matrix relies on `kilo run`, CLI documentation, and `packages/opencode` source. VS Code SecretStorage, extension settings, and editor-spawned server behavior are not evidence for this candidate except where Kilo documentation explicitly names the CLI.
- **Validation performed:** read-only local executable lookup; read-only inspection of the current Kilo primary documentation and source commit; no Kilo installation, login, configuration change, prompt, or credential access.
