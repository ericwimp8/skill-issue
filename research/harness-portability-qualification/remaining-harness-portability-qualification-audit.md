# Remaining Harness Portability Qualification Audit

**Research date:** 2026-07-21  
**Scope:** OpenAI Codex, Cursor, Claude Code, and Pi as the qualification baseline; GitHub Copilot CLI, Gemini CLI, Grok Build, OpenCode, and Kilo Code as the five deep-dive candidates.  
**Authority:** The constrained assignment corpus under [`assignments/`](assignments/) only. Production source findings in that corpus control behavior; documentation and smoke evidence qualify or validate only the surfaces they actually cover.

## Best-Supported Direction

Do not implement any of the five remaining adapters from the present record. Keep only **OpenCode** and **Kilo Code** as finite, evidence-blocked candidates. Prioritize a version-pinned OpenCode qualification probe first because an executable was locally observable and released source defines headless JSON, session continuation, skills, permissions, and session deletion. Retain Kilo as the second candidate because its first-party CLI and native ChatGPT/Codex OAuth route are source-backed, but local executable access, daemon ownership, credential-preserving isolation, and cleanup remain unproved. ## Rejected and Lower-Fit Interpretations

The five named CLIs were deep dives. Adjacent official provider and proxy surfaces were skimmed only far enough to classify them when they were unsafe, required separate access, or were lower fit: Copilot BYOK and enterprise custom models; Gemini's Gemini-protocol gateway; Grok third-party BYOK and external auth provider; OpenCode API-key/custom-provider endpoints; and Kilo Gateway, BYOK, and custom endpoints. These are provider configuration capabilities, not evidence that a paid Codex/ChatGPT subscription transfers to another harness. ([Assignments 11–15](assignments/11-github-copilot-codex-subscription.md))

Reject all of the following across candidates:

- copying, reading, printing, serializing, symlinking, or exporting credential files, OAuth tokens, cookies, keychain entries, or refresh material;
- treating a Codex CLI login, Pi auth directory, Cursor credential, or another harness's launcher as a transferable provider credential;
- Skill Issue-owned installation, repair, login, migration, backup, restore, logout, provider setup, or normal configuration mutation;
- permanent user/project skill, plugin, policy, trust, provider, or organization configuration changes made only to support evaluation;
- invasive home-directory substitution that requires copied credentials or lets refresh writes mutate normal user state through a run-owned path;
- undocumented compatibility assumptions, credential-extracting proxies, shared remote proxies, or a new Skill Issue-managed translation gateway;
- blanket permission modes such as `--yolo`, `--always-approve`, `--allow-all`, `--auto`, or `--dangerously-skip-permissions` as substitutes for a least-privilege noninteractive contract;
- claims that executable discovery, skill materialization, help output, source-level capability, or a two-turn smoke is campaign qualification. ([Existing authentication patterns](assignments/05-existing-auth-and-subscription-patterns.md), [Product support contract](assignments/02-product-support-and-setup-contract.md))

## Shared Qualification Contract

### Goal

Qualify a repeatable Skill Issue evaluation route for an already installed, operator-owned coding-agent CLI. A qualifying route must preserve the normal ownership of the harness executable, login, subscription, credentials, managed policy, and ordinary configuration while Skill Issue temporarily supplies evaluation skills, invokes one controlled conversation, captures sufficient evidence, and removes only state it created. ([Existing runtime implementation](assignments/01-existing-runtime-implementation.md), [Product support contract](assignments/02-product-support-and-setup-contract.md))

### Non-Goals

Skill Issue does not install or repair a target CLI; authenticate an account; migrate, back up, restore, or reset native state; own provider configuration; make a compatibility proxy; erase ordinary harness history; restore arbitrary workspace effects; or infer model access from account identity. It also does not turn a successful payload copy, discovery listing, source inspection, or bounded smoke into published support. ([Product support contract](assignments/02-product-support-and-setup-contract.md))

### Constraints

1. The target executable, native authentication, policy permission, and requested model access already exist and remain operator-owned.
2. Evaluation uses the narrowest supported native controls and maximal safe isolation. Unavoidable system or managed policy is recorded, not bypassed.
3. Temporary skills and runtime state are created only in Skill Issue-owned paths. Cleanup deletes only those paths and owned processes.
4. Prompts are replayed verbatim in one resumable or continuously live primary-agent conversation.
5. Model outcomes are scored only after a tooling-complete run. Launch, authentication, model, permission, session, protocol, marker, cancellation, or cleanup failures are tooling failures.
6. Credentials, private identifiers, local absolute paths, and raw secrets are excluded from public artifacts. Optional diagnostic artifacts are sanitized.
7. Documentation establishes candidate capability; a pinned executable and live, non-destructive observations establish qualification.

### The Two Gates

**Gate 1 — safe technical evaluation viability.** The route must prove executable/version/platform discovery; an accepted source-owned evaluator entry; maximal safe clean/default isolation; exclusive or fail-closed temporary skill supply; fixed model/reasoning inputs; prompt-free noninteractive execution; one stable multi-turn conversation; strict protocol, completion, failure, session, tool, and skill-activation/marker evidence; narrow permissions and sandbox behavior; user-owned authentication; cancellation and complete child ownership; bounded temporary state; and normal plus interrupted cleanup. ([Existing runtime implementation](assignments/01-existing-runtime-implementation.md))

**Gate 2 — existing Codex/ChatGPT subscription viability.** Only after Gate 1 passes, the route must use the operator's existing eligible OpenAI Codex/ChatGPT subscription and model access through a native, source-backed harness route, without a separate API key/credit balance, another harness subscription, copied credentials, ordinary-state mutation, or an unsupported proxy. A source-level native OAuth route may remain conditional until the pinned executable and account/model preflight prove it. If no viable route exists, the candidate leaves this campaign. ([Existing authentication patterns](assignments/05-existing-auth-and-subscription-patterns.md))

### Evidence Standard

Production source is the behavior authority. Official documentation defines public contracts and supported access boundaries. A pinned released executable, native help/version output, and live bounded probes validate the exact route. Tests are validation evidence only. Earlier research is source targeting, not current runtime proof. A retained two-turn smoke proves only the tested launcher, version, platform, authentication route, two turns, artifacts, and observed cleanup; it does not prove a governed 30-turn campaign or cross-platform behavior. ([Existing runtime implementation](assignments/01-existing-runtime-implementation.md), [Product support contract](assignments/02-product-support-and-setup-contract.md))

### Decision Vocabulary

- **Materialized:** Skill Issue copied its known payload to a native skill root.
- **Discovered:** the harness reported or exposed the supplied skill.
- **Activated:** native tool/protocol or attributed marker evidence shows the supplied skill was loaded or invoked for a turn.
- **Tooling-complete:** launch, permissions, session, protocol, attribution, and cleanup succeeded; expected skill calls may still be zero as model outcome data.
- **Locally smoke-qualified:** a bounded local route completed under its recorded version, platform, access, and scenario limits.
- **Campaign-qualified:** the full governed matrix and completion contract passed.
- **Caveated/fail-closed:** a retained ambient input or platform limitation is explicit and the route refuses to proceed when it cannot prove the allowed set.
- **Unsupported:** the required contract or evidence is absent.
- **Unrun:** no live execution evidence exists.
- **Blocked:** finite evidence or user-owned access is missing; implementation must wait.
- **Removed:** Gate 2 has no acceptable route under this campaign's access constraint.

## Reconstructed Qualification Process

The four existing routes share one governed lifecycle rather than four independent scripts. The process below is the repeatable qualification method reconstructed from production. ([Existing runtime implementation](assignments/01-existing-runtime-implementation.md))

1. **Validate the request before side effects.** Accept one embedded evaluation or a complete custom triplet; canonicalize the workspace and answer-sheet paths; reject invalid schemas, scenario/turn IDs, skill names, and answer sheets inside the evaluated workspace; require an output root outside the workspace; resolve effective model/reasoning defaults; and obtain the caller's confirmation.
2. **Resolve the exact native route.** Record executable path, version, platform, launcher form, native authentication source, requested model/reasoning, and applicable policy. Preflight authentication/model access before temporary skill installation where the harness exposes a safe status route.
3. **Create a run-owned boundary.** Allocate a restrictive unique output directory, private state under the output root, and any private runtime home, sessions, cache, config, plugin, launch, trust, or passed-skill roots. Keep the real workspace distinct from run-owned configuration discovery.
4. **Inventory and exclude ambient inputs.** Identify user, project, system, managed, plugin, MCP, instruction, rule, memory, extension, compatibility, remote-skill, daemon, and environment sources. Disable them through documented process controls or fail closed when the acceptable set cannot be proved. Retained managed policy is recorded as a prerequisite.
5. **Materialize instrumented skills.** Copy each selected skill, insert an opaque run marker immediately after frontmatter, and materialize atomically at the candidate's evaluation root. Reject collisions outside the current canonical Skill Issue payload. Persist installation state before replay for interruption recovery.
6. **Build a least-privilege native invocation.** Pass the fixed model/reasoning values, disable unrelated features, constrain tools and workspace access, permit only required actions plus the exact marker command, and select the native sandbox when its behavior is qualified. Reject broad unattended approval modes.
7. **Run one conversation.** Set the active turn, send the prompt verbatim, capture the full structured response, clear the active turn, and continue the same session through a native resume identifier or one long-lived RPC process. The session identity must be stable.
8. **Validate fail-closed protocol evidence.** Require parseable nonempty structured output, a native initialization/session record, a terminal success record or qualified exit/idle predicate, absence of structured errors, stable session identity, tool failure classification, and native activation or opaque marker evidence attributable to the active turn. Final prose is not activation evidence.
9. **Separate tooling from model outcome.** A successful tool run with no expected skill call is valid scored data. Missing completion, permission denial of the marker, unstable resume, model refusal, malformed output, or unattributed activation is a tooling failure.
10. **Capture bounded artifacts.** Always write result and website artifacts only after successful replay. Retain raw events/transcript only when requested, sanitize known paths and identity values, and keep private tokens/run records below the output-owned state root.
11. **Cancel and clean by ownership.** Cancel the lifecycle on interrupt, terminate the owned process group or job object, close/purge native sessions when supported, confirm all children are gone, restore only recorded canonical Skill Issue paths, remove paths introduced by the run, remove private runtime roots/token mappings/run state, and preserve completed result artifacts.
12. **Retain recovery and scope limits.** If interrupted after materialization, use the output root and run ID to repeat the same bounded restoration. Do not promise byte-for-byte restoration of arbitrary pre-existing Skill Issue directory modifications, arbitrary workspace rollback, native account cleanup, or harness-history deletion.

### Additional Production Prerequisites

The production baseline adds prerequisites that a documentation-only candidate audit can easily miss:

- Evaluation admission and runtime preparation must exist before generic replay or installer code is relevant. The five candidates are presently registered for ordinary skill installation but rejected as evaluators before candidate runtime preparation. ([Local candidate maps](assignments/03-local-candidate-research-copilot-gemini.md), [Grok/OpenCode/Kilo map](assignments/04-local-candidate-research-grok-opencode-kilo.md))
- Authentication/model preflight belongs before instrumentation so failure does not leave temporary skills. A native protocol confirmation may substitute only when it fails closed on provider/model access. ([Existing authentication patterns](assignments/05-existing-auth-and-subscription-patterns.md))
- A candidate-specific validator is required. The generic JSON parser and advisory session-key search do not prove native terminal completion, tool failure, activation, or marker attribution.
- The marker must be allowed without broadly authorizing shell access, and it must be attributed only while a turn is active. Session reuse alone does not establish skill activation.
- Cleanup must cover every native child, detached server, sandbox, daemon, subagent, extension, background task, and remote session the route creates or attaches to. Parent exit is insufficient.
- Normal user auth may need to remain visible while sessions/config/skills are private. This is safe only when the harness offers a native separation, as Pi does between its agent auth directory and temporary session root. Copying or bridging credentials to manufacture that separation is rejected.
- Platform evidence is route-specific. The existing smoke is macOS-shaped; Cursor's keychain bridge is specifically macOS, Unix process-group ownership is stronger than the current Windows direct-process cleanup, and Pi's direct-process close is weaker than the generic Unix adapter.
- The existing Claude route inherits host environment variables and used an explicitly selected user-owned compatibility launcher; it is a conditional local route, not a general proxy precedent.
- The retained four-harness evidence is a two-turn smoke, not the governed campaign. Candidate live qualification must preserve that distinction.

## Reusable Qualification Checklist

- [ ] Record exact executable path, version, distribution, OS, and runtime prerequisites.
- [ ] Confirm a source-owned Skill Issue evaluation entry and candidate-specific defaults.
- [ ] Preflight user-owned authentication, provider, model, reasoning/variant, and policy without reading secrets.
- [ ] Allocate output outside the workspace and all possible temporary roots beneath run-owned state.
- [ ] Inventory user/project/system/managed instructions, config, skills, plugins, MCP, memory, extensions, daemons, and remote inputs.
- [ ] Prove the allowed discovered set or fail closed; never rely on name shadowing.
- [ ] Materialize a unique instrumented temporary skill and prove discovery plus native activation/marker attribution.
- [ ] Use a least-privilege prompt-free permission profile and a qualified sandbox; reject blanket approvals.
- [ ] Run two verbatim turns in one stable session and capture requested/effective model and reasoning.
- [ ] Require strict initialization, completion, error, session, tool, denied-action, and activation predicates.
- [ ] Interrupt a run and prove every child/server/daemon/subagent/background process is gone.
- [ ] Inventory touched files and remote retention; delete only run-owned skills, sessions, config, cache, state, and diagnostics.
- [ ] Repeat recovery from persisted installation state after interruption.
- [ ] Keep credential, identity, path, and raw transcript material private and sanitized.
- [ ] Pass Gate 1 before testing whether a native Codex/ChatGPT subscription route satisfies Gate 2.

## Unresolved Blockers

### Shared

- No remaining candidate has a production evaluation admission, runtime owner, defaults, candidate-specific validator, or marker-attribution path in Skill Issue.
- No candidate has completed the required live two-turn default/explicit route with strict protocol evidence, a negative permission/tool case, cancellation, child inventory, and bounded cleanup.
- No remaining candidate has cross-platform qualification. Existing process and credential bridges are platform-specific and cannot be generalized.
- No candidate has passed the governed campaign; the four baseline routes have only bounded local two-turn smoke evidence.

### OpenCode Remaining Blockers

The finite blockers are a tested released version, successful isolated discovery/config inspection, native OAuth availability and selected model/variant on that version, a safe boundary between normal auth and private generated state, candidate-specific `sessionID`/terminal/tool validation, activation and marker evidence, narrow permissions, interruption/child cleanup, explicit session deletion, and complete run-root inventory/removal. ([OpenCode cross-check](assignments/19-opencode-cross-check.md))

### Kilo Code Remaining Blockers

The finite blockers are an installed `>=1.0` CLI, user-owned native OAuth/model access, proof of supported isolation controls and exclusive skill discovery, a strict exit/idle/tool event contract, exact marker permission, same-session replay, daemon avoidance or ownership, interrupted descendant cleanup, touched file/network inventory, and a state layout that preserves native OAuth while removing only generated state. ([Kilo cross-check](assignments/20-kilo-code-cross-check.md))

### Removed Candidates

Copilot, Gemini, and Grok retain technical unknowns, but those unknowns no longer control this campaign. Their decisive blocker is the lack of an official route that uses only the user's existing paid Codex/ChatGPT subscription/model entitlement. Reopening any of them requires an explicit campaign scope change or a new first-party native subscription route, followed by a fresh Gate 1 qualification against the then-current executable. ([Copilot cross-check](assignments/16-github-copilot-cross-check.md), [Gemini cross-check](assignments/17-gemini-cli-cross-check.md), [Grok cross-check](assignments/18-grok-build-cross-check.md))

## Final Audit Conclusion

The strongest portable path is a narrow, evidence-first continuation with OpenCode and Kilo only. OpenCode is the first probe because it has local executable evidence and a finite protocol mismatch to resolve; Kilo is the second because it has the strongest first-party native subscription statement but no local runtime and a harder daemon/auth-state boundary. Neither is ready for implementation. Copilot, Gemini, and Grok should leave this campaign rather than accumulating provider, proxy, credential, or permanent-configuration workarounds that violate Skill Issue's ownership and privacy contract.
