# Result Data Update Workflow

## Assignment

- **Goal:** Recommend the smallest easy, low-risk workflow for replacing the representative website evaluation data with accepted Work Block 3 results and publishing the static React/Vite site through GitHub Pages.
- **Scope:** Current production result writers, the website data and chart contracts, repository validation and Vite configuration, current project-completion authorities, completed website-chart-migration assignments 01-05, and current official Vite and GitHub Pages guidance. The compared update paths are manually curated TypeScript, checked-in generated JSON, deterministic export/transform generation, and runtime fetching.
- **Exclusions:** Implementation, an implementation plan, changes to evaluation meaning, selection or acceptance of campaign runs, new result-schema design owned by Work Block 3, GitHub repository-setting changes, deployment execution, release creation, and edits to production or source materials.

## Findings

### Finding 1: Work Block 3 acceptance is the hard upstream boundary

The CLI deterministically derives one compact projection from each in-memory per-run result and scenario, but it persists optional diagnostics, `result.json`, and `website.json` sequentially with direct `os.WriteFile` calls. A failure can therefore leave a partial public output directory, including `result.json` without `website.json`; directory presence is not proof of a complete artifact pair or an accepted run. Even a complete `website.json` represents only one run of one scenario and omits effective reasoning, evaluation identity, environment qualification, acceptance status, three-scenario suite membership, and campaign identity. The current project still has no accepted cell, campaign aggregation artifact, or production schema that binds per-run artifacts into accepted suites.

**Evidence:** `WebsiteResult` contains only schema version, run ID, scenario ID, harness, model, total turns, and points (`cli/internal/evaluation/evaluation.go:98-113`). Each invocation derives one result, then writes optional events/transcript, `result.json`, and `website.json` in order (`cli/internal/evaluation/evaluation.go:285-315`). The public JSON and JSONL helpers call `os.WriteFile` directly rather than staging and renaming a complete public result set (`cli/internal/evaluation/evaluation.go:767-792`). Assignment 01 confirms that failed or interrupted runs can leave partial public artifacts and specifically identifies the `result.json`-without-`website.json` case (`assignments/01-cli-evaluation-production-path.md:137-143`). Assignment 02 establishes that one complete compact artifact is one run/scenario and cannot be silently treated as a whole campaign (`assignments/02-replay-scoring-turn-attribution.md:93-117`). Assignment 03 establishes that unique run identity exists while suite, campaign, qualification, and acceptance identity remain Work Block 3 responsibilities (`assignments/03-methodology-and-result-identity.md:88-134,188-194`). The controlling plan states that no accepted cell exists and assigns aggregation of accepted project-owned and external runs to Work Block 3 (`plans/skill-issue-project-completion/skill-issue-project-completion-a-to-b-plan.md:21-30,71-83`).

**Implication:** A Work Block 3 accepted-campaign manifest or equivalent aggregation artifact is a true prerequisite, not website-owned convenience metadata. It must accept only complete, validated detailed/compact pairs; identify their exact full `run_id` values, suite/cell membership, effective reasoning, accepted environment evidence, and campaign version; and reject partial directories rather than treating file presence as completion. Until that owner exists, the current clearly labeled representative data is the only honest public chart content.

### Finding 2: The recommended smallest low-risk workflow is one deterministic export into checked-in build-time data

The best fit is one explicit repository command that consumes only the Work Block 3 accepted-campaign manifest and its named `result.json`/`website.json` pairs, validates their joins and current schema, and writes one canonical, sorted, checked-in generated data module for the website build. `siteData.ts` should remain the website integration owner: human-curated product/release/method copy stays there, while it imports the non-editable generated result projection. The generated output should retain source run/scenario identity and campaign provenance alongside display-ready points, rather than containing only anonymous chart numbers.

**Evidence:** Production already derives compact points from authoritative result evidence, so the downstream command need not inspect transcripts or reinterpret scoring (`cli/internal/evaluation/evaluation.go:700-741`). Work Block 4 explicitly requires compact-artifact consumption through `siteData.ts`, numeric `turn`, `called`/`missed`, derived sample size, and data-driven limits (`plans/skill-issue-project-completion/skill-issue-project-completion-a-to-b-plan.md:85-95`). The authority map keeps evaluation meaning upstream and makes `siteData.ts` the editable website consumer of qualified artifacts (`plans/skill-issue-project-completion/document-authority-and-update-map.md:35-51`). Assignment 05 confirms the same layered owner boundary (`assignments/05-website-layout-contracts.md:84-117`). The committed npm/Vite toolchain already compiles checked-in static inputs into `dist` (`package.json:6-14`; `vite.config.ts:1-10`).

**Implication:** This creates one repeatable handoff: accepted evidence in, reviewable static data out. It avoids hand transcription, output-directory inference, transcript parsing, runtime availability, and unpublished data changes. It is smaller than building a service or live results API and safer than manually maintaining the full campaign in `siteData.ts`.

### Finding 3: A generated TypeScript module is the best checked-in representation under the current repository checks

A generated TypeScript module is marginally lower risk than a new JSON file in the current repository because the existing formatter, ESLint, and TypeScript commands already include `src/**/*.ts`. The generator can emit a literal that `satisfies` the website dataset type, so schema drift fails normal static validation. Checked-in JSON remains viable, and `resolveJsonModule` is enabled, but the current Prettier scripts do not include JSON under `src`; a JSON path would therefore depend more heavily on generator-owned canonical formatting and explicit runtime/static validation.

**Evidence:** `format` and `format:check` include `src/**/*.{ts,tsx,css}` but omit `src/**/*.json` (`package.json:10-11`). ESLint targets TypeScript and TSX (`eslint.config.js:7-27`). The application compiler is strict and enables `resolveJsonModule` (`tsconfig.app.json:1-24`). Assignment 04 shows that the website still needs a typed adaptation layer because the compact artifact lacks card fields and uses different point names (`assignments/04-current-website-chart.md:34-40,90-104`).

**Implication:** Prefer a semantically named generated `.ts` result-data module that is never manually edited, with `siteData.ts` assembling it into website content. If project maintainers strongly prefer preserving JSON as the derivative format, add it only with deterministic formatting and schema validation that participates in the final validation command; merely checking in copied `website.json` files is insufficient.

### Finding 4: Provenance validation matters more than the serialization format

The export must fail closed when evidence cannot be joined exactly. At minimum, it should require the supported schema version, join detailed and compact artifacts by full `run_id`, compare harness/model/scenario identity, read reasoning and evaluation identity from the authoritative detailed result or accepted manifest, require accepted suite/campaign membership, reject duplicate run IDs, reject invalid or missing points, preserve `total_turns`, and derive each displayed sample size from `sum(called + missed)`. Canonical ordering by campaign cell, scenario, and turn keeps reruns and review diffs stable.

**Evidence:** Detailed and compact identity fields differ (`cli/internal/evaluation/evaluation.go:79-113`). Numeric points are sparse because turns without expected calls are omitted, while `total_turns` covers the whole scenario (`cli/internal/evaluation/evaluation.go:700-741`; `cli/README.md:150-173`). Assignment 01 warns that the structured CLI response does not expose the generated output-directory path, so an importer should receive explicit artifact paths rather than infer the newest directory (`assignments/01-cli-evaluation-production-path.md:41-47`). Assignment 03 requires joins by full run ID and accepted campaign metadata rather than filenames or directory adjacency (`assignments/03-methodology-and-result-identity.md:88-110`).

**Implication:** The reviewable unit is the accepted manifest plus deterministic generated diff, not a collection of manually selected output folders. Hashes or immutable source paths are useful when Work Block 3 defines them, but the website exporter should consume that provenance rather than invent a second acceptance system.

### Finding 5: Manually curated TypeScript remains suitable for copy, not full result counts

Manual TypeScript is the smallest edit for the current two-card mock and remains appropriate for audience-facing copy, tested-environment wording, release links, and descriptions that require editorial judgment. It is a poor fit for 39 per-scenario runs in the thirteen-cell minimum campaign because it invites transcription errors, anonymous aggregation, stale sample sizes, and divergence from accepted source artifacts.

**Evidence:** The current `siteData.ts` manually stores two descriptions, sample sizes, and point arrays (`src/data/siteData.ts:41-80`), and the completed mock-up deliberately documents manual data additions (`README.md:143-149`). Work Block 3 requires thirteen cells with three scenarios per cell, while Work Block 4 forbids manual transcript reinterpretation (`plans/skill-issue-project-completion/skill-issue-project-completion-a-to-b-plan.md:71-95`). Assignments 02 and 03 show that scenario provenance, sparse denominators, reasoning, and suite acceptance cannot be reconstructed from hand-authored chart values (`assignments/02-replay-scoring-turn-attribution.md:93-117`; `assignments/03-methodology-and-result-identity.md:88-134`).

**Implication:** Keep editorial content manual and result numerics generated. A manual all-TypeScript path is a conditional fallback only for a deliberately tiny, clearly labeled demonstration or emergency correction whose exact source artifact is linked and whose provenance can be reviewed directly; it should not be the normal Work Block 3-to-website handoff.

### Finding 6: Checked-in generated JSON is viable only as the output of the deterministic transform

Checked-in JSON offers clean separation between evidence-derived values and TypeScript presentation, and it can be imported at build time without a runtime API. By itself, however, JSON does not decide which runs are accepted, validate suite identity, preserve detailed-result joins, or prevent someone from editing numbers manually. Copying raw per-run `website.json` files into the site would also leave the chart without reasoning, environment, campaign, description, or suite metadata.

**Evidence:** JSON imports are enabled (`tsconfig.app.json:16`), and Vite builds static inputs into `dist` (`package.json:8`; Vite [Deploying a Static Site](https://vite.dev/guide/static-deploy.html#building-the-app)). The compact artifact intentionally omits detailed and environment evidence (`cli/README.md:171-173`; assignment 02 at `assignments/02-replay-scoring-turn-attribution.md:111-117`). The current website graph additionally requires card metadata absent from `website.json` (`assignments/04-current-website-chart.md:34-40,90-96`).

**Implication:** Generated JSON is a sound conditional representation when a deterministic command produces it from accepted inputs and repository validation checks it. It is lower fit than generated TypeScript under the current scripts, and it should never become an independently hand-maintained result authority.

### Finding 7: Runtime fetching is rejected for the MVP publication path

Fetching results at runtime adds a second availability and error-handling path, weakens the commit/build review gate, permits displayed data to change independently of the deployed application, and introduces URL, caching, CORS, and partial-data failure concerns without solving any current requirement. Publishing raw diagnostic output for runtime consumption also increases disclosure risk, whereas the compact build-time artifact deliberately excludes transcripts, commands, and raw events.

**Evidence:** The established architecture requires local benchmark data and no runtime API or database (`plans/website/reference-and-architecture.md:3-5`; `README.md:121-149`). GitHub states that a Pages site is publicly available, including when the repository itself may be private ([Configuring a publishing source](https://docs.github.com/en/pages/getting-started-with-github-pages/configuring-a-publishing-source-for-your-github-pages-site#about-publishing-sources)). Local CLI documentation warns that optional transcripts retain complete prompts, responses, commands, output, errors, and possibly confidential information, while `website.json` excludes them (`cli/README.md:136-173`).

**Implication:** Keep published chart data inside the reviewed static build. Runtime fetching should be reconsidered only if a future hosted-results product explicitly requires independent data releases, defines an authenticated/public API contract, and accepts its operational and disclosure boundaries.

### Finding 8: GitHub Actions building `dist` is the smallest supported Pages publication path

The repository should use GitHub Pages with GitHub Actions as its publishing source. The deployment workflow should install the committed lockfile with `npm ci`, run the repository's full `npm run validate`, upload `dist` as the Pages artifact, and deploy it with the official Pages action. The existing production default base `/skill-issue/` already matches the project-site URL shape for repository `skill-issue`; `VITE_BASE_PATH` remains the override for a different repository path, user site, or custom domain.

**Evidence:** The repository has a committed `package-lock.json`, `build` produces the Vite output, and `validate` includes formatting, linting, type checking, and build (`package.json:6-14`). Production base selection is `/skill-issue/` unless `VITE_BASE_PATH` is supplied (`vite.config.ts:4-10`). Vite requires `/<REPO>/` for a project site, directs users to select GitHub Actions, and shows checkout, Node setup, `npm ci`, build, Pages configuration, `dist` upload, and deployment ([Vite GitHub Pages guidance](https://vite.dev/guide/static-deploy.html#github-pages)). GitHub documents the same build/upload/deploy flow and requires `pages: write`, `id-token: write`, a linked build dependency when jobs are separate, and a deployment environment ([Using custom workflows](https://docs.github.com/en/pages/getting-started-with-github-pages/using-custom-workflows-with-github-pages#deploying-github-pages-artifacts)).

**Implication:** Do not commit `dist`, introduce a `gh-pages` branch, or add a second deployment service. The source commit, generated dataset, validation run, Pages artifact, and production deployment remain traceable in one GitHub Actions run.

### Finding 9: Publish on accepted default-branch changes, with manual dispatch retained

The best default trigger is a reviewed push to `main`, with `workflow_dispatch` retained for first deployment and safe reruns. Pull requests should run validation without deploying; deployment follows only the accepted default-branch state. If the repository cannot yet guarantee that `main` contains only publication-ready website data, a manual-only Pages trigger is the lower-automation conditional alternative until the acceptance gate is reliable.

**Evidence:** Vite's official sample triggers Pages deployment on pushes to `main` and manual dispatch ([Vite GitHub Pages guidance](https://vite.dev/guide/static-deploy.html#github-pages)). GitHub describes the custom workflow flow as default-branch push or manual execution and says deployment is skipped for pull requests; it also recommends a `github-pages` environment protected so only the default branch deploys ([Configuring a publishing source](https://docs.github.com/en/pages/getting-started-with-github-pages/configuring-a-publishing-source-for-your-github-pages-site#creating-a-custom-github-actions-workflow-to-publish-your-site)). The project completion plan requires publishing only after product and evaluation evidence stabilize (`plans/skill-issue-project-completion/skill-issue-project-completion-a-to-b-plan.md:85-95`).

**Implication:** Automatic deploy-on-main gives the smallest repeatable publishing loop after Work Block 3 acceptance and code review. Manual-only deployment is safer during the first migration if repository governance is still incomplete, but it increases the risk that the public site lags the reviewed source.

### Finding 10: Current prerequisites and gaps are concrete and publication-blocking

The static build foundation exists, but real-data publication is not ready. Current gaps are: no accepted Work Block 3 run or qualified cell; no accepted-campaign/suite aggregation schema; no production result dataset for the website; no Pages workflow in `.github/workflows`; no recorded Pages deployment; no GitHub Release or final download assets; a mock-only chart schema and percentage/fixed-axis component; and no pinned project Node version even though the installed Vite package requires Node `^20.19.0 || >=22.12.0`.

**Evidence:** The parent plan records no accepted result, no Pages deployment/workflow, and no release (`plans/skill-issue-project-completion/skill-issue-project-completion-a-to-b-plan.md:11-14,21-30`). The reorganization audit records unfinished campaign aggregation, real-result ingestion, release, and deployment (`plans/skill-issue-project-completion/reorganization-dependency-audit.md:89-113`). Repository inspection found no `.github/workflows` directory. `siteData.ts` contains the actual repository destinations and explicit mock content (`src/data/siteData.ts:16-80`). Assignments 04 and 05 establish the required chart migration rather than a data-only swap (`assignments/04-current-website-chart.md:34-104`; `assignments/05-website-layout-contracts.md:66-117`). Installed `node_modules/vite/package.json:178-180` records the Node range; `package.json` has no `engines` field.

**Implication:** The true blocker is upstream accepted campaign identity/data. Once that exists, the remaining website and Pages changes are bounded: migrate the chart contract, generate/import accepted static data, replace public routes/copy, add the official Pages workflow, enable GitHub Actions as the Pages source, and verify the deployed product against the Work Block 4 audit boundary. Pages administrator/maintainer access is also required to select the publishing source ([GitHub publishing-source requirements](https://docs.github.com/en/pages/getting-started-with-github-pages/configuring-a-publishing-source-for-your-github-pages-site#who-can-use-this-feature)).

### Finding 11: Existing validation commands already provide the core deployment gate

The repository's exact fast owner checks after TypeScript/TSX edits are `npm run typecheck` followed by `npm run lint`. Markdown, JSON, CSS, HTML, and configuration edits require `npm run format:check`. Before website work is presented as complete, the required aggregate command is `npm run validate`, which executes format checking, lint, type checking, and the production build. `npm run preview` serves the compiled project-site build at `http://127.0.0.1:4173/skill-issue/`; `npm run dev -- --host 127.0.0.1` is the browser-automation development server.

**Evidence:** These commands and their order are binding in `AGENTS.md:8-22`. Their exact implementations are in `package.json:6-14`. The production preview path and base override are documented in `README.md:134-149` and implemented by `vite.config.ts:4-10`.

**Implication:** The data-export command should be deterministic and idempotent, then the existing owner checks and `npm run validate` should gate both review and Pages deployment. A post-deployment inspection must still verify production navigation, turn-based charts, themes, download and repository links, base-prefixed assets, tested-environment labels, limitations, and public claims because a successful static build does not establish those live behaviors.

### Finding 12: Lower-fit publication and data paths should remain conditional or rejected

Publishing prebuilt files from a branch or committed `dist` is lower fit because this Vite site requires a build and the official guidance prefers a custom Actions workflow for non-Jekyll build processes. Generating the website dataset during the Pages job from uncommitted external results is also lower fit because reviewers would see source code but not the exact derived data before deployment. Generating the entire `siteData.ts` is lower fit because it mixes editorial copy with machine-owned values and risks overwriting the website's human semantic owner.

**Evidence:** GitHub recommends a custom workflow when a site needs a build process other than Jekyll or should not keep compiled files on a dedicated branch ([Configuring a publishing source](https://docs.github.com/en/pages/getting-started-with-github-pages/configuring-a-publishing-source-for-your-github-pages-site#about-publishing-sources)). Vite's documented GitHub Pages path builds and uploads `dist` inside Actions ([Vite GitHub Pages guidance](https://vite.dev/guide/static-deploy.html#github-pages)). The local website contract assigns all editable copy and release meaning to `siteData.ts` (historical `plans/website/data-and-design-contract.md:3-34` at commit `22445be`).

**Implication:** Keep generated result data checked in and reviewable, keep copy curated, and let deployment consume the exact accepted repository state. Branch publishing is a fallback only if GitHub Actions is unavailable; deployment-time remote generation should be rejected for the initial public release.
