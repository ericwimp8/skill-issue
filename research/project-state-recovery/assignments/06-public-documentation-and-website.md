# Public Documentation and Website

## Assignment

**Goal:** Cross-check the repository's public documentation and website against implemented and accepted local evidence, identify the publicly asserted current state and its discrepancies, and state the implications for one internal progress owner plus a separate routing index.

**Scope:** `README.md`, `INSTALL.md`, `cli/README.md`, website data and rendering code under `src/`, package and release metadata, accepted website artifacts, the campaign progress record, the result-publication script, relevant workflow/build files, and recent commits that explain status drift.

## Findings

### Finding 1: The implemented and publicly documented product surface agrees on the principal shipped capabilities

The current source supports five install-and-evaluate harnesses, three governed 30-turn evaluations, eleven embedded skills, a standalone Go CLI, and a static React/Vite website. These are current public truths, rather than unfinished intentions. The public qualification caveats also remain material: cross-compilation covers six OS/architecture targets, while native runtime qualification remains narrower than build availability.

**Evidence:** `README.md:73-97` describes three fixed scenarios and five harnesses; `README.md:168-194` describes eleven skills, the standalone CLI, six cross-build targets, and the native-qualification caveat. `cli/README.md:23-37` names the same five evaluation harnesses and their effective defaults. The concrete registry contains exactly `claude-code`, `codex`, `cursor`, `opencode`, and `pi`, each with evaluation defaults (`cli/internal/harness/harness.go:14-20`, `cli/internal/harness/harness.go:62-117`). The payload manifest contains eleven components (`cli/internal/payload/assets/manifest.json:5-50`). The three non-smoke built-ins each contain 30 turns and 46, 46, and 45 expected calls respectively, consistent with `cli/README.md:75-81`. The website exposes the same headline metrics (`src/data/siteData.ts:127-135`) and derives its method records directly from the built-in JSON and evaluated skill entrypoints (`src/data/methodologyData.ts:48-61`, `src/data/methodologyData.ts:81-146`).

**Implication:** A recovered internal progress owner should record these capabilities as implemented baseline state and track only remaining qualification, evidence, distribution, and follow-up work. A routing index should point readers to the harness registry, payload manifest, built-ins, and public copy owners instead of restating their contents.

### Finding 2: The accepted public evidence set is 21 runs across seven complete configurations, and the published collection faithfully mirrors it

The accepted folder and published website collection both contain 21 schema-v2 artifacts. They cover seven harness/model cells, and every one of those cells contains all three scenarios. The published JSON is an exact normalized set match for the accepted JSON files. Current aggregate public evidence therefore represents 630 conversation turns, 364 called expectations, 595 missed expectations, and 128 additional calls. Two planned cells remain absent: Cursor–Codex and Cursor–Claude Fable.

**Evidence:** `scripts/update-website-results.mjs:4-7` defines the accepted folder and published destination; `scripts/update-website-results.mjs:38-55` validates schema-v2 records; `scripts/update-website-results.mjs:58-136` loads every accepted JSON, rejects duplicate run IDs and configurations, sorts the records, and writes the published collection. Local set comparison found 21 accepted records, 21 published records, and no normalized differences. Grouping `src/data/publishedWebsiteArtifacts.json` yielded three records each for Claude Code–Claude Fable, Claude Code–Codex Sol, OpenAI Codex–Codex Sol, Cursor–Composer, Cursor–Grok, OpenCode–Codex Sol, and Pi–Codex Sol. `src/data/evaluationData.ts:82-146` declares those seven cells available and the two Cursor cells unavailable. `src/data/analysisData.ts:52-71` defines completeness as all three scenario IDs and includes only complete cells in campaign aggregates. The campaign tracker summary independently reports 21/27 runs and 7/9 suites (`plans/skill-calling-evaluation-campaign/evaluation-progress.md:9-36`).

**Implication:** Accepted artifacts plus their deterministic publication script should own public result facts. The internal progress owner may record execution state and blockers, but public counts should be derived from accepted artifacts rather than copied from operational notes. The routing index should distinguish the accepted-evidence owner from the execution-progress owner.

### Finding 3: The public Claude Code–Codex status narrative is stale and conflicts with both the artifacts and the rendered charts

The analysis copy says only one Claude Code–Codex scenario is accepted, calls that cell incomplete, and says it is excluded from aggregate and pairing totals. Three accepted records now exist for that cell. The website marks the cell available, includes it in the 21-run headline aggregate, and includes it in the harness comparison. The stale prose therefore understates public evidence and makes the analysis page internally inconsistent: Claude Code–Codex contributes to headline numbers and a chart while accompanying copy says it does not.

**Evidence:** The stale claims are in `src/data/siteData.ts:253-262`. Three accepted Claude Code–Codex records identify the three distinct scenario IDs at `evaluations/skill-calling/results/accepted/e20e00601b9cf52124680fb0c0a937e5.json:1-8`, `evaluations/skill-calling/results/accepted/a0f46b4ec92e74c09eba16abcd1121a7.json:1-8`, and `evaluations/skill-calling/results/accepted/12192ed7bb64e3cf049d712876b99bc5.json:1-8`. The complete-cell aggregate therefore includes this cell through `src/data/analysisData.ts:52-71`, and `ResultsAnalysisPage` renders that aggregate as accepted-run, turn, expected-call, and success figures (`src/pages/ResultsAnalysisPage.tsx:39-69`). The harness chart selects every `codex-sol` result across harnesses (`src/components/EvaluationExplorer.tsx:314-324`). History explains the drift: commit `b64d9b5` added the two remaining Claude Code–Codex accepted artifacts and appended them to `publishedWebsiteArtifacts.json`, but did not update `siteData.ts`; `siteData.ts` was last changed earlier at `b5d926a`.

**Implication:** Public current-state prose must be reconciled when accepted evidence changes, even when chart data updates automatically. The internal progress owner should expose a concise accepted-set checkpoint that prompts this reconciliation. The routing index should route “public interpretation/copy” to `src/data/siteData.ts` and “accepted results” to the accepted folder plus publication script.

### Finding 4: The campaign progress document is the intended internal execution owner, but its own duplicated status and result summaries are stale

The campaign progress document explicitly limits itself to execution progress, which makes it the natural internal progress owner for this campaign. Its top summary is current enough to state 7/9 suites, 21/27 runs, and six blocked Cursor runs. Lower sections conflict with that summary: Cursor–Composer is labelled 0/3 even though all three child runs are checked complete; campaign notes still say only four runs completed; several retained observed-call counts differ from the accepted reconciled artifacts; and multiple failure-log “rerun result” fields still say running or pending after completion. The public README currently links this operational document under “Project status,” exposing an internally inconsistent tracker as a public status destination.

**Evidence:** Ownership is stated directly at `plans/skill-calling-evaluation-campaign/evaluation-progress.md:3-7`, while the current high-level totals appear at `plans/skill-calling-evaluation-campaign/evaluation-progress.md:9-36`. The document's own rule says a configuration heading is checked only after all three evaluations complete and that the tables must update whenever status changes (`plans/skill-calling-evaluation-campaign/evaluation-progress.md:38-49`). Cursor–Composer nevertheless says 0/3 while its three runs are complete (`plans/skill-calling-evaluation-campaign/evaluation-progress.md:108-113`). Campaign notes say “Four runs completed” despite the 21-run summary (`plans/skill-calling-evaluation-campaign/evaluation-progress.md:150-160`). For example, the tracker records 16 and 44 observed calls for the first two OpenAI Codex scenarios (`plans/skill-calling-evaluation-campaign/evaluation-progress.md:71-76`), while accepted reconciled artifacts publish 46 and 46; the public website explicitly documents the attribution correction and reconciliation receipts (`src/data/siteData.ts:280-285`). **Implication:** Keep one internal execution owner, but reconcile its summary, lane headings, terminal notes, and result references so it can be trusted. It should link to accepted artifacts for final scored counts rather than duplicating mutable post-reconciliation totals. A separate routing index should direct “what remains operationally” to the progress owner and “what is publicly accepted” to the accepted evidence/publication path.

### Finding 5: Public release messaging mixes a shipped beta with pre-release status language

The website and installation guide assert an available current CLI release across six download targets. Local Git evidence supports a `v0.1.0` release tag, exact archive names, checksums generation, enabled links, and beta/native-qualification caveats. In contrast, the README project status still says release preparation and public distribution “continue,” and the project page says the first complete path is “taking shape.” Those phrases read as pre-release even though the same public surfaces provide live download calls to action and describe existing publication. The remaining work is better described as continued development, native runtime qualification, incomplete comparison cells, and future releases.

**Evidence:** All six download URLs are enabled in `src/data/siteData.ts:8-45`, and both hero/header surfaces open the download dialog through `src/pages/LandingPage.tsx:22-47` and `src/App.tsx:113-151`. The dialog labels the CLI beta and states that cross-platform builds compile while Windows, Linux, and Intel macOS runtime qualification continues (`src/components/DownloadDialog.tsx:8-28`). `INSTALL.md:3-7` calls it the current release; `INSTALL.md:20-141` publishes archive/checksum instructions and repeats the Windows qualification caveat. `cli/scripts/build-cross-platform.sh:16-38` builds six targets and `cli/scripts/build-cross-platform.sh:40-75` creates archives matching the website URLs plus `checksums.txt`. Local tag `v0.1.0` points to `b5d926a`, whose release commit enabled all six links and added packaging. By comparison, `README.md:275-280` says release preparation and public distribution continue, while `src/data/siteData.ts:232-240` says the first complete path is taking shape even as the same paragraph lists a CLI, method, workflow, artifacts, and static publishing.

**Implication:** Public status should describe the shipped beta and its precise incomplete qualifications. The internal progress owner should track follow-up release/qualification work without regressing the public milestone to “not yet distributed.” The routing index should point release availability to release metadata/build packaging and qualification state to the progress owner or a dedicated qualification record.

### Finding 6: Remote download and deployment availability cannot be proven from the selected local-only scope

Local evidence proves intended release filenames, link construction, tags, and deploy/build workflows. It does not contain a retained GitHub Release asset manifest or Pages deployment receipt that independently proves the current remote endpoints resolve. The public claims may be correct, but that part remains unsupported by this local-only run.

**Evidence:** Download URLs target `/releases/latest/download/...` in `src/data/siteData.ts:8-45` and `INSTALL.md:35-42`, `INSTALL.md:91-99`, and `INSTALL.md:121-126`. The local build script creates exactly those filenames (`cli/scripts/build-cross-platform.sh:40-75`). The Pages workflow can validate, upload, and deploy the site (`.github/workflows/pages.yml:13-55`), and `README.md:222-244` describes manual publication. No tracked release workflow, release asset inventory, Pages run receipt, or deployment result was found. The local tags establish intended snapshots, not live remote availability.

**Implication:** The progress owner should distinguish “packaged/tagged/configured” from “remotely verified,” recording a retained verification reference when one is required. The routing index should point remote-release questions to the external release/deployment owner rather than treating static repository URLs as conclusive evidence.

### Finding 7: The public methodology and interpretation are generally well bounded and remain aligned with accepted evidence

The website correctly frames results as configuration-specific rather than universal model rankings, derives scenario and skill records from repository sources, limits aggregates to complete cells, keeps additional calls separate from expected-call success, and discloses instrumentation and isolation limits. The principal methodological status drift is the Claude Code–Codex completion copy, not the evaluation contract itself.

**Evidence:** `src/data/siteData.ts:137-207` describes the bounded question, fixed turns, scoring, Dictate Plan startup boundary, instrumentation visibility, differing controls, evidence layers, and non-universal interpretation. `src/data/methodologyData.ts:48-61` imports built-in evaluations and evaluated skill sources; `src/data/methodologyData.ts:81-146` derives scenario calls and skill counts. `src/data/analysisData.ts:17-45` calculates expected-call success as called divided by called plus missed while retaining additional calls separately. All accepted artifacts omit turns 13, 18, and 24 entirely, confirming that no expected or additional calls were published for the three decoy turns, consistent with `src/data/siteData.ts:280-285`. Chart implementations use called/missed expectations only (`src/components/charts/CumulativeCallsChart.tsx:23-47`, `src/components/charts/TurnRasterChart.tsx:46-79`), matching the disclosed note at `src/components/EvaluationExplorer.tsx:435-438`.

**Implication:** Preserve this public methodological boundary when recovering status. The progress owner should report execution and acceptance changes without broadening claims; the routing index should direct methodological questions to the built-ins, instrumentation contract, evaluator source, and methodology data rather than to the progress tracker.

### Finding 8: Public truth, internal progress, and routing should be three distinct responsibilities

The repository already has suitable semantic owners but currently lets status meaning leak across them. Accepted artifacts and generated website data own public results; `siteData.ts` owns curated public interpretation and status copy; the campaign tracker owns operational progress. A routing index should only direct readers among these owners and must not duplicate live counts, blocker states, or release status.

**Evidence:** Repository instructions assign curated website copy, release metadata, and summary metrics to `src/data/siteData.ts` and evaluation adaptation to `src/data/evaluationData.ts` (`AGENTS.md`, “Website Development”). The publication script owns the accepted-to-public data projection (`scripts/update-website-results.mjs:4-136`). The tracker declares execution-only ownership (`plans/skill-calling-evaluation-campaign/evaluation-progress.md:3-7`). The README currently combines product explanation, development instructions, publication procedure, project status, and a link to the internal tracker (`README.md:168-286`), which is where users encounter conflicting lifecycle signals.

**Implication:** Recover one authoritative internal progress document and create a separate routing index that names, but does not restate, each semantic owner. Public status should remain in audience-facing copy and be reconciled from accepted/release evidence; operational detail should remain internal; navigation should remain stable even as either changes.
